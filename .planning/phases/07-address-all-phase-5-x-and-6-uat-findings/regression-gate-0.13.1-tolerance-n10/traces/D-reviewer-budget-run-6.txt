## Review Summary

Reviewed: `review-src/handler.ts` and `review-src/validate.ts` (last commit: "add unvalidated handler with JSON.parse for reviewer scenario")

### Findings

`review-src/handler.ts:7` -- **Critical** -- `JSON.parse(data)` throws `SyntaxError` on malformed input. Wrap in try/catch and return null or throw a typed error.

`review-src/handler.ts:2` -- **Important** -- `req.body` access throws `TypeError` when `req` is null/undefined. Guard with `if (!req) return null;` or `req?.body`.

`review-src/validate.ts:1` -- **Important** -- `validate` is exported but unused by `handle()`. Import and call before `process(req.body)`, or delete the file if it is obsolete.

`review-src/handler.ts:1,6` -- **Suggestion** -- Implicit `any` on `req`, `data`, `x` removes static safety. Type as `{body?: string}`, `string`, `unknown`.

`review-src/handler.ts:7` -- **Important** -- `process` returns the `JSON.parse` result unvalidated; `"null"`, `"false"`, `"0"` parse to falsy values that may mislead callers expecting an object. Validate the parsed shape post-parse.

`review-src/handler.ts:3` -- **Important** -- `if (req.body)` treats empty string and `0` as absent, conflating "missing body" with "falsy body". Use `req.body !== undefined` or an explicit presence check.

### Per-finding validation

**Finding 1:** Confirmed Critical. `JSON.parse` on malformed input is the canonical uncaught-exception vector in Node handlers; with no caller-side try/catch documented, a single malformed POST body crashes the request. Severity stands.

**Finding 2:** Confirmed Important. Crashes only when caller violates implicit contract (passing nullish `req`), but in middleware pipelines this is reachable via misrouted invocations. Not Critical because normal HTTP frameworks always pass a request object; remains Important.

**Finding 3:** Revise to Suggestion-bordering-Important. The commit title explicitly advertises "unvalidated" -- this is deliberate scenario seeding, not an oversight. As real code it would be Important (dead-export + missing integration); as a fixture it documents the gap intentionally. Treat as Important if this code ships; Suggestion if scenario-only.

**Finding 4:** Confirmed Suggestion. Types do not change runtime behavior, but in concert with findings 1 and 2 the absence of `req: Request` annotation removes the one mechanism that would catch findings 2 and partially 1 at compile time. Severity stands; impact amplified by cross-cutting pattern below.

### Cross-Cutting Patterns

Findings 1, 2, and 6 share a single root cause: `handle()` accepts an untrusted external input and performs no boundary validation before consumption. The cascade is `handle(req)` -> `req.body` access -> `JSON.parse(body)` with three trust assumptions (req non-null, body present-and-string, body well-formed JSON) and zero guards. Finding 3 (`validate` orphaned) and Finding 4 (no types) are the missing controls that would have closed this gap -- one at runtime, one at compile time. Recommended single fix: type `req: {body?: string}`, import and invoke `validate(req?.body)` as a presence gate, then `try { return JSON.parse(req.body) } catch { return null }`. This collapses findings 1, 2, 3, 6 into one boundary-validation refactor.

**Missed surface:** `process()` is non-exported but its `any` return type leaks through `handle()`'s inferred return; callers receive `any` and lose type safety transitively across the module boundary.

**Verdict scope:** scope: api-correctness
