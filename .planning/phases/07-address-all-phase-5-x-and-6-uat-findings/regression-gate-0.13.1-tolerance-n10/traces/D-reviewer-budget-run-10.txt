## Review Summary

Reviewed: `review-src/handler.ts` and `review-src/validate.ts` (commit d5363ad)

### Findings

`review-src/handler.ts:7: crit: `JSON.parse(data)` throws `SyntaxError` on malformed body, propagates unhandled. Wrap in try/catch, return typed error result.`
`review-src/handler.ts:2: imp: `if (req.body)` drops falsy-but-valid bodies (`"0"`, `""`, `false`). Use `req.body != null`.`
`review-src/validate.ts:1: imp: `validate` exported but never imported by `handler.ts`; validation layer bypassed. Import and gate `process()` on `validate(req.body)`, or delete dead module.`
`review-src/handler.ts:1-3: imp: `req` and `data` parameters untyped (implicit `any`). Add `{ body: unknown }` and `string` annotations to restore type safety.`
`review-src/handler.ts:7: imp: `process` shadows Node.js global `process`. Rename to `parseBody` to avoid confusion and accidental global reference.`
`review-src/handler.ts:7: imp: return type of `JSON.parse` is `any`; flows out of `handle()` untyped to all callers. Annotate as `unknown` and narrow at call site.`
`review-src/validate.ts:1: sug: `validate(x)` only checks non-null; name overpromises. Rename `isPresent`, or expand to schema validation matching the JSON-parsed shape.`

### Cross-Cutting Patterns

Findings 1, 2, 3, and 4 share one root cause: the boundary between untrusted HTTP input and internal logic has no validation contract. `req.body` enters as `any`, is filtered by a truthy check, parsed without error handling, and returned as `any`. A single zod/valibot schema applied at entry would close findings 1-4 simultaneously and give `validate.ts` a real job. Findings 5 and 6 are local hygiene independent of the boundary issue.

### Missed surfaces (optional)

No callers of `handle()` were packaged; verify upstream error handling does not assume `handle()` is total before shipping the try/catch fix.
