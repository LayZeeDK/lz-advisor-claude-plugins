## Review Summary

Reviewed: `review-src/handler.ts` and `review-src/validate.ts` (commit `346cf91` -- "add unvalidated handler with JSON.parse for reviewer scenario")

### Findings

`review-src/handler.ts:7: crit: `JSON.parse(data)` throws `SyntaxError` on malformed input; propagates uncaught. Wrap in try/catch, return error result or null.`

`review-src/handler.ts:2: imp: truthy guard drops `""`, `0`, `false`, `[]`. Replace with `req.body != null`.`

`review-src/handler.ts:1: imp: `validate` from sibling module never imported; handler bypasses validation before `JSON.parse`. Import and call `validate(req.body)` before `process`.`

`review-src/validate.ts:1: imp: `validate` only checks `x != null`; accepts non-string types `JSON.parse` will reject and does not verify parseability. Tighten to `typeof x === 'string' && x.length > 0` (or schema-validate post-parse).`

`review-src/handler.ts:1,6: sug: implicit `any` on `req`, `data`; no return type. Type as `{ body?: unknown }` and `string`; annotate return.`

`review-src/validate.ts:1: sug: implicit `any` on `x`, no return-type annotation. Type `x: unknown`, return `boolean`; consider type predicate `x is string`.`

`review-src/handler.ts:7: imp: `process` returns `any` from `JSON.parse`; caller receives untyped data. Validate parsed shape with schema (zod/io-ts) or narrow before return.`

### Per-finding validation

Validation of Finding 1: Confirmed Critical. `JSON.parse` on `'{'`, `'undefined'`, or any non-JSON string throws synchronously. With no try/catch and no caller-side boundary visible, this is a guaranteed crash on first malformed payload -- standard request-handler footgun.

Validation of Finding 2: Confirmed Important. Severity correct: empty-string bodies are common (DELETE requests, empty POSTs) and `0` / `false` are valid JSON bodies a parser would happily accept. The fix `req.body != null` preserves intent while admitting falsy-but-present values.

Validation of Finding 3: Confirmed Important. The dead-code signal is strong: same directory, sibling file, exported symbol, no other consumers visible. Author intent almost certainly was `if (validate(req.body)) return process(req.body)`. Ship the wiring.

Validation of Finding 4: Confirmed Suggestion. Not correctness-affecting at runtime but blocks downstream type-safety. Bumping to Important is tempting but the runtime bugs (1-3) are the actual user-facing failures; types are the prevention layer, not the bug.

### Cross-Cutting Patterns

Findings 1, 3, and 4 share a root cause: the boundary between untrusted input (`req.body`) and parsed/processed data has no validation, no error handling, and no type narrowing. A single zod (or equivalent) schema applied at the boundary would resolve all three: parse safely, validate shape, and produce a typed result. Finding 2 is an independent logic bug in the guard expression. The added Finding 4 (validate is too weak) and Finding 7 (untyped parse result) extend the same boundary-validation theme -- the codebase treats `JSON.parse` output as trusted, which it is not.

### Missed surfaces (optional)

Check whether `handle()` callers expect `null` vs throw for invalid input -- the contract is currently undefined and the fix in Finding 1 must pick one.
