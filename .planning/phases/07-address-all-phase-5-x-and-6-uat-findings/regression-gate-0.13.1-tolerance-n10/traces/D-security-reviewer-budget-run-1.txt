## Security Review Summary

Reviewed: `review-src/handler.ts` and `review-src/validate.ts` -- last commit (`1557849`, "add unvalidated handler with JSON.parse for security-reviewer scenario")

### Findings

`review-src/handler.ts:6-8: crit: [A08] JSON.parse on attacker-controlled req.body enables prototype pollution via __proto__ payload on Node<23. Reject __proto__/constructor keys via reviver or use safe parser (secure-json-parse).`
`review-src/handler.ts:6-8: imp: [A03] JSON.parse lacks try/catch; SyntaxError on malformed input crashes process or leaks stack trace. Wrap in try/catch, return 400 on parse failure.`
`review-src/handler.ts:1-3: imp: [A04] exported validate never imported in handler; defense-in-depth bypassed entirely. Import and gate process() behind validate(req.body) plus schema check.`
`review-src/validate.ts:1: imp: [A04] validate(x) only checks x != null; permits any payload shape. Replace with schema validator (zod/ajv) enforcing type, structure, and length bounds.`
`review-src/handler.ts:5: sug: [A09] handle() returns null on missing body with no logging; failed/malformed requests leave no audit trail. Add structured logging on validation/parse failure paths.`
`review-src/handler.ts:2: sug: [A04] handler signature lacks TypeScript types (req:any implicit); type system cannot catch shape mismatches. Type req as {body?: string} and enable strict/noImplicitAny.`

### Per-finding validation

Validation of Finding 1: Severity confirmed Critical, but primary OWASP tag revised from A03 to A08 (Software and Data Integrity Failures) -- prototype pollution is the integrity-violation vector, not classical injection. Note: Node.js 21+ JSON.parse still preserves `__proto__` as own-property (not pollution by itself); pollution occurs when downstream code merges/assigns the parsed object into another via Object.assign/lodash.merge/spread-into-prototype patterns. Severity remains Critical because such sinks are common and not visible in scope.

Validation of Finding 2: Severity confirmed Important, OWASP retagged from A03 to A04 (Insecure Design) -- the defense exists but the wiring is absent; this is a design/integration flaw, not an injection class per se.

Validation of Finding 4: Severity confirmed Suggestion-to-Important; upgraded to Important because combined with Finding 2 it represents the entire input-validation layer being non-functional.

### Threat Patterns

Findings 2+1 chain into the dominant attack path: validate() is never invoked (Finding 2), so attacker-controlled req.body reaches JSON.parse unchecked (Finding 1), enabling both prototype pollution (Critical) and parse-crash DoS (Finding 3). Finding 4 compounds: even if Finding 2 were fixed, the validator itself is a no-op, so the chain remains open. Root cause is systemic: input-trust boundary is absent end-to-end. Single fix (zod schema + safe-parse + wired validate call) closes findings 1, 2, 3, and 4 simultaneously.

### Missed surfaces (optional)

Downstream consumers of `process()` return value not in scope; if any caller does `Object.assign({}, parsed)` or lodash.merge, prototype pollution becomes immediately exploitable beyond theoretical.
