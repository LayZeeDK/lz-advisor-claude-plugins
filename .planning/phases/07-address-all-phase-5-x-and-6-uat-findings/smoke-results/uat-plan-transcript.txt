## 📚 Storybook + Compodoc + Angular: Docs Tab & JSDoc Setup

### 🔍 What is Compodoc in Storybook for Angular?

When used with Storybook, Compodoc can automatically generate documentation for your Angular components by analyzing their TypeScript code, decorators, and JSDoc comments.

To make the most out of Storybook in your Angular project, you can set up Compodoc integration and Storybook decorators based on your project needs.

---

### ✍️ Using JSDoc Comments

You can include JSDoc comments above components, directives, and other parts of your Angular code to include documentation for those elements. Compodoc uses these comments to generate documentation for your application.

In Storybook, it is useful to add explanatory comments above `@Inputs` and `@Outputs`, since these are the main elements that Storybook displays in its user interface. The `@Inputs` and `@Outputs` are elements you can interact with in Storybook, such as controls.

As noted in the Compodoc documentation, "Compodoc uses the Typescript AST parser and it's internal APIs, so the comments have to be JSDoc comments."

#### Supported JSDoc Tags
Compodoc only supports a small number of JSDoc tags: `@returns`, `@ignore`, `@param`, `@link`, and `@example`. By far, the most useful of these in Storybook is `@ignore`. The `@ignore` tag will remove the property from the Storybook Controls panel, which helps hide properties and functions that external components should not use.

---

### ⚙️ Setting Up Compodoc (Automatic vs Manual)

When installing Storybook via `npx storybook@latest init`, you can set up Compodoc automatically. If you have already installed Storybook, you can set up Compodoc manually.

#### `angular.json` Configuration:
In your `angular.json`, enable Compodoc for both `storybook` and `build-storybook` builders:
```json
"storybook": {
  "builder": "@storybook/angular:start-storybook",
  "options": {
    "compodoc": true,
    "compodocArgs": ["-e", "json", "-d", "."]
  }
}
```

#### `.storybook/preview.ts` Configuration:
Go to your `.storybook/preview.ts` and add the following:
```ts
import { setCompodocJson } from '@storybook/addon-docs/angular';
import docJson from '../documentation.json';
setCompodocJson(docJson);
```

---

### 📄 The Docs Tab

When you run Compodoc, it will generate a `documentation.json` file. Storybook will then use that file to render the documentation in the **Docs tab**.

Do not forget to enable **Storybook Autodocs** in your Storybook configuration.

---

### 🏗️ Built-in Compodoc (No Separate Script Needed)

Note that Compodoc is now built into `@storybook/angular`; you don't have to call it explicitly. If you were running Compodoc in your `package.json` scripts, you can remove the related script.

---

### 📁 TypeScript Config for Compodoc (Nx Setup)

You can add the paths to the component files (where Compodoc will look for JSDoc comment blocks) in the `include` array of the `.storybook/tsconfig.json` file. This is the file that Storybook will use to compile the TypeScript files, and it will also be the file that Compodoc will use to compile the TypeScript files.

---

### ⚡ Performance Tip

In Angular, Storybook uses Compodoc to generate the documentation that will be displayed in the stories and controls. Sometimes the generation becomes slow, and it is even more noticeable if you are constantly re-doing the documentation or improving it.

To speed things up, you can add optional flags like `--minimal`, `--disablePrivate`, `--disableInternal`, `--disableSourceCode`, and `--disableLifeCycleHooks` to your `compodocArgs`.

---

### 📁 The `documentation.json` File

When Compodoc builds, it creates a large file in the root directory of the application called `documentation.json`. This file is created when Compodoc is run and does not need to be committed to git — it can safely be included in a `.gitignore` file.

### 📦 How argTypes Are Inferred via Compodoc

In the Angular framework, the `argTypes` are generated **purely from the JSON file that Compodoc outputs**, unless the story manually defines them.

If you are using the Storybook docs addon, Storybook will infer a set of `argTypes` for each story based on the component specified in the meta (or default export) of the CSF file. To do so, Storybook uses **various static analysis tools depending on your framework**.

Storybook's controls are currently not able to reliably extract the type definitions for **input signals** due to a bug in Compodoc, which is used to analyze a component's signature and input properties.

### ⚠️ Known Issue: Compodoc Args Broken in Storybook 9.0.12+

A recent (2025) bug report is relevant:

Migrating from Storybook 9.0.11 to 9.0.13 **broke the Compodoc generation**. The error is: *"No configuration file found, switching to CLI flags. `tsconfig.json` file was not found, please use `-p` flag."*

Specifying the `tsconfig` as a parameter in `compodocArgs` **does not work**, since no argument is passed at all.

---

### 🔧 Storybook 10 Migration — Key Angular Breaking Changes

Storybook 10 is a **breaking maintenance release focused on ESM-only package distribution**.

Storybook has **dropped support for Angular versions 15–17**. The minimum supported version is now **Angular 18**. If you're using an older version, you'll need to upgrade to Angular 18 or newer.

Storybook has added a new feature flag `angularFilterNonInputControls`, which **filters out non-input controls** from Angular components in Storybook's controls panel. To enable it, set the feature flag in your `.storybook/main.ts` file.

---

### ⚙️ argTypesRegex Deprecation (Actions)

In Storybook 7, Storybook inferred if a component accepts any action props by checking if the prop name starts with `onX` (e.g., `onClick`), or as configured by `actions.argTypesRegex`. If that was the case, it would fill in Jest spies for those args automatically. This behavior has since changed — `argTypesRegex` is effectively deprecated in favor of explicit `fn()` spy usage.

At the moment, Storybook only adds an action for a story that manually specifies `action: 'something'` in the `argTypes` properties, or if the property is recognized as an `@Output` in the Compodoc JSON.

## The Core Issue

The discussion centers on incomplete argument table documentation after migrating to a monorepo with NX. The user reports that their Storybook Args table, which previously displayed all component properties automatically, now appears mostly empty after the migration.

## New Behavior Identified

As explained by the collaborator Marklb, the new behavior involves a specific dependency chain:

**"In the Angular framework, the argTypes are generated purely from the json file that compodoc outputs, unless the story manually defines them."**

This means Storybook's Args table relies entirely on Compodoc's documentation generation rather than auto-discovering component properties.

## How argTypes Inference Works

Without `setCompodocJson`, the inference process requires three components:

1. **Compodoc configuration** - The build system must generate a `documentation.json` file with proper component metadata
2. **Preview setup** - The `.storybook/preview.ts` file must import and register this JSON via `setCompodocJson(docJson)`
3. **TypeScript configuration** - The tsconfig used by Compodoc must correctly resolve component file paths

## Root Cause Resolution

The issue stemmed from an incompatible TypeScript configuration created during the NX migration. Once the user merged their original tsconfig settings with the new one, the Args table populated correctly again.

## The Problem

When upgrading from Storybook 9.0.11 to 9.0.12+, the `compodoc` command fails with an error about missing `tsconfig.json` and configuration files.

## What Changed

The issue stems from a change in how arguments are passed to the compodoc executable. In version 9.0.11, the command structure was:
```
yarn compodoc -p tsconfig.json [other args]
```

In 9.0.12+, it changed to:
```
yarn exec compodoc -p tsconfig.json [other args]
```

The addition of the `exec` keyword breaks argument passing—compodoc receives no parameters at all, only the executable path.

## Root Cause

The issue is likely related to PR #31557 ("CLI: Improve support for upgrading Storybook in monorepos"), which modified how the `JSPackageManager` constructs and executes package commands in the codebase.

## The Fix

PR #32717 was opened to address this regression and restore proper argument passing to the compodoc command, ensuring parameters like `-p`, `-e`, `-d`, and `--disablePrivate` reach the executable correctly.

## `setCompodocJson` in Storybook (Angular + Compodoc) — Current Status

### ✅ Still the Official Recommended API (as of Storybook 8, 9, and the `next` branch for 10)

According to the **official Storybook documentation** and the **`next` branch source** on GitHub, `setCompodocJson` is **still the documented and recommended** way to load Compodoc data into Storybook for Angular. The current docs instruct you to add to your `.storybook/preview.ts`:
```ts
import { setCompodocJson } from '@storybook/addon-docs/angular';
import docJson from '../documentation.json';
setCompodocJson(docJson);
```

This same snippet appears in the `next` branch of the Storybook GitHub repository (`docs/get-started/frameworks/angular.mdx`), confirming it has not yet been officially removed.

---

### ⚠️ Breaking Issue Reported in Storybook 10 (v10.2.16)

There is a **reported bug** in Storybook 10 where the import path itself fails. Users running Storybook 10 (e.g., v10.2.16) are seeing TypeScript compilation errors:
```
Cannot find module '@storybook/addon-docs/angular' or its corresponding type declarations.
import { setCompodocJson } from "@storybook/addon-docs/angular";
```

This suggests that the `@storybook/addon-docs/angular` subpath may no longer be resolvable in Storybook 10, even though it hasn't been formally deprecated in the official migration notes.

---

### 🔧 How `setCompodocJson` Works Internally

Under the hood, `setCompodocJson` simply assigns the JSON to a global window variable: `window.__STORYBOOK_COMPODOC_JSON__ = compodocJson`. This is important context if you need a workaround.

---

### 💡 Early-Proposed Alternative (not yet official)

Back in a 2021 GitHub issue, a community contributor suggested that `setCompodocJson` shouldn't be part of the public API at all, and proposed a parameters-based alternative: 
```ts
// preview.ts
import compodocJson from 'path/to/compodoc/documentation.json';
export const parameters = {
  docs: {
    documentation: compodocJson
  }
}
```

However, this was **never officially adopted** by the Storybook team as a replacement.

---

### 🔄 Migration Notes: What *Has* Changed in Storybook 8/9/10

The official `MIGRATION.md` (from the `next` branch) does not mention `setCompodocJson` deprecation specifically. The notable Angular-related migration changes are:

- Compodoc is now **built into `@storybook/angular`**; you don't have to call it explicitly as a separate script. If you were running compodoc in your `package.json` scripts, you can remove the related script.
- Starting in Storybook **8.2**, `preview.js` `globals` were deprecated and renamed to `initialGlobals`. They were removed in **9.0**:
  ```js
  // Before (deprecated)
  export default { globals: { a: 1 } };
  // After
  export default { initialGlobals: { a: 1 } };
  ```

- Remove overly permissive signatures from third-party types.

(The actual value type, `unknown`, is irrelevant and could be any type. Only the key type matters.)

```

Instead of causing a type error like the above, you can also use a [conditional type](https://www.typescriptlang.org/docs/handbook/2/conditional-types.html) to test whether a type is assignable to another:

```
type Indexed = {} extends Record<string, unknown>
	? '✅ `{}` is assignable to `Record<string, unknown>`'
	: '❌ `{}` is NOT assignable to `Record<string, unknown>`';
// => '✅ `{}` is assignable to `Record<string, unknown>`'

type Keyed = {} extends Record<'foo' | 'bar', unknown>
	? "✅ `{}` is assignable to `Record<'foo' | 'bar', unknown>`"
	: "❌ `{}` is NOT assignable to `Record<'foo' | 'bar', unknown>`";
// => "❌ `{}` is NOT assignable to `Record<'foo' | 'bar', unknown>`"

import type {RemoveIndexSignature} from 'type-fest';

type RemoveIndexSignature<ObjectType> = {
	[KeyType in keyof ObjectType // Map each key of `ObjectType`...
	]: ObjectType[KeyType]; // ...to its original value, i.e. `RemoveIndexSignature<Foo> == Foo`.
};

...whether an empty object (`{}`) would be assignable to an object with that `KeyType` (`Record<KeyType, unknown>`)...

		as {} extends Record<KeyType, unknown>
