**Phase 1: Orient (PLAN-01)**

Write an `<orient>` XML section with these instructions:

    <orient>
    ## Phase 1: Orient

    Explore the codebase to understand the task scope and current state.

    - Read files relevant to the user's request
    - Examine directory structure and dependencies
    - Identify constraints, existing patterns, and integration points
    - Note what exists and what needs to change

    Do not write code or make changes during orientation. Do not consult
    the advisor yet -- orientation is preparation, not substantive work.
    </orient>

Use goal-oriented framing per CONTEXT.md discretion area: "understand the task scope and codebase state" rather than prescriptive ("run these specific glob patterns"). The executor (Sonnet 4.6) is competent at exploration when given a clear goal.

**Phase 2: Consult (PLAN-02, PLAN-03, D-02, D-03)**

Write a `<consult>` XML section with these instructions:

    <consult>
    ## Phase 2: Consult the Advisor

    Invoke the lz-advisor:advisor agent via the Agent tool with a focused
    prompt containing:

    1. The user's original task (quote their request)
    2. Key findings from orientation (files examined, patterns found,
       constraints discovered)
    3. A specific question: "Given these findings, what approach and
       sequence of steps would you recommend?"

    Keep the prompt summarized -- do not paste entire files. The advisor
    starts with a fresh context and cannot see the conversation. All
    relevant context goes in the prompt.

    One advisor consultation per plan invocation. The advisor returns
    concise strategic direction (enumerated steps). This becomes the
    foundation of the plan.
    </consult>

Key points to include:
- Explicit context packaging instructions (task, findings, question) -- addresses subagent context isolation
- "Keep the prompt summarized" -- anti-pattern 5 (no raw file dumps)
- "One advisor consultation per plan invocation" -- D-02 locks this
- Do NOT include word count constraint in skill body -- the advisor agent system prompt already enforces under 100 words (ADVR-02); duplicating it wastes skill tokens

**Phase 3: Produce (PLAN-04, PLAN-05, D-01, D-04)**

Write a `<produce>` XML section with these instructions:

    <produce>
    ## Phase 3: Produce the Plan

    Expand the advisor's enumerated guidance into a detailed, actionable
    plan. Write the plan to a markdown file.

    ### Plan File Location

    Write to: `plan-<task-slug>.md` in the project root directory.
    Use a short kebab-case slug derived from the task description.
    Example: for "add user authentication", write `plan-add-user-auth.md`.

    ### Plan File Format

    The plan file contains these sections:

    # Plan: <Task Title>

    ## Strategic Direction

    <The advisor's guidance, attributed and quoted. Present the advisor's
    enumerated steps as the strategic foundation for this plan.>

    ## Steps

    1. **<Step title>**
       - File: `<path/to/file>`
       - Change: <specific description of what to create or modify>
       - Rationale: <why this step, connecting to advisor guidance>

    2. ...

    (Continue for all steps. Each step includes file paths and specific
    changes -- matching the granularity of Claude Code's native /plan
    output.)

    ## Key Decisions

    - **<Decision>**: <rationale from advisor guidance and executor analysis>
    - **<Risk>**: <what could go wrong and how to mitigate>

    ## Dependencies

    <Order constraints between steps, if any. Note which steps can be
    done in parallel and which are sequential.>

    Present the plan file path to the user after writing it.
    </produce>

Key points:
- Plan file naming convention: `plan-<task-slug>.md` in project root (RESEARCH.md Assumption A1; discoverable by execute skill for IMPL-05)
- Explicit section template: Sonnet 4.6 follows literal format instructions (MODEL-OPTIMIZATION-SONNET.md; D-04)
- Strategic Direction section preserves the Opus contribution visibly (CONTEXT.md specifics)
- Steps section matches `/plan` granularity (numbered, file paths, specific changes) per D-04
- Key Decisions section includes approach rationale and risk warnings per D-04
- Dependencies section for execution ordering

**Closing paragraph:**
After writing the plan file, present the file path to the user and note that the plan can be reviewed, edited, and then passed to `/lz-advisor.execute` for implementation.

- No aggressive language ("MUST", "CRITICAL", "ALWAYS")
- No duplicated timing/weight/packaging guidance from advisor-timing.md (loaded via @references/ include)
- No `disable-model-invocation` in frontmatter (Pitfall 5 -- GitHub #26251 bug)
