**Why this exact shell:**
- `sed '/^\*\*Critical:\*\*/,$d' "$OUT_E"` deletes from the first line starting with `**Critical:**` through EOF. Line-anchored (`^`) protects against indented fences (RESEARCH.md Risk R2 mitigation).
- `| wc -w` counts words on the remaining content.
- `| tr -d ' '` strips leading-space padding that `wc` adds on some platforms (R-02 empirical).
- `"$WC" -le 100` is numeric-integer comparison; passes when the strip result is 100 or fewer words.
- `[OK]` / `[ERROR]` prefixes match the existing D, E, F style.
- Uses `$OUT_E` (thin-context plan call) rather than `$OUT_D` because E has Assuming frames which exercise the longest realistic numbered-list shape; D is calibrated to NOT produce Critical, so D's Strategic Direction is shorter and might pass trivially (per RESEARCH.md Open Question 4: single OUT_E assertion recommended).

**Edge case handling confirmed from RESEARCH.md R-02:**
1. Critical block absent -> sed deletes nothing -> counts full response (correct; still gated at 100).
2. Multiple Critical blocks -> sed deletes from FIRST to EOF (correct).
3. Empty advisor response -> `wc -w` returns 0 -> passes gate (correct; zero words is under 100).
4. Advisor preamble ("Let me verify...") -> caught indirectly by Finding E's `Assuming .*\(unverified\)` assertion, not by the word-count gate.
