## R-02: Git Bash `wc -w` portability

### Empirical test (Git Bash on Windows arm64 / MSYS2)

Tested four approaches on `tmp-s1-block.txt` (the verbatim S1 advisor response, 10 lines, 117 words full, 80 words with `**Critical:**` stripped):

| Approach | Command (abbreviated) | Full count | Critical-stripped count |
|----------|-----------------------|------------|-------------------------|
| 1. `wc -w` on file | `wc -w tmp-s1-block.txt` | 117 | - |
| 2. `wc -w` via cat pipe | `cat tmp-s1-block.txt \| wc -w` | 117 | - |
| 3. `wc -w` via herestring | `wc -w <<< "$(cat tmp-s1-block.txt)"` | 117 | - |
| 4. `awk '{n+=NF} END {print n}'` | `awk '{n+=NF} END {print n}' tmp-s1-block.txt` | 117 | - |
| 5. Node `split(/\s+/).filter(Boolean).length` | `node -e "..."` | 117 | - |
| 1+sed strip | `sed '/^\*\*Critical:\*\*/,\$d' tmp-s1-block.txt \| wc -w` | - | 80 |
| 4+sed strip | `sed '/^\*\*Critical:\*\*/,\$d' file \| awk '{n+=NF} END {print n}'` | - | 80 |
| 5+JS strip | Node regex `/\*\*Critical:\*\*[\s\S]*$/` replace, then split | - | 80 |

All five approaches produce identical counts on this multi-line input. No newline-handling divergence.

### Recommended D-05 approach

**Recommend approach 1 (sed + wc -w pipeline)**, with this exact command inside the smoke-test block:

```bash
WC=$(sed '/^\*\*Critical:\*\*/,$d' "$ADVISOR_OUT" | wc -w | tr -d ' ')
if [ "$WC" -gt 100 ]; then
  echo "[ERROR] Finding S1-S4 word budget: advisor Strategic Direction $WC words (>100)"
  FAIL=1
else
  echo "[OK] Advisor word-budget: $WC words (<=100)"
fi
```

**Rationale:**
- Uses only POSIX tools (`sed`, `wc`, `tr`) -- no Node dependency, no bashisms.
- `tr -d ' '` strips the leading-space padding `wc` adds on some platforms.
- `sed '/^\*\*Critical:\*\*/,$d'` deletes from the first `**Critical:**` line to EOF, exactly mirroring D-15's budget-exclusion semantics.
- Matches the existing DEF script style (shell pipeline, `rg`-style assertion with `[OK]` / `[ERROR]` prefix).
- Portable to any Git Bash / macOS / Linux agent running `claude -p`.

**Edge case the planner must handle in D-05 acceptance criteria:**
1. Critical block absent -> sed deletes nothing -> counts full response (correct).
2. Multiple Critical blocks (unlikely but possible) -> sed deletes from FIRST occurrence to EOF, which includes subsequent Criticals (correct).
3. Code fences containing `**Critical:**` inside the numbered list -> false-positive strip. Mitigation: the advisor.md contract places Critical AFTER the numbered list; a fence inside numbered content should never contain the literal `**Critical:**` token at column 0. Acceptance criterion: anchor the sed pattern to line-start (already done via `^`).
4. Empty advisor response -> `wc -w` returns 0 -> passes gate correctly.

### Upgrade path (if the simple pipeline fails in CI)

If a future environment has a `wc` that counts differently (e.g., BusyBox on Alpine containers), fall back to a Node one-liner:

```bash
WC=$(node -e "const fs=require('node:fs');const t=fs.readFileSync(0,'utf8').replace(/\*\*Critical:\*\*[\s\S]*\$/,'');process.stdout.write(String(t.split(/\s+/).filter(Boolean).length))" < "$ADVISOR_OUT")
```

Per CLAUDE.md: uses fd 0 (stdin), not `/dev/stdin`, so it works across Git Bash translation quirks.

## Implementation risks and mitigations

### Risk R1: Qualitative cue (D-02, D-08) drift under Sonnet

**Risk:** D-02 and D-08 are both qualitative (no numeric thresholds, no keyword lists per CONTEXT.md positive-framing convention). Sonnet 4.6 literal-following is good but not perfect. A cue like "when you identify a framework in the task, surface at least one framework-convention claim" could be interpreted as "only if the user names the framework" vs "if the task touches any framework ecosystem."

**Mitigation:** Include a worked example in the cue itself. The existing D-14 sentence ("When dependency behavior is load-bearing, verify and surface the result as a Pre-Verified Package Behavior Claim") works because the word "load-bearing" is qualitative and the adjacent guidance (targeted reads, specific function/config entry) calibrates the discipline. D-02 should follow the same shape. Acceptance criterion for planner: the cue MUST include at least one example shape of a framework-convention claim (e.g., "a claim about build-tool caching, dependency ordering, or lifecycle convention").

### Risk R2: D-05 word-count fragility on edge cases

**Risk list (verified in R-02 exploration):**
1. **Multi-line `**Critical:**` block** -- `sed '/^\*\*Critical:\*\*/,$d'` handles multi-line correctly because the range `,$` extends to EOF.
2. **No Critical block present** -- sed deletes nothing; counts full response (correct).
3. **Code fences containing `**Critical:**` inside numbered list** -- false positive strip. Mitigation: advisor.md contract places Critical AFTER the numbered list; fenced content inside numbered list should never contain literal `**Critical:**` at column 0. The `^` anchor in the sed pattern protects against indented matches.
4. **Empty advisor response** -- `wc -w` returns 0; passes gate correctly.
5. **Advisor emits preamble-then-1-point** -- advisor.md lines 56-58 forbid preamble ("no preamble, no intent-announcing phrases"); smoke test E already catches preamble drift indirectly via the "Assuming X" assertion.

**Acceptance criterion for planner:** add a comment above the word-count block in the smoke test explaining that `**Critical:**` stripping is line-anchored (not string-anchored) and that the gate is 100 post-strip.

### Risk R3: D-06 G/H re-using F-block's `claude -p` session

**Risk:** If the planner decides to reuse the F-block session (two assertions against one `claude -p` output instead of three separate invocations), the assertion would test `/lz-advisor.review` output against `### Threat Patterns` -- which is a security-reviewer header. Review-skill output would fail both G and H structurally (wrong skill), masking the real question of whether security-reviewer emits the two slots.

**Mitigation:** Use a separate `claude -p` invocation targeting `/lz-advisor.security-review`, as recommended above. Three `claude -p` calls per smoke run; ~$2-3 per full script execution; acceptable cost for per-CI-run coverage.

### Risk R4: S4 Bun crash re-occurrence
