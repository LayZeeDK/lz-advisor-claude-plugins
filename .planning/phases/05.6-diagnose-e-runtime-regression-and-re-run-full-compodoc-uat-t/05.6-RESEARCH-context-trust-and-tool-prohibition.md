**Researched:** 2026-04-25
**Domain:** Claude Code SKILL.md prompt engineering for tool-use suppression and context-grounding contracts
**Confidence:** HIGH on Anthropic-published patterns; MEDIUM-HIGH on Pink Elephant tradeoff; LOW on empirical magnitude (no public A/B benchmark for this exact failure mode)

## Executive Summary

Plan 05's fix failed because `prefer WebFetch over reading node_modules` is a *comparative preference* that mentions the forbidden behavior. Sonnet 4.5/4.6 has documented over-investigation tendencies and treats comparative preferences as soft hints; the bullet's mention of `node_modules` may also Pink-Elephant-prime the very behavior it's trying to suppress.

**Top 3 candidate fixes, ranked:**

1. **(Recommended)** Add a `<context_trust_contract>` XML block to the Orient phase that detects inlined authoritative documentation and *positively prescribes* using it as ground truth. Pattern lifted from this project's own `reviewer.md` Context Trust Contract section (already proven by green DEF F/G/H assertions), structurally parallel to Anthropic's published `<investigate_before_answering>` template.
2. **(Complementary, do both)** Restructure the Orient exploration ranking as an XML-tagged decision tree (`<orient_exploration_ranking>`) instead of a free-prose bullet. Anthropic's Sonnet 4.5+ system prompts switched to XML-tagged sections specifically because the model parses tag-bounded constraints more reliably than prose.
3. **(Defense-in-depth, narrow value)** Tighten `allowed-tools` frontmatter. Material caveat: Phase 5.6 UAT runs with `--dangerously-skip-permissions`, which bypasses every permission rule, so this provides zero protection in UAT but may help non-UAT interactive sessions. Defer.

**Recommendation: land #1 + #2 byte-identically across all four SKILL.md files. Defer #3.**

## Source-grounded findings

### 1. Anthropic explicit guidance: positive over negative instructions

> "**Tell Claude what to do instead of what not to do.** Instead of: 'Do not use markdown in your response'. Try: 'Your response should be composed of smoothly flowing prose paragraphs.'"

> "**Less effective:** 'NEVER use ellipses'. **More effective:** 'Your response will be read aloud by a text-to-speech engine, so never use ellipses since the text-to-speech engine will not know how to pronounce them.' ... Claude is smart enough to generalize from the explanation."

[CITED: [Anthropic prompting best practices](https://platform.claude.com/docs/en/build-with-claude/prompt-engineering/claude-prompting-best-practices)]

**Implication:** Plan 05's "prefer X over Y" bullet violates this twice. It is comparative (not primary), and it does not state when Orient ends. The model interprets it as a tool-selection tip, not a phase-termination criterion. The fix must be a *primary positive instruction* about what to do — and a clear exit condition for Orient.

**Confidence:** HIGH (multiple verbatim Anthropic citations).

### 2. Anthropic's `<investigate_before_answering>` template — and its inverse

Anthropic's anti-hallucination template:

> ```
> <investigate_before_answering>
> Never speculate about code you have not opened. If the user references a specific file, you MUST read the file before answering. Make sure to investigate and read relevant files BEFORE answering questions about the codebase. Never make any claims about code before investigating ...
> </investigate_before_answering>
> ```

[CITED: [same Anthropic page, "Minimizing hallucinations in agentic coding"](https://platform.claude.com/docs/en/build-with-claude/prompt-engineering/claude-prompting-best-practices)]

**This is exactly the failure mode lz-advisor is hitting.** Sonnet has internalized "investigate before answering" so strongly that it over-applies even when the user prompt has inlined the authoritative answer. The fix is a *bounded inverse* — a tag that switches investigation off when the authoritative answer is already in context.

Anthropic publishes a structurally parallel inverse pattern in the same docs page:

> ```
> <do_not_act_before_instructions>
> Do not jump into implementation or change files unless clearly instructed ... When the user's intent is ambiguous, default to providing information, doing research, and providing recommendations rather than taking action.
> </do_not_act_before_instructions>
> ```

This shows the canonical Anthropic shape: bounded XML tag + positive scope condition (`when X, default to Y`) + concrete substitute action. Apply the same shape for context trust.

**Confidence:** HIGH (template structure is verbatim Anthropic-published).

### 3. Pink Elephant Problem: bare prohibitions can backfire

- [arXiv 2402.07896 — Suppressing Pink Elephants with Direct Principle Feedback](https://arxiv.org/abs/2402.07896) [CITED]: formalizes the case where instructing an LLM to avoid an entity *increases* the entity's surfacing probability.
- [arXiv 2404.15154 — Do not think about pink elephant!](https://arxiv.org/html/2404.15154) [CITED]: mechanistic explanation. Attention-based architectures cannot cleanly represent absence of a concept; the negation token does not subtract the concept from the activation space.
- [eval.16x.engineer Pink Elephant analysis](https://eval.16x.engineer/blog/the-pink-elephant-negative-instructions-llms-effectiveness-analysis) [CITED]: practical observation — Claude Code creating duplicate files despite `NEVER create duplicate files` rules in CLAUDE.md is a documented manifestation.

**Implication for Plan 06:** a bare `DO NOT use rg/Read against node_modules/**` directive may *worsen* the regression. Mitigations from the literature:

- Pair every prohibition with a concrete positive substitute.
- Use *bounded scope* ("when X, do Y") rather than blanket `DO NOT do Z`.
- Use sparingly — multiple stacked `don't` clauses degrade output.

**Confidence:** HIGH on the principle; LOW on the magnitude for our specific case (cited papers test entity-avoidance, not tool-selection in agentic loops). Treat as a constraint on solution shape.

### 4. Sonnet 4.5/4.6 specific quirks: over-investigation is a documented tendency

Verbatim Anthropic guidance:

> "Claude Opus 4.6 does significantly more upfront exploration than previous models, especially at higher `effort` settings ... If your prompts previously encouraged the model to be more thorough, you should tune that guidance for Claude Opus 4.6: **Replace blanket defaults with more targeted instructions.** Instead of 'Default to using [tool],' add guidance like 'Use [tool] when it would enhance your understanding of the problem.'"

> "Claude Opus 4.6 has a strong predilection for subagents and may spawn them in situations where a simpler, direct approach would suffice. For example, the model may spawn subagents for code exploration when a direct grep call is faster and sufficient."

> "`high` or `xhigh` effort settings show substantially more tool usage in agentic search and coding."

[CITED: [Anthropic best practices](https://platform.claude.com/docs/en/build-with-claude/prompt-engineering/claude-prompting-best-practices), "Overthinking", "Subagent orchestration", "Tool use triggering" sections]

**Anthropic's published mitigation pattern uses bounded, motivation-paired guidance** (the "Overeagerness" template uses `Don't X` clauses but each is paired with positive scope and motivation). This validates a hybrid Pattern A approach: bounded prohibition is acceptable when paired with positive substitute and concrete reasoning.

**Sonnet vs. Opus:** UAT data confirms Sonnet/high regresses *worse* than Sonnet/medium (86 pre-advisor tool uses at `high`). This matches Anthropic's documented "high effort = more tool use" pattern. Opus 4.7 specifically "has a tendency to use tools less often than Claude Opus 4.6 and to use reasoning more" — Opus would likely behave better, but switching the executor to Opus violates the lz-advisor cost contract.

**Confidence:** HIGH (Anthropic explicitly documents this tendency and publishes mitigation templates).

### 6. Claude Code path-scoped allowed-tools: limited value under UAT

Per [Claude Code permissions docs](https://code.claude.com/docs/en/permissions) [CITED]:

- Syntax exists: `Read(node_modules/**)`, `Bash(node -e *)`, `WebFetch(domain:example.com)`. Rules evaluated `deny -> ask -> allow`.
- Verbatim caveat: "**Read and Edit deny rules apply to Claude's built-in file tools, not to Bash subprocesses. A `Read(./.env)` deny rule blocks the Read tool but does not prevent `cat .env` in Bash.**"
- Verbatim caveat: "**Bash permission patterns that try to constrain command arguments are fragile.**"
- `allowed-tools` in SKILL.md frontmatter is CLI-only [CITED: [agentpatterns.ai SKILL.md reference](https://www.agentpatterns.ai/tool-engineering/skill-frontmatter-reference/)].

**Critical for our case:** Phase 5.6 UAT replay invokes `claude -p` with `--dangerously-skip-permissions` (per `05.6-CONTEXT.md` D-08), which bypasses every permission rule. So `allowed-tools` narrowing has *zero effect under UAT* and would create a false sense of safety. Defer to a future phase as defense-in-depth for non-UAT interactive sessions.

**Confidence:** HIGH on the syntax; HIGH on the bypass caveat.

## Concrete prompt patterns (copy-pasteable)

Designed byte-identical across plan, execute, review, security-review SKILL.md per Phase 5.4 canon.

### Pattern A: `<context_trust_contract>` block, FIRST child of `<orient>`

```markdown
<context_trust_contract>
Before reading any file, scan the user prompt for an inlined authoritative source block. An authoritative source block is any of:

1. A `---` delimited block at the top of the user message containing pasted documentation, specification text, release notes, or a published guide (canonical context-packed format).
2. A clearly-marked quoted block of pasted documentation (e.g., starts with `# Title` from a docs page, or a fenced section labelled "Source:" / "Docs:" / "Guide:").
3. A `<fetched source="...">...</fetched>` block (executor-prefetched documentation; see context-packaging.md Rule 5a).

When such a block is present, treat its content as ground truth for the public-API and framework-convention questions it answers. Your Orient phase ends as soon as you have parsed the block plus read the local project files needed to compose the consultation. Specifically, when an authoritative source is present:

- The consultation packaging step (Phase 2) is the next action, not further verification.
- Local project reads (project.json, package.json, .storybook/main.ts, src/**) are still in scope -- the authoritative source describes the library, not your project.
- Reading inside `node_modules/` is out of scope. The authoritative source is the contract for the library's public API; the compiled `dist/` is implementation detail you will not infer correctly from minified chunks anyway.
- WebFetch and WebSearch against the same library are out of scope for the same reason -- the source is already in context.

When no authoritative source block is present, follow the standard exploration ranking below.
</context_trust_contract>
```

**Why this works:**
- Positively framed (`your Orient phase ends as soon as ...`) instead of `do not read node_modules`.
- XML tag boundary the model parses reliably (Section 7).
- Mirrors the in-repo `reviewer.md` Context Trust Contract that is empirically working (Section 5).
- **Bounded** — prohibition fires only when an authoritative source is present. Pink Elephant risk mitigated because the prohibition is conditional, not blanket (Section 3).
- States *what to do instead* (Phase 2 packaging is next action) — direct application of Section 1 guidance.

### Pattern B: `<orient_exploration_ranking>` block, SECOND child of `<orient>`

Replaces the current Plan 05 bullet:

```markdown
<orient_exploration_ranking>
When you need information about a third-party library, framework, or public API beyond what the authoritative source block provides, take ONE of the following actions:

1. **Local-project read first** -- if the question is about how *your project* uses the library (config files, existing usage patterns), read project files only: `project.json`, `package.json`, `.storybook/`, `src/`, `tsconfig*.json`. Stop when the project-side question is answered.

2. **WebFetch for public-API questions** -- if the question is about *the library's documented behavior* and no authoritative source block was inlined, WebFetch the official docs. The library's homepage, GitHub README, release notes, and migration guides are all valid first-fetch targets.

3. **WebSearch for version/compatibility questions** -- when you don't know the right docs URL, WebSearch with the library name + version + the specific API or symbol.

4. **`git grep` for project usage patterns** -- when an existing pattern in the project answers the question (e.g., "how does this project already configure Storybook addons?"), `git grep` against project source.

If none of steps 1-4 produces the answer you need, name the gap explicitly in the consultation Findings section and proceed. Do not extend Orient indefinitely. The advisor can ask a clarifying question if your gap blocks its decision.
</orient_exploration_ranking>
```

**Why this works:**
- Numbered decision tree (positive instructions) instead of "prefer X over Y" (preferential).
- Each step has explicit entry conditions and exit criteria.
- Step 4 gap-surfacing converts "I should investigate more" into "I should declare the gap and consult."
- **No mention of `node_modules`.** Per Pink Elephant principle (Section 3), absence of mention is stronger than `DO NOT read node_modules`.

### Pattern C: Replace Rule 5 in `references/context-packaging.md`

```markdown
5. **Verify external claims; trust packaged ones.** If your consultation prompt asserts anything about third-party package behavior or framework semantics, verify the claim during Orient and include the verification in a Pre-Verified Package Behavior Claims block. Verification ranking is defined in the skill's `<orient_exploration_ranking>` block. When the user has inlined an authoritative source (see `<context_trust_contract>` in the skill), the source IS the verification -- no further fetch or read is required, and re-verifying it via tool use wastes the executor's tokens and the advisor's turn budget.
```

(Rule 5a `<fetched>` wrapping unchanged. Rule 6 tool-failure recovery unchanged.)

## Risks and tradeoffs

**Risk 1 — Pink Elephant on the new block itself.** Pattern A mentions `node_modules/` once (in the "out of scope" clause). Deliberate tradeoff: removing the explicit mention risks the model not understanding what "extending verification beyond the source" means in practice; including it once *in bounded scope* is the minimum-mention compromise. If UAT replay shows even this triggers the regression, fall back to a version that names only the positive substitute. **Probability:** LOW. Bounded-scope mention is qualitatively different from blanket `DO NOT`.

**Risk 2 — Block doesn't fire because executor doesn't recognize the inlined-doc pattern.** Pattern A detects authoritative-source blocks via three heuristics (`---` delimiter, "Source:" labels, `<fetched>` tags). UAT prompt format is `/lz-advisor.plan <one sentence>:\n---\n<authoritative doc>` — the `---` heuristic should match. For non-canonical formats, the executor falls to Pattern B exploration ranking — still tighter than Plan 05's bullet. **Probability:** MEDIUM for non-canonical prompts; acceptable degradation.

**Risk 3 — Permission-layer enforcement is unavailable under `--dangerously-skip-permissions`.** Phase 5.6 UAT replay bypasses every permission rule, so `allowed-tools` narrowing (candidate fix #3) has zero protection in UAT. **Probability:** N/A — architectural limitation. Implication: prompt-layer fix (Pattern A + B) is the only fix that survives UAT replay.

**Risk 4 — Symmetry break with review and security-review.** Byte-identical-symmetry rule (Phase 5.4 canon) applies. Review and security-review don't consult external library docs as their primary task; they validate findings. The Pattern A block still applies (their authoritative-source blocks are CLAUDE.md excerpts and project guidelines). Adapt language to be skill-agnostic ("when an authoritative source is inlined" works for any of the four skills). **Probability:** LOW.
