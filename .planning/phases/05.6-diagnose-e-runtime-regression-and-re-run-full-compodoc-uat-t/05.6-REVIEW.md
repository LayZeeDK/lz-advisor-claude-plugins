## Info

### IN-01: Verification template lacks a placeholder header for the new "Prior Advisor Guidance" block

**File:** `plugins/lz-advisor/references/context-packaging.md:197-237`
**Issue:** Rule 7 (line 58) instructs the executor to add a `## Prior Advisor Guidance` block to the Verification template's prompt structure (quote: "...as a dedicated `## Prior Advisor Guidance` block (Verification template)"), but the Verification Consultation Template's static structure block (lines 197-237) does not include that section, even as an `(if any)` placeholder. The Proposal template already models the right pattern with `## Source Material (if any)` (line 101) and `## Pre-Verified Package Behavior Claims (if any)` (line 107) -- conditional blocks that pre-render in the template so the executor sees the placement at a glance. The Verification template should mirror that convention.

The current state is workable -- Rule 7 is the source of truth and a careful executor will read it -- but a placeholder in the rendered Verification template would reduce the chance of an executor placing the prior-SD block in the wrong position (for example, after `## Findings` rather than before, where it informs scope). Most mid-execute and final-review consultations are Verification-shaped, so this is the higher-traffic template.

**Fix:** Add a placeholder block to the Verification Consultation Template structure (between `## Pre-Verified Package Behavior Claims (if any)` on line 207 and `## Findings` on line 220):

```
## Prior Advisor Guidance (if any)
[Verbatim Strategic Direction from prior consultations within the same
skill invocation. Use literal `--- Prior Strategic Direction (consultation N) ---`
/ `--- End Prior Strategic Direction ---` markers. See Common Contract
rule 7. Include only agent-emitted Strategic Direction (numbered items
plus any **Critical:** block), not full prior prompts.]
```

A matching placeholder under the Proposal template's `## Source Material (if any)` is optional -- Rule 7 says prior SD goes "inside the new prompt's `## Source Material` section (Proposal template)", which already exists, so the placement is unambiguous there.

### IN-02: Rule 7 routes prior SD into "Source Material" but the section description still characterises that block as user-pasted or fetched content

**File:** `plugins/lz-advisor/references/context-packaging.md:101-105`
**Issue:** Rule 7 (line 58) tells the executor to embed prior Strategic Direction "inside the new prompt's `## Source Material` section (Proposal template)". The Proposal template's `## Source Material (if any)` block description (lines 102-105) currently characterises that section narrowly:

> User-pasted docs, error messages, or third-party content. Preserved verbatim. Use `<fetched source="<URL>" trust="untrusted">...</fetched>` XML tags for web content the executor pre-fetched during orientation. Use BEGIN/END delimiters or blockquotes for user-pasted material.

Prior Strategic Direction is neither user-pasted nor fetched -- it is executor-collected output from a previous advisor turn within the same skill invocation. The literal `--- Prior Strategic Direction (consultation N) ---` markers introduced by Rule 7 are distinguishable from user-pasted and fetched content, so there is no functional ambiguity, but a reader following the Proposal template top-down will not see the prior-SD use case until they cross-reference Rule 7. This is a soft cohesion gap: Rule 7 widens the semantics of `## Source Material` without updating the section description.

**Fix:** Append a sentence to the `## Source Material (if any)` description noting the third permitted content type:

```
## Source Material (if any)
[User-pasted docs, error messages, or third-party content. Preserved
verbatim. Use `<fetched source="<URL>" trust="untrusted">...</fetched>`
XML tags for web content the executor pre-fetched during orientation.
Use BEGIN/END delimiters or blockquotes for user-pasted material. When
forwarding prior Strategic Direction from an earlier consultation within
the same skill invocation, use the literal markers defined in Common
Contract rule 7.]
```

This is purely a doc-completeness improvement; runtime behavior of an executor following Rule 7 is correct as-is.
