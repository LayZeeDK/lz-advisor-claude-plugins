## Review Summary

Reviewed: `review-src/handler.ts` and `review-src/validate.ts` (commit `3c296d5` -- "add unvalidated handler with JSON.parse")

### Findings

**Finding 1 -- Confirmed, Critical.** `JSON.parse` is well-known to throw `SyntaxError` on malformed input; with no try/catch the exception escapes `handle` and propagates to whatever framework middleware invoked it. Root cause: the handler treats parsing as infallible. In an HTTP context the practical manifestation is a 500 response (or an unhandled promise rejection if wrapped in async middleware) for any client that posts non-JSON -- trivially triggered by a typo or wrong `Content-Type`. Severity upheld: user-facing crash path on normal inputs.

**Finding 2 -- Confirmed, Important.** The truthy check is a real defect: a JSON body of `0`, `false`, or `""` is valid per RFC 8259 and would be dropped as `null`. However, practical impact depends on whether callers actually send scalar bodies; most REST APIs send objects. Severity appropriate -- correctness gap on a specific edge case, not a universal crash.

**Finding 3 -- Confirmed, Important.** Accurate framework analysis. Express's `express.json()` sets `req.body` to a parsed object; `JSON.parse({...})` coerces to `"[object Object]"` and throws. This compounds Finding 1 -- the same missing guard causes two distinct failure modes. Severity upheld.

**Finding 4 -- Confirmed, Suggestion (downgrade from Important).** Dead code is a maintainability smell, not a correctness defect in itself. The validator's absence only becomes "important" if its intended contract (rejecting null/undefined bodies) is a requirement -- which is not established in the packaged context. Downgraded because no behavior is currently wrong solely due to the missing wiring; the handler's actual bugs are Findings 1-3.

### Cross-Cutting Patterns

Findings 1, 2, and 3 share a single root cause: **the handler makes unchecked assumptions about the shape and validity of `req.body`**. It assumes the value is present-when-truthy (Finding 2), always a string (Finding 3), and always well-formed JSON (Finding 1). A single input-validation boundary at the top of `handle` -- checking `req.body != null`, branching on `typeof req.body === 'string'`, and wrapping `JSON.parse` in try/catch -- would close all three. Finding 4 is the structural mirror of the same gap: a validator exists in the same commit but was never wired in, suggesting the author intended boundary validation and forgot to connect it. The commit message ("add unvalidated handler") implicitly acknowledges this. The systemic theme is missing input-boundary discipline; the fix is one consolidated guard, not four separate patches.
