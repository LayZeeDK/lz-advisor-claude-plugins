## Warnings

### WR-01: advisor.md Edge Cases examples do not match the mandated sentence frame

**File:** `plugins/lz-advisor/agents/advisor.md:141-145`

**Issue:** The Edge Cases section mandates a literal sentence frame `Assuming X (unverified), do Y. Verify X before acting.` with explicit instruction to "substitute X (the unverified premise) and Y (the action) as appropriate" and to "keep the words `Assuming`, `(unverified)`, `do`, `Verify`, and `before acting.` intact". The examples that follow violate both the substitution rule and the punctuation:

- Example 1: `Assuming X (unverified) the frontend sends cookies over HTTPS only, do Y add the Secure and HttpOnly flags on session cookie emission. Verify X before acting.`
- Example 2: `Assuming X (unverified) the CI environment has Node 20+, do Y use the built-in fetch API directly. Verify X before acting.`

Three concrete deviations:

1. **X and Y are not substituted -- they are appended to.** The literal letters `X` and `Y` remain in the example alongside the premise/action clause. The mandate says to substitute them. This is internally contradictory.
2. **Comma placement drifts.** The frame has `(unverified),` (comma adjacent). The examples have `(unverified) the frontend...` with the comma displaced to after the premise clause (`...HTTPS only, do Y`). The comma in the mandated frame is therefore not preserved intact in either example.
3. **Period after `Y` is missing.** The frame has `do Y.` (period). The example has `do Y add the Secure...` with no period until the end of the action clause. The period-after-Y in the mandated frame is not preserved.

Opus following these examples will emit text that neither matches the literal grep pattern the executor is specified to route on nor matches the "keep intact" rule. This defeats the purpose of the E-runtime contract.

**Fix:** Rewrite both examples so they consistently substitute X and Y (do not keep the letters). Two concrete options:

Option A -- full substitution (preferred; matches the "substitute" directive):

```
- `Assuming the frontend sends cookies over HTTPS only (unverified), add the Secure and HttpOnly flags on session cookie emission. Verify the frontend sends cookies over HTTPS only before acting.`
- `Assuming the CI environment has Node 20+ (unverified), use the built-in fetch API directly. Verify the CI environment has Node 20+ before acting.`
```

If option A is chosen, drop `do` from the "keep intact" list (it reads naturally only after short Y clauses) or change the frame itself to `Assuming X (unverified), Y. Verify X before acting.` to match reviewer/security-reviewer (see WR-03).

Option B -- keep X and Y as short named placeholders that are substituted for short tokens (not full clauses), preserving `do`:

```
- `Assuming HTTPS-only cookies (unverified), do add Secure and HttpOnly flags on session cookie emission. Verify HTTPS-only cookies before acting.`
- `Assuming Node 20+ in CI (unverified), do use the built-in fetch API directly. Verify Node 20+ in CI before acting.`
```

Either way, the examples must fully replace the literal `X` and `Y` letters, preserve `(unverified),` as a unit, and place a period immediately after the Y clause.

### WR-02: advisor.md Edge Cases lacks a forbidden-phrase list to catch paraphrase drift

**File:** `plugins/lz-advisor/agents/advisor.md:138-147`

**Issue:** The positive mandate ("MUST begin with the literal sentence frame...") is present, but there is no enumerated list of forbidden paraphrase drift patterns an Opus model is likely to emit instead. Without such a list, the model is free to inject plausible-looking alternatives that the executor's grep will not match. The scope_note explicitly asked whether the forbidden-phrase list fully covers paraphrase drift -- the answer today is that no forbidden-phrase list exists. The section only bans clarification requests in prose (line 139).

Likely drift patterns:

- `It depends on whether X...` (pure clarification hedge)
- `If X, then Y; otherwise Z` (conditional with no `Assuming` / `unverified` marker)
- `Could you clarify whether X?` (once banned in prose, but easy to forget without a bullet)
- `Assuming X is true, do Y` (drops `(unverified)`)
- `Assuming X (unconfirmed), do Y` (paraphrases `(unverified)` as `(unconfirmed)`)
- `Assuming X (unverified), Y` (drops `do`; also matches the reviewer/security-reviewer form, confusing)
- `Given X, do Y` (replaces `Assuming` with `Given`)
- A separate `## Assumptions` or bulleted `Assumptions:` section (forbidden once in prose; easy to miss)

**Fix:** Add a `Do NOT emit:` sub-list immediately after the sentence-frame mandate, modeled on the reviewer.md Output Constraint `Do NOT paraphrase... Do NOT wrap... Do NOT translate...` style:

```markdown
Do NOT emit any of the following paraphrase drifts -- they all break the executor's grep route:

- `Given X, do Y` (wrong leading verb; must be `Assuming`)
- `Assuming X (unconfirmed), do Y` (wrong marker word; must be `(unverified)`)
- `Assuming X (unverified), Y` (missing `do`)
- `Assuming X is true, do Y` (missing the `(unverified)` marker)
- `If X, then Y` (no Assuming frame at all)
- `Could you clarify X?` or any other question-form clarification request -- no follow-up turn exists.
- A separate `## Assumptions` or `Assumptions:` bulleted section preceding or following the numbered list.
```

### WR-03: Reviewer/security-reviewer conditional-guidance pattern uses `Y.` but advisor uses `do Y.`

**File:** `plugins/lz-advisor/agents/advisor.md:141`, `plugins/lz-advisor/agents/reviewer.md:140`, `plugins/lz-advisor/agents/security-reviewer.md:155`

**Issue:** Three agents now specify the same "conditional guidance inline" pattern but with divergent literal text:

- advisor.md:141: `Assuming X (unverified), do Y. Verify X before acting.`
- reviewer.md:140: `Assuming X (unverified), Y. Verify X before acting.`
- security-reviewer.md:155: `Assuming X (unverified), Y. Verify X before acting.`

The reviewer and security-reviewer forms drop `do`. If any downstream tooling (the review/security-review SKILL.md Phase 3 pass-through, a future parser, or an automated route) greps for a single canonical frame, the asymmetry will cause misses in one of the two modes. Nothing in the SKILL.md files currently parses this frame (Phase 3 is pure verbatim pass-through), so the asymmetry is non-fatal today. It becomes fatal the moment either skill adds any programmatic handling of conditional findings.

The reviewer/security-reviewer form may be deliberate, since their Findings entries are prose rather than imperative numbered steps -- "do" reads awkwardly in prose. That is a reasonable design choice; the fix below preserves it.

**Fix:** Pick one:

1. **Align all three agents on a single literal frame** (preferred for grep simplicity). Reviewer/security-reviewer prose can still read well with the advisor form: "Assuming the request is authenticated (unverified), do treat this finding as post-auth only. Verify the request is authenticated before acting."

2. **Document the asymmetry explicitly.** Add a one-line note in each of the three agents' Edge Cases / Final Response Discipline sections stating that the wording differs intentionally because advisor output is imperative numbered steps while reviewer output is prose. Any future parser or tooling will then see the split is deliberate rather than accidental.

Pick option 2 if the prose/imperative distinction is load-bearing; pick option 1 if future tooling might grep for this pattern. Document the choice in the agent whichever way.

### IN-01: advisor.md Edge Cases "keep intact" list under-specifies punctuation

**File:** `plugins/lz-advisor/agents/advisor.md:141`

**Issue:** The "keep intact" list enumerates words (`Assuming`, `(unverified)`, `do`, `Verify`, `before acting.`) but not the required punctuation positions: the comma immediately after `(unverified)` and the period immediately after `do Y`. Punctuation is what actually distinguishes the canonical grep pattern from drift (see WR-01 for the concrete impact). The current wording implies that the words alone, in any order or punctuation, satisfy the frame.

**Fix:** Extend the "keep intact" clause to explicitly include the punctuation. Example:

```
keep the words `Assuming`, `(unverified),` (including its trailing comma),
`do`, the period that follows the action clause (after `Y`), `Verify`, and
`before acting.` intact -- and in that order.
```

### IN-02: "Examples (illustrative, not exhaustive)" label is ambiguous

**File:** `plugins/lz-advisor/agents/advisor.md:143`

**Issue:** The examples are labeled `Examples (illustrative, not exhaustive):`. This framing suggests domain coverage ("here are a couple of problem domains; others work too") but readers will reasonably treat the examples as the canonical fill-in template for imitation. Because the examples diverge from the frame (WR-01), this ambiguous label makes it unclear which to trust -- the frame's literal text or the examples' structure.

**Fix:** Replace the label with something narrower. Two options:

- `Two canonical fills of the frame above (copy the sentence shape, not the domain):`
- `Two worked substitutions (X and Y are fully replaced; the rest of the frame is intact):`

Then ensure the examples actually demonstrate full substitution per WR-01.

### IN-03: review/security-review SKILL.md Phase 3 "commit range" does not cover all scope sources

**File:** `plugins/lz-advisor/skills/lz-advisor.review/SKILL.md:103`, `plugins/lz-advisor/skills/lz-advisor.security-review/SKILL.md:107`

**Issue:** Phase 3's Required output shape uses `[scope description -- files, directories, or commit range]` as the placeholder for the scope line. But Phase 1 Scope Derivation (steps 1-4) actually accepts four distinct scope sources: user-specified files/directories, uncommitted diff (`git diff HEAD --name-only` plus `git ls-files --others --exclude-standard`), commit-range (`git diff <base>..HEAD --name-only`), and directory-sibling expansion. An executor rendering Phase 3 output for an uncommitted-diff scope has no canonical wording inside the Phase 3 example.

**Fix:** Update the placeholder in both SKILL.md files:

```
[scope description -- files, directories, commit range, or uncommitted diff]
```

### IN-04: Phase 3 `Do NOT` lists do not forbid translating the headers at the skill layer

**File:** `plugins/lz-advisor/skills/lz-advisor.review/SKILL.md:106-111`, `plugins/lz-advisor/skills/lz-advisor.security-review/SKILL.md:110-115`

**Issue:** The reviewer and security-reviewer agents' Output Constraints forbid translation of `### Findings`, `### Cross-Cutting Patterns`, and `### Threat Patterns` into another language. The SKILL.md Phase 3 `Do NOT` lists forbid reformatting into severity groups, stripping/renaming/bolding headers, dropping sections, adding wrappers, and Opus attribution -- but they do not forbid translation at the pass-through layer. The agent-side prohibition is invisible to the executor Sonnet; only the SKILL.md is in the executor's instruction set when it renders Phase 3. In a non-English session, the executor might localize output headers during "rendering verbatim."

**Fix:** Add a bullet to both Phase 3 `Do NOT` lists. Example for the review skill:

```
- Translate the `### Findings` or `### Cross-Cutting Patterns` headers
  into another language; pass them through in English.
```

And for security-review:

```
- Translate the `### Findings` or `### Threat Patterns` headers into
  another language; pass them through in English.
```

This is Info (not Warning) because (a) the agent-side prohibition already exists, and (b) the executor's "verbatim" directive is strong enough to likely suppress translation even without the explicit bullet. The bullet closes the ambiguity.
