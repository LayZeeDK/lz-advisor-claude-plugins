<objective>
Close the **E-runtime** gap opened by Phase 05.4 Plan 05. Plan 05 promoted the `Assuming X (unverified), do Y. Verify X before acting.` frame from a format suggestion to an imperative output-shape contract inside Edge Cases, passed 18/18 static must-haves, but FAILED at runtime: on the DEF-response-structure.sh Finding E fixture (prompt: `"Add auth to this API. The frontend uses cookies."`), the advisor went straight to Fastify/Redis/cookie-session/CSRF recommendations with a "Two risks flagged" closer and no `Assuming X (unverified)` sentence anywhere.

This plan targets four distinct prompt-text defects that explain the static/runtime divergence:

1. **Internal contradiction** (root cause #5, confirmed by reading advisor.md). The `Verification Process` section at lines 96-98 still carries the PRE-Plan-05 soft form `"Assuming X, do Y. If X is wrong, do Z instead."`. Opus sees two frames in one prompt; the soft one sits higher and reads as general guidance, while the strict one sits in Edge Cases and reads as scope-specific. The soft frame must be fully removed.
2. **Contract location** (hypothesis #2). The strict frame lives only in Edge Cases (section 137+). Opus treats Edge Cases as scope-specific. Promoting a short top-of-prompt anchor into `## Output Constraint` makes the frame an unconditional output contract.
3. **Broken examples** (hypothesis #3 / code-reviewer WR-01). The two Edge Cases examples mix literal `X`/`Y` placeholders with substituted clauses (`Assuming X (unverified) the frontend sends cookies over HTTPS only, do Y add the Secure and HttpOnly flags...`). A model imitating these emits text that neither fully substitutes X/Y nor preserves the frame punctuation. Clean substitution is required.
4. **No forbidden-paraphrase list** (hypothesis #4 / code-reviewer WR-02). The contract bans clarification requests in prose but does not enumerate paraphrase patterns the runtime advisor actually emits ("Two risks flagged", "Based on", "Given", etc.). Paraphrase drift has no anchor.

Hypothesis #1 (prompt not thin enough to Opus) is not addressable by editing advisor.md. If Opus truly judges cookies+auth context sufficient, the DEF-response-structure.sh fixture is the wrong regression test. This plan does NOT modify the smoke-test prompt; that would hide the gap rather than close it. The diagnostic task (Task 1) gathers evidence to confirm hypothesis #1 is false before prompt-text surgery begins.

Purpose: strengthen D-16 at the runtime behavioral layer so the advisor emits the literal frame on thin-context prompts. No new decisions introduced; no plugin.json version bump (within-version gap closure, same as Plan 05).

Output: advisor.md updated (single file; four coherent edits); two runtime trace artifacts capturing pre-plan and post-plan behavior.
</objective>
