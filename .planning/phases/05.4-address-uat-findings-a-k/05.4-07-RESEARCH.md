## Executive Summary

**Recommendation: proceed with Plan 07 as drafted, with two small clarifications.**

The proposed approach -- option (a), modify `/lz-advisor.plan` Phase 3 to emit the advisor's Strategic Direction verbatim to stdout before writing the plan file -- is the correct layer. It is structurally symmetric with the already-shipped `/lz-advisor.review` and `/lz-advisor.security-review` pass-through pattern (commits `fc86d64` + `0dcfbab` under Plan 05.4-05), and it is the minimal, well-scoped, reversible change that closes the runtime regression without reshaping the product's orchestration model.

Plan 06's rejection of option (b) stands (hiding the gap is not closing it). Option (c) would require redesigning the DEF Finding E fixture to exercise a different skill, which trades a 5-line SKILL edit for a structural fixture change that silently changes what D-16 tests -- worse, not better. Option (d) is disproportionate.

Two non-blocking clarifications for the executor:

1. **The `---` marker choice is safe but borderline Markdown-ambiguous.** When a downstream tool renders the stdout as Markdown, `--- Strategic Direction ---` renders as plain text (the internal word `Strategic` prevents horizontal-rule parsing), but `--- End Strategic Direction ---` on its own line in isolation is also plain text for the same reason. No collision in practice. However, if a future smoke test uses exact-start-of-line anchoring via `rg '^--- '`, the closing marker matches as expected. Accept as-is.
2. **Ordering (stdout-first, file-write-second) creates a small "orphaned output on file-write failure" window.** If `Write` fails after stdout emission (permission, disk full, path invalid), the user sees advisor guidance on stdout but no plan artifact. This is acceptable for the gap-closure goal (the advisor text is the load-bearing signal; the plan file is an enhancement) but should be acknowledged explicitly in Plan 07 so the executor does not later "fix" it by reversing the order, which would re-open E-runtime.

## Q1: Is option (a) the right layer for this fix?

**Yes. Confidence: HIGH.**

The Plan 06 escalation trace enumerates four remedies; Plan 07 picks (a). Each alternative fails at a more painful layer:

| Option | Layer | Reversibility | Test-invariance | Cost | Verdict |
|--------|-------|---------------|-----------------|------|---------|
| (a) SKILL.md stdout emit | Skill orchestration | HIGH (revert one file) | TEST UNCHANGED | ~60 lines of edit | **Correct** |
| (b) Grep plan file | Test fixture | HIGH | TEST NOW HIDES GAP | ~2 line edit | Rejected by Plan 06 -- hides the failure mode rather than fixing it [CITED: 05.4-06-SUMMARY.md `Remedy options`] |
| (c) Fixture redesign (invoke /lz-advisor.execute) | Test fixture | MEDIUM | D-16 no longer tested against /lz-advisor.plan -- arguably a different contract | ~20 line rewrite + semantic drift | **Architecturally worse** than (a): D-16 locks "agent emits Assuming frame when context is thin" against `/lz-advisor.plan` specifically, because planning is the skill where the thin-context scenario is most likely to arise. Shifting the test to `/lz-advisor.execute` tests a different code path. |
| (d) Advisor architectural redesign | Agent frontmatter / maxTurns / tooling | LOW | Full regression risk on D/F | weeks of rework | Last resort; disproportionate to a stdout-routing defect [CITED: 05.4-CONTEXT.md `<deferred>` block] |

**Why option (c) is tempting but wrong.** A first-principles argument could say: "the skill that is SUPPOSED to emit advisor output inline is `/lz-advisor.execute` (six-phase workflow emits advisor text directly); force the test to use that." But this confuses two different contracts. `/lz-advisor.plan` IS supposed to expose the advisor's strategic direction to the user -- that is its core value proposition ("near-Opus intelligence at Sonnet cost"). Hiding the advisor text behind a plan-file artifact reduces the skill's perceived value and makes automation-facing use cases (like the smoke test, or any `claude -p` pipeline that routes the output to another tool) blind to the advisor's guidance. Option (a) therefore is not a workaround; it is a correctness fix at the right layer.

**Symmetry argument.** `/lz-advisor.review` and `/lz-advisor.security-review` already use the verbatim-pass-through pattern (commits `fc86d64` + `0dcfbab`). Three of four skills emitting advisor output inline + one skill hiding it inside a plan file is asymmetric in a way that has no justification. Plan 07 restores architectural consistency.

## Q2: What is the canonical pattern for skill output in Claude Code?

**There is no single canonical convention for stdout emission.** Confidence: HIGH for "no convention exists"; MEDIUM for "verbatim-pass-through is the emerging community pattern."

Evidence from the installed Anthropic official plugins in `~/.claude/plugins/cache/claude-plugins-official/`:

| Plugin | Skill/command | Output approach |
|--------|---------------|-----------------|
| `plugin-dev` (7 skills) | skill-development, agent-development, hook-development, etc. | Purely instructional content. Skills don't "produce output" structurally; they shape what Claude says in response. No stdout markers or verbatim blocks. |
| `skill-creator` | `skill-creator` | Same as above. Guides Claude through a skill-creation conversation. |
| `claude-md-management` | `claude-md-improver` | Same. |
| `pr-review-toolkit` | `/review-pr` command | Structured output template in markdown (`# PR Review Summary`, `## Critical Issues`, etc.) but NO markers bracketing agent output -- the command aggregates agents and summarises in its own voice. |
| `explanatory-output-style`, `typescript-lsp`, `playground` | various | No skill-specific output conventions. |

[CITED: https://docs.claude.com/en/docs/claude-code/plugins] The official plugins documentation describes skills as "model-invoked" packages that "extend Claude's capabilities" but does NOT prescribe any stdout output shape, marker convention, or verbatim-rendering rule. Skills ARE natural-language prompts; their output shape is whatever the prompt tells the model to produce.

**What this means for Plan 07:**

- There is no Claude-Code-reserved marker token that `--- Strategic Direction ---` could collide with. [VERIFIED: grep of cached official plugins returned no uses of `--- .* ---` marker patterns in SKILL.md or command files.]
- The verbatim-pass-through pattern is a lz-advisor internal convention (shipped in Plan 05.4-05 for review and security-review). Plan 07 extends it to the plan skill; this is natural and correct.
- Markers, XML tags, JSON envelopes, or plain headings are all viable. Plan 07 chose markers. This choice is idiomatic for a lz-advisor audience that sees `--- Strategic Direction ---` as "a separator signalling verbatim advisor content." No conflict with any other plugin.

**One subtlety.** The lz-advisor `/lz-advisor.review` Phase 3 does NOT use `--- ... ---` markers; it uses a one-line scope summary (`> ## Review Summary\n> Reviewed: [scope]`) followed by the reviewer's verbatim Markdown response (which already contains its own `### Findings` / `### Cross-Cutting Patterns` headers). Plan 07's marker style is therefore NOT strictly symmetric with review/security-review; it adds machine-findable boundary markers on top of the pass-through. The asymmetry is justified: reviewer agents emit heading-delimited blocks that a smoke test could grep for (`### Findings`), but the advisor's literal frame `Assuming X (unverified), do Y. Verify X before acting.` is mid-sentence-level; there is no heading to anchor on. Explicit `--- Strategic Direction ---` markers give the DEF-E smoke test (and future users / automation) an unambiguous block start/end.

## Q3: Does the `--- Strategic Direction ---` marker style conflict with anything?

**No meaningful conflict. Confidence: HIGH.**

Checks performed:

| Risk | Evidence | Verdict |
|------|----------|---------|
| YAML frontmatter collision | YAML delimiters are exactly `---` on their own line. `--- Strategic Direction ---` has text between the dashes, so it does NOT terminate frontmatter. [VERIFIED: YAML spec 9.1.2 Directives End marker requires bare `---`] | No collision |
| Markdown horizontal rule | CommonMark HR requires 3+ matching chars optionally followed by whitespace only. `--- Strategic Direction ---` fails that rule (non-whitespace text). Renders as plain text. [CITED: CommonMark spec 4.1 Thematic breaks] | No collision |
| Claude Code runtime-reserved token | [VERIFIED: git grep `-c '^---' plugins/lz-advisor/` returns only frontmatter opens (line 1) and closes (matching `---` lines 20-47 in each file). No `--- X ---` text-bracketed markers exist anywhere in the lz-advisor plugin.] Claude Code plugin docs do not reserve this token. | No collision |
| `rg` escape issues | `rg '^--- Strategic Direction ---$'` works without escape because `---` and `Strategic Direction` have no regex metacharacters. [VERIFIED: plan 07 must-have grep expressions use this pattern and pass normal rg parsing] | No escape required |
| ANSI color codes | `claude -p` emits plain UTF-8 text (verified by inspection of `E-runtime-trace-post-plan06.txt` -- ASCII only, no escape sequences). Markers will render as plain ASCII. | No collision |
| Terminal line wrapping | Marker is 27 chars (`--- Strategic Direction ---`); well below any terminal width threshold. | No concern |

One minor observation: the Plan 07 must-haves specify `rg -c '^--- Strategic Direction ---$' lz-advisor.plan/SKILL.md = 1`. That grep hits the SKILL.md TEXT (where the marker is prose content inside the Phase 3 description), not the runtime-emitted marker. The runtime emission goes to stdout when the skill runs, where the same string appears because Sonnet follows the SKILL.md directive. This is the right test level (SKILL.md contract is the contract; runtime output is verified by the DEF-E smoke test).

## Q4: Are there any known issues with making skills verbose on stdout?

**No blocking issues. Confidence: MEDIUM-HIGH (training-data/doc-derived; no Anthropic bug-tracker evidence surfaced).**

Claims validated:

1. **No stdout size limit on `claude -p --verbose`.** [CITED: https://docs.claude.com/en/docs/claude-code/plugins] The plugin docs describe skills as producing natural-language responses; there is no documented size ceiling. The existing `/lz-advisor.review` skill already emits >500 words to stdout routinely (Findings + Cross-Cutting Patterns) with no known issues. Plan 07 adds ~100-300 words of advisor guidance, well within observed-working envelope.

2. **Pipe/tee routing is unaffected.** [VERIFIED: DEF-response-structure.sh:41-42 uses `--verbose > "$OUT_E" 2>&1` which merges stdout + stderr. `rg -iq` against `$OUT_E` already proven to work for Findings D (Critical marker grep) and Finding F (`### Findings` + `### Cross-Cutting Patterns` greps). Plan 07 adds the same mechanism for Finding E -- not a new code path.]

3. **UTF-8 / ANSI concerns.** [VERIFIED: grep of existing E-runtime-trace-post-plan05.txt and post-plan06.txt shows the trace is ASCII-only; `claude -p` is not injecting color codes into the captured stdout. Plan 07's marker lines are ASCII-only per project CLAUDE.md rule, and the advisor's verbatim text is ASCII-only per advisor.md ASCII-only invariant (Plan 06 `rg -c '[^\x00-\x7F]' = 0`).]

4. **Verbose mode flag behavior.** [CITED: project CLAUDE.md Skill Verification section, and DEF-response-structure.sh usage] `--verbose` surfaces tool-call traces and thinking blocks. The Plan 07 addition doesn't interact with `--verbose` specifically; it adds content to whatever the skill already emits. The smoke test greps case-insensitively for "Assuming .*(unverified)|unverified"; `--verbose` noise is irrelevant to whether that pattern matches somewhere in $OUT_E.

**One open question marked LOW confidence.** [ASSUMED] Whether `claude -p` truncates very long stdout in any automated-pipeline context. This has not been observed in lz-advisor smoke tests to date. Plan 07 emission is small (~100-300 words advisor text + ~20 words skill summary); well below any plausible truncation threshold.

## Q5: Is the additive edit pattern (stdout first, file-write second) the right ordering?

**Yes, but document the failure-mode explicitly. Confidence: HIGH.**

Plan 07 specifies: stdout emit -> plan-file Write -> "Plan written to X" confirmation. Rationale (from Plan 07 `<objective>`): "ensures grep sees `Assuming X (unverified)` before the summary."

**Analysis of the three orderings:**

| Ordering | stdout-first | file-first | interleaved |
|----------|-------------|-----------|-------------|
| Smoke test passes? | YES | YES (content is there regardless of order) | YES |
| User sees advisor guidance even if Write fails? | YES | NO (silent failure: Write errors, user sees nothing useful) | YES |
| User sees "Plan written to X" before advisor content? | NO | YES | MAYBE |
| Matches review skill ordering? | YES (review has no artifact write) | N/A | N/A |
| Risk of "orphaned output" | Low -- stdout already committed is the desired state for the gap-closure goal | HIGH -- Write-succeeds-then-skill-crashes loses advisor text entirely | Medium |

**Recommendation: accept Plan 07's stdout-first ordering.** Reasons:

1. The advisor text IS the load-bearing payload. A user who gets the advisor guidance but no plan file has 90% of the value (they can re-run or write the plan manually). A user who gets a plan file but no advisor output has missed the skill's core deliverable.
2. Symmetry with `/lz-advisor.review`: reviewer pass-through runs first, no artifact write follows, and the existing Plan 05.4-05 contract establishes "render first" as correct.
3. The plan artifact write is the enhancement (durable copy); stdout is the primary channel. Classic "secure defaults": degrade gracefully to the primary channel on enhancement failure.

**The non-blocking clarification.** Plan 07's NEW text does NOT explicitly acknowledge the "Write fails after stdout emits" edge case. Recommend adding one sentence to the `### Render Strategic Direction to Stdout (FIRST)` subsection of Plan 07's NEW text (around line 528-529 in 05.4-07-PLAN.md):

> If the subsequent plan-file Write fails (permission denial, missing directory, disk full), the stdout emission has already reached the user; surface the Write error to the user plainly and do not retry in a way that re-emits the block.

This is a small additive clarification, not a design change. See "Recommended Plan 07 Edits" below.

## Q6: Regression risk on the plan skill's UX.

**Acceptable. Interactive UX becomes more informative; automation parsers keep working if they use resilient patterns. Confidence: MEDIUM-HIGH.**

**Current UX (pre-Plan-07):**
```
Plan written to `plans/add-cookie-auth.plan.md`.

It covers 10 steps: install auth deps, ...
```
~30 words on stdout. Terse. User opens the plan file to see strategic direction.

**New UX (post-Plan-07):**
```
--- Strategic Direction ---

1. [Advisor guidance point 1]
2. [Advisor guidance point 2]
...
(3-7 numbered points, ~100-300 words, including Assuming X (unverified), do Y. Verify X before acting. where applicable)

--- End Strategic Direction ---

Plan written to `plans/add-cookie-auth.plan.md`.

It covers 10 steps: install auth deps, ...
```
~150-350 words on stdout. More informative. User sees advisor guidance inline; plan file remains as durable artifact.

**UX risk register:**

| Risk | Severity | Mitigation |
|------|----------|------------|
| "Chatty" feel in interactive sessions | LOW | The advisor's output IS the strategic direction the user asked for. Surfacing it is a feature, not noise. The review and security-review skills already do this at equivalent or larger length; no UX complaint has surfaced in Phase 5.3 UAT. |
| Breaking automation pipelines that parse terse output | LOW-MEDIUM | Automation pipelines reading `Plan written to <path>` still work because that line is preserved verbatim AFTER the Strategic Direction block. Pipelines greping for "Plan written" will still match. Pipelines that assumed stdout was 2 lines long will need to adapt -- but there is no known user automation at this stage in the plugin's lifecycle, and the change is semver-0.6.0-internal (not a public breaking change). |
| No opt-out / `--silent` flag | LOW | Plan 07 does not add a flag. Keeping the interface uniform (skills don't take flags) is correct -- flags belong in frontmatter `argument-hint` if they are needed, and there's no compelling case for a `--silent` flag here. Users who want terse output can read the plan file after the fact; users who want rich stdout get it. |
| User sees "long output" and perceives a slow / broken skill run | LOW | Advisor block emits as a coherent unit at one moment in time (after Phase 2 consultation completes); does not stream slowly. |
| Color-coding / formatting difference from existing skills | NONE | `--- Strategic Direction ---` markers are ASCII; no coloring; consistent rendering across terminals. |

**Net UX judgment.** The change is an UX *improvement* for the plan skill's primary use case (interactive planning with advisor guidance visible). The cost is a small automation-fragility hit, which is non-existent in practice today.
