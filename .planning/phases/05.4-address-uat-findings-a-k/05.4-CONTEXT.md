### Finding J (review skill plan-acceptance)

- **D-04:** Finding J is fundamentally a scope-derivation regression, not a plan-input feature. The review skill never had an explicit plan-accept feature; the regression emerged because the scan phase's "review files they were working on" clause was biased by conversation-context narrative describing what a plan claimed was or wasn't changed. Fix targets scope derivation, not plan input.
- **D-05:** Adopt Mechanism C (mechanical scope derivation + narrative-isolation rule) in the review skill's scan phase.
  - **Mechanical scope:** scan phase always derives file set via `git diff --name-only` and/or `git log` (plus explicit user-specified files). Scope expands to include all sibling files in directories the diff touches.
  - **Narrative-isolation rule:** plan/narrative in conversation is background about WHY code exists -- never a scope signal about WHAT to investigate. Files a plan says are "unchanged" are still in scope if they live in a directory the diff touches.
- **D-06:** Apply Mechanism C symmetrically to the security-review skill. Security-review's scan phase is structurally identical to review's. Narrative bias would be even more dangerous for security audits (a plan declaring "auth unchanged" could cause a security reviewer to skip the very files with a vulnerability). Touches both `lz-advisor.review/SKILL.md` and `lz-advisor.security-review/SKILL.md`.

### Web tools + verification economics (Findings K, C, B)

- **D-07:** Add `WebSearch, WebFetch` to all four skill frontmatter `allowed-tools` lists (Finding K-1). Agents (advisor, reviewer, security-reviewer) keep `tools: ["Read", "Glob"]`. First-principles aligned with canonical advisor strategy:
  - `research/anthropic/docs/advisor-tool.md:279` -- "The advisor itself runs without tools and without context management... only the advice text reaches the executor."
  - `research/anthropic/blog/the-advisor-strategy-give-agents-an-intelligence-boost.md:49` -- "Sonnet or Haiku runs the task end-to-end as the executor, calling tools, reading results, and iterating toward a solution... The advisor never calls tools."
  Giving agents web tools would widen their role toward "independent investigator," contradict Phase 5.3's Context Trust Contract, consume scarce maxTurns:3 budget on I/O at Opus rate, and expand prompt-injection surface unnecessarily. Resolves Compodoc UAT's 50-Bash excavation (Failure 2) directly. Resolves advisor hallucination (Failure 1) via the K+B+C+E doctrine -- executor pre-verifies external claims during orient; packages verified facts as Pre-Verified Package Behavior Claims; agent synthesizes from the verified package, not from stale training data.
- **D-08:** Add Common Contract rule on fetched-content trust with BEGIN/END FETCHED markers as the starting pattern (Finding K-2). See `<research_directives>` for the planner investigation directive to validate current best practices before locking the exact format.
- **D-09:** Add new first-class `### Pre-Verified Package Behavior Claims` section in both the Proposal and Verification templates in `references/context-packaging.md` (Finding C). Peer to Findings and Source Material, not nested under them. Each entry pairs a claim with the tool-output excerpt that verified it. See `<research_directives>` for the planner investigation directive on exact format.
- **D-10:** Add rule 5 to Common Contract in `references/context-packaging.md` codifying the executor-side pre-verification obligation (Finding B). Approximately: "Verify your claims before consulting. If your prompt asserts anything about third-party package behavior, framework semantics, or external API contracts, verify during orient and include the verification as a Pre-Verified Package Behavior Claim. The agent trusts packaged content as authoritative; do not assert what you have not verified." Lives in Common Contract (single source inherited by all four skills), not in each agent's Context Trust Contract -- executor-side obligation, category-correct on the skill side.

### Tool discipline + portability (Findings H, I, A)

- **D-11:** Adopt the candidates-doc per-skill `allowed-tools` table with K-1 web tools folded in (Finding H-1). Three-profile ladder preserved, signalling responsibility in the manifest:
  - `lz-advisor.plan`: `Agent(lz-advisor:advisor), Read, Glob, Bash(git:*), Write, WebSearch, WebFetch`
  - `lz-advisor.execute`: `Agent(lz-advisor:advisor), Read, Glob, Edit, Write, Bash(git:*), WebSearch, WebFetch`
  - `lz-advisor.review`: `Agent(lz-advisor:reviewer), Read, Glob, Bash(git:*), WebSearch, WebFetch`
  - `lz-advisor.security-review`: `Agent(lz-advisor:security-reviewer), Read, Glob, Bash(git:*), WebSearch, WebFetch`
  - Review + security-review converge (Profile A: read-only); plan adds Write (Profile B: artifact authoring); execute adds Edit (Profile C: implementation).
  - `allowed-tools` is additive pre-approval; user deny rules always win. Missing tools still invokable but fall through to user settings, producing permission prompts -- which is the intended behavior for infrequent primitives (e.g., plan editing an existing plan file is a one-prompt safety signal, not a blocker).
- **D-12:** Do not declare test-runner tools in `lz-advisor.execute` allowed-tools (Finding H-2). Test-runner tools are project-specific (npm/pnpm/yarn/cargo/pytest/jest/vitest/playwright); declaring any subset punishes non-matching projects. Common Contract coaches executor to use project-appropriate primitives; plugin does not pre-commit to a test framework. Users layer their own allowlist.
- **D-13:** Add rule 6 to Common Contract in `references/context-packaging.md` codifying graceful degradation on tool-use failure (Finding I). Four sub-rules:
  - (a) **First-denial primitive swap.** If the failure was a permission denial AND a cheaper or more direct primitive could retrieve the same information (e.g., Read replacing a language-runtime parse of a structured file), retry once using the alternative primitive. The user may be redirecting toward the right primitive.
  - (b) **Sustained unavailability.** If the alternative also fails or no alternative applies, treat the information as unavailable. Note in the consultation's Findings section (one line); proceed.
  - (c) **Denial as scope signal.** A denial on one file in a directory or package is a signal about the scope, not just that file. If a call against one file in a scope is denied, do not continue attempting calls against sibling files in the same scope. User is flagging the scope; respect it.
  - (d) **Never halt on failure.** Do not wait for user intervention unless the missing information blocks every possible path forward. In non-interactive automation (`claude -p`), halting is a permanent hang.
  Each SKILL.md body adds a one-line pointer at the top of its orient/scan phase referencing the Common Contract rule.
- **D-14:** Reword Phase 5.3 D-13 orientation-budget instruction in `lz-advisor.plan/SKILL.md` and `lz-advisor.execute/SKILL.md` only (Finding A). Keep in those two SKILL.md bodies; do not move to Common Contract (review/security-review are git-scope-bounded post-Finding J and rarely legitimately enter node_modules). New wording approximately:
  > "Stop exploring when you have enough context to formulate a specific question for the advisor. Inside `node_modules`, read with discipline: targeted reads only (specific function, config entry, or a few lines around a symbol), never full-file for bundled or minified content. When dependency behavior is load-bearing, verify and surface the result as a Pre-Verified Package Behavior Claim (see context-packaging.md)."
  Replaces the current blanket ban on `node_modules/*/dist/` reads that was actively blocking Finding C's economics premise.

### Response-structure polish (Findings D, E, F)

- **D-15:** Adopt Position B for advisor's handling of out-of-scope observations (Finding D). Advisor surfaces out-of-scope ONLY when the observation is correctness-affecting, security-critical, or data-loss-risking (high bar: "silence would cause regret"). Non-critical "helpful but unasked" observations stay silent. When surfaced, items use `**Critical:**` or `**Must address:**` marker, budget-excluded from the 100-word cap, and treated as primary signal by the execute skill (same weight as direct answers, not a bonus-triage section).
  - Rationale: aligns with first-principles reading of advisor strategy (blog line 49 -- advisor is consulted at decision points; executor owns scope). Avoids transferring scope-triage cost to the user. Simpler downstream than Position C's marker + segregation + user-triage UX. Collapses part of Finding D into Finding E's conditional-framing machinery.
  - **UAT-H3 reference case under Position B:** the observed ci check would have stayed silent (helpful but non-critical). Advisor response would be 4 numbered points answering 3 questions, clean answer-to-question ratio, within the 100-word cap. Clean.
  - **Scope:** advisor only. Reviewer and security-reviewer produce structured severity-classified findings; their Response Structure already segregates findings by severity. Finding D is not relevant to them.
- **D-16:** 'Assuming X (unverified)' conditional framing lives inline within numbered answers; no separate Response Structure section (Finding E). Format: "Point N: [advice], assuming [unverified premise]; verify before acting." Sentence-level convention, not a section. Keeps the response tight; keeps conditional items tied to their direct question. Applies to all three agents (advisor, reviewer, security-reviewer) where they give advice that depends on unpackaged context.
- **D-17:** Resolve reviewer word-cap tension by giving Cross-Cutting Pattern its own named slot with its own budget (Finding F). Response Structure becomes two named slots:
  - `### Findings` -- budget 250w; numbered findings with file:line, each ~60w max.
  - `### Cross-Cutting Patterns` -- budget 100-150w; synthesis across findings (shared root causes, recurring structural issues). Distinct content, not overflow.
  Total ~400w per response; each slot independently budgeted. Apply symmetrically to security-reviewer -- the synthesis slot is named `### Threat Patterns` there (security-focused variant of the same structural role).

## Specific Ideas

- **Anthropic-source citations are load-bearing for Finding K.** The first-principles argument against agent web tools rests on literal readings of `advisor-tool.md:279` ("advisor runs without tools") and blog line 49 ("executor holds tools"). Any future pushback on D-07 should cite these same passages and articulate a first-principles counter-argument before reopening the decision.
- **Position B (Finding D) relies on advisor self-discipline.** The rule "surface Critical only if silence would cause regret" is subjective; Opus 4.7's literal-following helps but doesn't guarantee perfect calibration. Plan 4 should include a smoke test specifically for the "helpful-but-unasked" case -- a scenario similar to UAT-H3's ci check -- to confirm the advisor stays silent rather than inventing an "innocuous Critical" marker.
- **Mechanism C's git-derived scope (Finding J) changes the review skill's relationship with uncommitted changes.** If a user invokes `/lz-advisor.review` on code they haven't committed yet, `git diff --name-only` (for changes vs main) will include uncommitted work; `git log` alone won't. The rewritten scan phase should use `git diff HEAD --name-only` or equivalent to include uncommitted-but-staged changes. Planner should confirm the exact git invocation during implementation.
- **Plan artifact update workflow is a first-class use case, not an edge case.** Users iterate on plans during refinement conversations. The plan SKILL.md body under D-11 should include explicit guidance: "If the user supplied an existing plan file (via @ mention or path reference), update it using Edit rather than rewriting wholesale. Preserve unchanged sections verbatim. Each edit corresponds to a specific user request; do not refactor the plan beyond what was asked. The plan skill never modifies source code under any circumstance -- use /lz-advisor.execute for implementation." The first permission prompt the user sees for `Edit` on the plan file is a safety signal, not a friction bug.
- **Critical marker is budget-excluded but response-weighted.** The advisor response under Position B has two implicit categories: direct answers (100w cap, numbered, primary signal) and `**Critical:**` (budget-excluded, sparse-usage, ALSO primary signal from execute's POV). This differs from candidates-doc Position C where out-of-scope is bonus content; under Position B, when the advisor DOES use `**Critical:**`, it carries the same actionability weight as the numbered answers.
- **Finding F's two-slot restructuring naturally surfaces the reviewer's highest-value content.** Cross-Cutting Patterns (the 80w section in UAT-H3) is described in Phase 5.3 UAT-RESULTS as the highest-signal part of the reviewer's response. Giving it a named slot with its own budget acknowledges this and removes the "must fit under 300w total" pressure that was suppressing it.
