## Decision Log for Multi-Option Findings

### WR-01: picked Option 1 (alignment) over Option 2 (documentation)

The review offered two paths: align all three agents on the same literal frame, or document the divergence in each agent file. I picked **Option 1 (alignment)** -- promoted `do Y` into reviewer.md and security-reviewer.md so all three agents now use `Assuming X (unverified), do Y. Verify X before acting.` verbatim.

Reasoning:
- Plan 05.4-07 introduced the `--- Strategic Direction ---` marker contract in lz-advisor.plan Phase 3, which anchors on the advisor's literal output. The plan skill's downstream greps and future parsers key off the `do Y` shape.
- If any future tool extends that contract to review-style skills, divergent frames would silently break the match.
- The prose-vs-imperative distinction is real but cosmetic -- reviewer prose reads fine with `do` ("Assuming the request is authenticated (unverified), do treat this finding as post-auth only. Verify the request is authenticated before acting.").
- One canonical contract across three files is worth more than documenting a permanent "if" in each agent.
- I also added a one-sentence rationale to each agent's Final Response Discipline section so future maintainers understand why `do` is load-bearing even in prose.

### WR-02: retained the bullet even though WR-01 was fixed via alignment

The review noted that if WR-01 is fixed via alignment (Option 1), the WR-02 bullet becomes unnecessary because reviewer/security-reviewer no longer canonicalize a competing `do`-less form. I still added the bullet because:
- Written English flows more smoothly without `do` ("Assuming HTTPS (unverified), add the Secure flag" reads better than "...do add the Secure flag"). That natural-drift risk is independent of what reviewer/security-reviewer specify.
- The cost of adding one bullet is negligible; the benefit is belt-and-suspenders defense against the single most likely paraphrase the advisor Opus model would produce.
- The bullet's parenthetical notes the shared frame with reviewer/security-reviewer so the rationale stays coherent after WR-01's alignment.

**Linkage to WR-01**: After WR-01's alignment, the WR-02 bullet is defensive rather than load-bearing. If a future maintainer re-diverges reviewer/security-reviewer (e.g., to restore a prose-only form), the bullet would again become the canonical defense.

### IN-03: picked Option 2 (SKILL note) over Option 1 (new template)

The review offered two options: add a new advisor-specific Phase 5 template to context-packaging.md, or add a SKILL note that the Verification template's Findings-list structure is optional for Phase 5 of execute. I picked **Option 2 (SKILL note)**.

Reasoning:
- Option 2 is load-light: one paragraph in execute/SKILL.md Phase 5 that tells the executor to adapt the Verification template for the advisor's output shape.
- The advisor's 100-word Output Constraint already forbids Findings-list output (numbered steps beginning with `1.`, no severity tags). The template mismatch is a documentation issue, not a functional one.
- Adding a new template to context-packaging.md duplicates scaffolding for a single call site. The SKILL note reuses existing scaffolding and clarifies adaptation inline.
- The note explicitly directs the executor to package a summary block + single targeted Request, which matches what the advisor needs for final-review.

### IN-05: bumped plugin.json and lz-advisor.plan/SKILL.md to 0.7.0 (minor bump)

The review recommended bumping plugin.json and lz-advisor.plan/SKILL.md to 0.7.0 because Plan 05.4-07 introduced a materially new user-facing contract (the `--- Strategic Direction ---` stdout markers) without a version bump. Per the context block's recommendation, this is a minor bump under 0.x semver (new user-facing interface = minor). The other three skills (execute, review, security-review) stayed at 0.6.0 since their contracts did not change in Plan 07.
