## Trust Boundaries

Consolidated from all 7 plan threat models. Duplicates deduplicated; plan-specific variants merged where they describe the same boundary.

| Boundary | Description | Data Crossing |
|----------|-------------|---------------|
| conversation narrative / attached plan file -> review & security-review scan phases | Attacker-influenced plan narrative or quoted plan file could bias the reviewer to skip a vulnerable file. Finding J regression. | Scope decisions (filenames, directories) |
| user-provided `@plan.md` reference -> review scope | Plan file could contain adversarial text ("skip file X for security reasons") that a naive scope derivation would honor. | Plan file body |
| network fetch -> executor skill context | WebFetch brings untrusted remote content into the Claude Code session during orientation. | Fetched documentation, release notes, API references |
| fetched content -> agent prompt | Executor packages fetched content into the agent's system prompt, crossing into the advisor/reviewer trust domain. | `<fetched source=... trust="untrusted">...</fetched>` XML-tagged blocks |
| pre-verified claim text -> agent synthesis | Claims are marked authoritative; malicious/mistaken claims poison the agent's reasoning. | `<pre_verified source=... claim_id="pv-N">` XML blocks |
| user-governance (settings.json) -> plugin manifest `allowed-tools` | `allowed-tools` is additive pre-approval, not a restriction. User deny rules override. Confusion about enforcement is the user-facing risk. (GitHub anthropic/claude-code#18837) | Tool permission decisions |
| denied tool call -> workflow continuation | If executor halts on every denial, plugin becomes unusable in locked-down environments. | Tool-use outcomes, error states |
| denied tool call in scope A -> subsequent call in sibling scope B | Rule 6 sub-rule (c): treat denial as a scope signal; executor must not keep probing. | Filesystem probe patterns |
| advisor output budget -> runtime response | `**Critical:**` marker is budget-excluded; inflation risk (advisor abuses exclusion to bypass 100w cap). | Advisor numbered list + Critical block |
| conditional recommendation inline (Assuming X (unverified)) | Unpackaged-context assumptions surface in advisor/reviewer response; executor must verify before acting. | Assumption premises, verification triggers |
| reviewer / security-reviewer response shape -> downstream consumers | Response shape change visible to consumers; downstream parsers expecting old 300w combined cap see new two-slot structure. | `### Findings` / `### Cross-Cutting Patterns` / `### Threat Patterns` headers |
| advisor.md prompt text -> Opus runtime | Prompt text is the only input to Opus 4.7; any contract defect translates directly to a behavioral defect. Drift between strict frame and examples is a tampering vector. | Advisor system prompt |
| lz-advisor.plan/SKILL.md Phase 3 -> claude -p stdout | Phase 3 verbatim pass-through is where advisor Strategic Direction crosses from skill-internal orchestration to user terminal (and automated grep). | Strategic Direction block between `--- Strategic Direction ---` markers |
| DEF-response-structure.sh fixture -> regression-test scope | Fixture is load-bearing regression input; modifying to make smoke-test pass would hide the gap rather than close it. | Test prompt body |
| smoke-script `$(git rev-parse --show-toplevel)` expansion | Captured (REPO_ROOT) BEFORE `cd "$SCRATCH"` to prevent path-confusion in subshells inside scratch repos. | Filesystem root resolution |
| diagnostic trace files -> future gap-closure rounds | Trace files are evidence; they must capture raw stdout verbatim so post-hoc analysis has a source of truth. | claude -p stdout + stderr, grep counts |
