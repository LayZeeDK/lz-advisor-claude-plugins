## Warnings

### WR-01: Direct contradiction between context-packaging.md and advisor-timing.md on whether to inline file contents

**File:** `plugins/lz-advisor/references/context-packaging.md:16-23` vs. `plugins/lz-advisor/references/advisor-timing.md:57-59`

**Issue:** All four SKILL.md files `@`-load both `advisor-timing.md` and `context-packaging.md` in the same header block (plan: lines 26-28, execute: 27-29, review: 27-29, security-review: 28-30). The two references give directly opposing instructions to the executor about how to pack context for the advisor:

- `context-packaging.md` rule 1 (lines 16-23): "Inline file contents for any file the advisor might need ... include concrete file contents (key functions, type definitions, config blocks) in your prompt ... Rule of thumb: if you referenced a file by path and the advisor would reasonably want to see it, inline the relevant lines."
- `advisor-timing.md` lines 57-59: "Keep the prompt focused and summarized. The advisor works best with clear, organized context -- not raw file dumps or verbose tool output. Summarize findings; do not paste entire files."

The executor, reading both files into context, sees contradictory directives. The new Context Trust Contracts in all three agents reinforce the inline-contents side ("When a file's relevant contents are quoted in your prompt, you do not need to Read that file"), which makes the `advisor-timing.md` "do not paste entire files" wording stale relative to the new model. This is the same failure mode (advisor re-reading files already in the prompt) that Phase 5.3 F1 explicitly targets, only now re-introduced from the executor side: the executor may underpack because one of the two loaded references tells it to.

**Fix:** Update `advisor-timing.md` to align with the new packaging contract. Two options:
1. Replace lines 57-59 in `advisor-timing.md` with a pointer: "For the full packaging contract and templates, see `references/context-packaging.md`. Summarize your own orientation findings; preserve source material and relevant file contents verbatim per rule 1 of the Common Contract."
2. Rewrite the paragraph to say "Summarize your own orientation findings; do not paraphrase source material or file contents the advisor would otherwise Read."

Either keeps the two files consistent and preserves the intent behind the F1 fix.

Note: this file was not listed in the Phase 5.3 change set, so flagging as a follow-up rather than blocking. The phase brief calls out "absence of contradictory instructions across the Context Trust Contract sections" as a review target -- this contradiction sits one level up, between the Contract and the other loaded reference.

### WR-02: `Context Trust Contract` in reviewer agent conflicts with its own `Edge Cases` guidance on re-reading files

**File:** `plugins/lz-advisor/agents/reviewer.md:82-104` vs. `plugins/lz-advisor/agents/reviewer.md:134-136`

**Issue:** The new Context Trust Contract tells the reviewer: "When a finding includes a code snippet with file:line reference, use that snippet as your primary evidence. Tool use is for reaching nearby code the snippet did not include, not for verifying the snippet itself." (lines 88-90).

The pre-existing Edge Cases section says the opposite: "When a finding lacks sufficient code context to evaluate, use Read to check the actual file before dismissing it. A finding that looks minor in isolation may be significant when you see the surrounding code." (lines 134-136).

These are not strictly contradictory -- one governs when a snippet IS present, the other when it is NOT -- but the reviewer, faced with an underpacked finding, now has two competing impulses: trust the snippet vs. Read the file. Given that F2 (agents ignoring anti-re-read instructions) is a field-observed failure, the softer "check the actual file" wording likely wins over the Contract's "Tool use is for reaching nearby code". The same pattern exists in `security-reviewer.md` but is less acute because that agent's Edge Cases section does not contain a similar re-read directive.

**Fix:** Tighten the Edge Cases wording in `reviewer.md` to route through the Contract's batching rule:

```diff
-When a finding lacks sufficient code context to evaluate, use Read to
-check the actual file before dismissing it. A finding that looks minor
-in isolation may be significant when you see the surrounding code.
+When a finding lacks sufficient code context to evaluate, batch a single
+verification turn per the Context Trust Contract -- issue Read (and any
+adjacent Glob) in parallel, then synthesize. Do not dismiss a finding
+without this check; a finding that looks minor in isolation may be
+significant in context.
```

This preserves the escape hatch (don't silently dismiss) while making the Contract's batching discipline the governing rule.
