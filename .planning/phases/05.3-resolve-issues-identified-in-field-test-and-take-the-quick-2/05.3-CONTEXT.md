## Implementation Decisions

### Fix sequencing & measurement

- **D-01:** Parallel sequencing in one phase: baseline UAT on Opus 4.7 with current prompts -> prompt fixes committed -> post-fix UAT on same scenario. Produces A/B evidence in a single phase.
- **D-02:** Reuse the Phase 5.2 field-test scenario: Compodoc+Storybook setup on `D:\projects\github\LayZeeDK\ngx-smart-components`. Git-reset or worktree-teardown between runs. Preserves direct A/B comparison against the 57% baseline (4/7 first-try successes).
- **D-03:** Embed Nx Compodoc integration docs in the skill prompt so the scenario is self-contained. Fetch source: `https://markdown.new/https://nx.dev/docs/technologies/test-tools/storybook/guides/angular-storybook-compodoc?method=auto&retain_images=true`. Researcher captures the resolved Markdown once; planner wires it into the prompt template used across all UAT runs.
- **D-04:** Acceptance gate for "fix succeeded": dual threshold. (a) First-try success rate >= 80% across the 7 equivalent consultations (6/7 or 7/7), AND (b) advisor averages <= 4 tool calls per consultation. Both gates must pass for the phase to close as verified.
- **D-05:** If the post-fix UAT misses either gate, Phase 5.3 closes with the failure recorded in `05.3-VERIFICATION.md` and **Phase 5.4 is inserted** for a from-scratch advisor redesign research track. No in-phase prompt-iteration loop. Out-of-scope redesign candidates include: remove `maxTurns` cap; `SendMessage`-based bidirectional communication (Agent asks for missing context in its output, Skill resumes with the missing context); advisor with 0 tools; multi-model team (Sonnet main session via `model: inherit`, Opus 4.7 advisor, Haiku 4.5 codebase exploration); online research for other patterns.

### Agent prompt hardening (field findings F1, F2)

- **D-06:** Apply batching + context-trust instructions to all three Opus agents (advisor, reviewer, security-reviewer). Symmetric application even though the field test only stressed advisor and reviewer; F2 (soft instructions ignored) is a model-behavior pattern, not scenario-specific. Security-reviewer's lone successful consultation used only 2 tool calls, which doesn't prove immunity.
- **D-07:** Add a new `## Context Trust Contract` section to each of the three agent system prompts. Positive framing ("treat quoted files as authoritative" rather than "DO NOT re-read"), one concrete one-shot example, an explicit batching directive, and a turn-budget note. Matches Opus 4.7's "literal instruction following" and the user's CLAUDE.md preference for positive examples over negative prohibitions.
- **D-08:** Calibrate the Context Trust Contract section per agent role, not identical copy-paste. Advisor emphasizes compression (tight with `effort: high` and 100-word cap). Reviewer and security-reviewer emphasize efficient verification (their `effort: xhigh` legitimately sanctions deeper tool use for correctness-critical analysis per Anthropic's 4.7 guidance).
- **D-09:** Edit existing `## Verification Process` / `## Review Process` sections to trim overlap with the new Context Trust Contract. Contract becomes the authoritative place for tool-use rules; existing sections retain guidance unrelated to tool-use (assumptions, edge-case handling). Prevents literal-following conflicts under 4.7.

### Executor context packaging (field finding F3)

- **D-10:** Write a new shared reference file `plugins/lz-advisor/references/context-packaging.md`. Structure: `## Common contract` (4 shared rules: inline file contents for any file the advisor might need, targeted specific request ending the prompt, enumerated/tagged structure, turn-budget-aware packing) + `## Proposal consultation template` (used by plan skill and by execute skill's pre-consult + mid-consult call sites) + `## Verification consultation template` (used by review skill, security-review skill, and by execute skill's final-review call site).
- **D-11:** Each SKILL.md (all four) replaces its existing inline packaging prose with `@${CLAUDE_PLUGIN_ROOT}/references/context-packaging.md` plus a one-line pointer to which sub-template applies at each consult site. Mirrors the existing `advisor-timing.md` pattern exactly. Single source of truth.
- **D-12:** Include one worked example per sub-template, adapted from the field test's successful security-review consultation (Verification template) and a synthesized example for the Proposal template. ~15 lines each. Matches the `260417-lhe-ASSESSMENT.md` recommendation to use positive examples for Opus 4.7 behavior tuning.

### Executor orientation discipline (field finding F5)

- **D-13:** Include F5 fix as an adjacent edit inside Area C's SKILL.md changes for `plan` and `execute`. Add an orientation-budget instruction to the executor's orient phase: "Stop exploring when you have enough context to formulate a specific question for the advisor. Do not read bundled, minified, or files under `node_modules/*/dist/`." Marginal diff cost because the packaging `@`-load edits already touch those files.

### UAT execution methodology

- **D-14:** Automate UAT items 1, 2, 3, 4, 6 via `claude -p --verbose` + spawned subagents. Automation coverage: UAT 1 (tool-call count) via `--verbose` log parsing; UAT 2 (100-word advisor cap) via output word-counting; UAT 3 (300-word reviewer/security-reviewer cap) same; UAT 4 (tone drift on reviewer severity) via LLM grader subagent scoring A/B pre-fix vs post-fix transcripts against a defined rubric; UAT 6 (`maxTurns: 3` at `xhigh`) via transcript inspection for preamble-only responses and exhaustion events.
- **D-15:** UAT 5 (`model: opus` auto-routes to Opus 4.7 on Pro and Free plan tiers) recorded as `human_needed` in `05.3-VERIFICATION.md`. Structurally impossible to verify from a Team subscription; document as a cross-tier follow-up test for whoever has access to Pro or Free accounts.
- **D-16:** UAT orchestration shape: per-skill parallel subagents per pass. Spawn 4 `general-purpose` subagents per pass (baseline + post-fix = 8 subagents total), each with `Agent(isolation: "worktree")` for plugin-repo isolation. Each subagent creates a scratch Compodoc environment via `git worktree add` on `D:\projects\github\LayZeeDK\ngx-smart-components`. Inside, runs `claude -p --plugin-dir <plugin-worktree>/plugins/lz-advisor --model sonnet -p "/lz-advisor.<skill> <Compodoc task prompt>" --verbose`. Each subagent returns its captured transcript + parsed metrics as part of its final message payload (not left in the worktree). After both passes complete, one analysis/grader subagent consumes all 8 returns and writes `05.3-UAT-RESULTS.md`.

### Claude's Discretion

- Exact format of the Proposal vs Verification sub-template bodies in `context-packaging.md` (heading hierarchy, bullet structure) -- constraint is the 4 shared rules; planner decides layout
- Number of plans in the phase breakdown -- expected range is 3-5 plans, but planner decides based on coupling (e.g., whether agent prompt changes and SKILL.md changes are one commit or separated for independent revert-ability)
- Commit granularity within each plan -- prefer one-commit-per-logical-change per the project convention, but fine-grained atomicity is planner's call
- Version bump strategy for `plugin.json` -- 0.3.0 was the Phase 5.2 version; 5.3 could be 0.3.1 (patch, prompt-level only) or 0.4.0 (minor, new shared reference file). Planner decides based on whether the new `references/context-packaging.md` counts as a user-visible behavior change.
- Exact UAT rubric for tone-drift grader (UAT 4) -- define as a short scoring schema inside the orchestration plan; candidate dimensions: direct-vs-validation-forward, severity-classification-consistency, remediation-specificity

<specifics>
## Specific Ideas

- The field test documented the Compodoc+Storybook scenario as "the canonical stress test" for cross-skill advisor behavior. Reusing the same scenario for 5.3 UAT preserves direct comparison to the 57% baseline and doesn't require designing a new stress fixture.
- The `Agent(isolation: "worktree")` pattern combined with `git worktree add` inside `ngx-smart-components` gives full bidirectional state isolation: plugin-repo state is frozen at spawn time (pre-fix or post-fix depending on pass), and Compodoc scratch state is disposable per run. This is cleaner than a `git reset --hard HEAD` loop and avoids destructive operations the user is cautious about.
- The LLM grader subagent for UAT 4 (tone drift) works because tone classification is A/B comparative, not absolute. Given two transcripts scored against the same rubric, a grader can reliably say "post-fix is more direct" or "post-fix is more validation-forward" even if the absolute severity classifications aren't perfectly objective.
- The planner can assume all three `@`-loadable reference files (`advisor-timing.md`, new `context-packaging.md`, and whatever future references arrive) will follow the same ~4K-token size ceiling that `advisor-timing.md` already observes. Keeps the skill body under compaction limits.
- The Compodoc task prompt for UAT should embed the Nx doc content once at prompt construction time; the subagent doesn't need to re-fetch it mid-run. Fetch uses the `markdown.new` path from the URL in D-03; fall back to the CLAUDE.md fetch chain if `markdown.new` rate-limits.
- Phase 5.3 is the first phase that exercises `Agent(isolation: "worktree")` in this plugin's workflow at scale. Planner should note any friction (e.g., worktree creation latency) in `05.3-VERIFICATION.md` so future phases can plan around it.

</specifics>
