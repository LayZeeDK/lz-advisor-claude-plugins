```markdown
[Compodoc](https://compodoc.app/) is a documentation tool for Angular applications. When used with Storybook, Compodoc can automatically generate documentation for your Angular components by analyzing their TypeScript code, decorators, and JSDoc comments.

## What is compodoc?

Compodoc automatically generates documentation by:

- Analyzing TypeScript interfaces and classes
- Reading Angular decorators (`@Input`, `@Output`, etc.)
- Parsing JSDoc comments
- Generating `argTypes` for Storybook controls

## Setting up Compodoc with Storybook

### Prerequisites

You need to have Storybook already configured for your Angular project. If you haven't set up Storybook yet, follow the [Angular Storybook setup guide](/docs/technologies/test-tools/storybook/guides/overview-angular).

### Install compodoc

Install Compodoc as a dev dependency:

    npm install --save-dev @compodoc/compodoc

### Configure Compodoc in your Storybook

Add Compodoc configuration to your Storybook's `main.ts` file:

    // .storybook/main.ts
    import type { StorybookConfig } from '@storybook/angular';

    const config: StorybookConfig = {
      stories: ['../src/**/*.stories.@(js|jsx|ts|tsx|mdx)'],
      addons: ['@storybook/addon-essentials', '@storybook/addon-interactions'],
      framework: {
        name: '@storybook/angular',
        options: {},
      },
      typescript: {
        check: false,
        checkOptions: {},
        reactDocgen: 'react-docgen-typescript',
        reactDocgenTypescriptOptions: {
          shouldExtractLiteralValuesFromEnum: true,
          propFilter: (prop) =>
            prop.parent ? !/node_modules/.test(prop.parent.fileName) : true,
          compilerOptions: {
            allowSyntheticDefaultImports: false,
            esModuleInterop: false,
          },
        },
      },
      docs: {
        autodocs: true,
      },
    };

    export default config;

### Generate compodoc documentation

Add a script to your `package.json`:

    {
      "scripts": {
        "docs:json": "compodoc -p tsconfig.json -e json -d .",
        "storybook": "npm run docs:json && nx storybook your-project"
      }
    }

### Configure your Angular component for better documentation

    // button.component.ts
    import { Component, Input, Output, EventEmitter } from '@angular/core';

    export interface ButtonProps {
      /** The button label */
      label: string;
      /** Primary or secondary styling */
      primary?: boolean;
      /** Disabled state */
      disabled?: boolean;
      /** Button size */
      size?: 'small' | 'medium' | 'large';
    }

    /**
     * A customizable button component
     */
    @Component({
      selector: 'app-button',
      template: `
        <button
          [class]="classes"
          [disabled]="disabled"
          (click)="onClick.emit($event)"
        >
          {{ label }}
        </button>
      `,
      styleUrls: ['./button.component.css'],
    })
    export class ButtonComponent implements ButtonProps {
      @Input() label: string = '';
      @Input() primary: boolean = false;
      @Input() disabled: boolean = false;
      @Input() size: 'small' | 'medium' | 'large' = 'medium';
      @Output() onClick = new EventEmitter<Event>();

      get classes(): string {
        return [
          'btn',
          this.primary ? 'btn-primary' : 'btn-secondary',
          `btn-${this.size}`,
        ].join(' ');
      }
    }
```

### Common Contract (sketch, ~25 lines final)

The four rules:

1. **Inline file contents for any file the advisor might need.** The advisor has Read and Glob tools, but every tool call costs one of its 3 turns. Include the concrete file contents (key functions, type definitions, config blocks) in your prompt. The advisor can verify; it should not need to orient from scratch. Rule of thumb: if you referenced a file by path and the advisor would reasonably want to see it, inline the relevant lines.

2. **End the prompt with a targeted, specific request.** Not "what do you think?" but "given findings A-D, which of approach X or approach Y is correct?" or "validate these 4 findings against OWASP Top 10." A targeted request gets a targeted response; an open request gets exploration.

3. **Use enumerated, tagged structure.** Findings as a numbered list. Code snippets in fenced blocks with file paths above them. Verbatim user-pasted source material clearly delimited (BEGIN/END markers or blockquotes). Your own orientation findings summarized, but source material preserved verbatim per the Phase 5.1 verbatim-vs-summary rule.

4. **Pack with turn-budget awareness.** The advisor has 3 turns total. If your prompt is well-packaged (rule 1), the advisor usually responds in 0-1 tool calls. If your prompt is under-packaged, the advisor will spend 2-3 turns reading files and may exhaust its budget before synthesizing. Pack the prompt to minimize the advisor's need to verify -- that is the highest-leverage optimization.

(Bridges from Phase 5.1's existing "What to Include Verbatim" text in plan/SKILL.md lines 60-74 and execute/SKILL.md lines 71-85, which migrate here.)

### Proposal Consultation Template (sketch, ~40 lines final)

Used when the executor is asking for strategic direction before or during work. Call sites: `lz-advisor.plan` Phase 2, `lz-advisor.execute` Phase 2 (pre-execute) and Phase 3 (mid-execute re-consultation).

**Structure:**

```
## Task
[Quote the user's request verbatim, in a blockquote. One paragraph.]

## Orientation Findings
[Numbered list. What files were read, what patterns were found, what constraints
apply. Summarized in your own words. Each bullet one sentence.]

## Source Material (if any)
[User-pasted docs, error messages, or third-party content. Preserved verbatim.
Use BEGIN/END delimiters when the block exceeds ~10 lines.]

## Relevant File Contents
[Path-labelled fenced code blocks for files the advisor might need.
Include only the sections that matter for the decision.]

## Question
[One specific question. Not "what do you think?" Not a list of questions.
A single targeted ask, phrased so that a numbered-list answer is possible.]
```

**Worked example (~15 lines):**

```
## Task
> Set up Compodoc in Storybook for the ngx-smart-components library.

## Orientation Findings

   neither is in package.json as a direct dependency.

3. project.json has a `storybook` Nx target but no `docs:json` pre-step.

## Source Material
[Nx Compodoc guide verbatim; see BEGIN NX GUIDE ... END NX GUIDE block.]

## Relevant File Contents
[.storybook/main.ts, project.json, package.json -- in order, paths labelled.]

## Question
Given Storybook 10.3.5 removed the top-level `docs.autodocs` API shown in
the Nx guide, what is the correct sequence of steps to enable Compodoc
```

## Per-Agent Context Trust Contract Sketches

The three agents share the same core (4 rules + batching + turn budget + concrete one-shot) but diverge per role per D-08.

### Advisor Context Trust Contract (sketch, ~20 lines final)

Role-specific angle: compression. The advisor is at `effort: high` with a 100-word output cap. Every tool call burns a turn that could have been used to think about synthesis.

Draft text:

```markdown
## Context Trust Contract

The executor packages context into your prompt -- file contents, orientation
findings, source material -- because its job is to orient and yours is to
synthesize. Trust what the executor packaged:

- When a file's relevant contents are quoted in your prompt, you do not need
  to Read that file. The executor already did.
- When source material (documentation, error messages, pasted docs) is in
  your prompt verbatim, treat it as authoritative for the consultation.
- When orientation findings are summarized, they stand as evidence -- do not
  re-verify each finding by reading files.

Use Read and Glob only for facts NOT in your prompt. If you must verify
multiple files, do it in a single turn with parallel tool calls -- e.g.,
two Reads in one turn, not two Reads across two turns. Your budget is 3
turns total.

One-shot: if the prompt includes `.storybook/main.ts` contents and asks
whether the `docs.autodocs: true` line is correct for Storybook 10, answer
directly from the quoted contents. Do not Read `.storybook/main.ts` -- it
is already in your prompt.
```

**Why this phrasing survives 4.7 literal-following:** Positive phrasing ("trust what the executor packaged") not "DO NOT re-read." Concrete example grounds the rule. Batching directive ("parallel tool calls in a single turn") is explicit, not implied. Turn budget is stated (3 turns).

### Reviewer Context Trust Contract (sketch, ~20 lines final)

Role-specific angle: efficient verification. The reviewer is at `effort: xhigh` with a 300-word output cap. Verification IS the job -- the rule is not "don't verify" but "batch verifications efficiently."

Draft text:

```markdown
## Context Trust Contract

The executor packages 3-5 curated findings with file:line references, code
snippets, and initial severity assignments. Your job is to validate these
findings, not to re-discover them. Trust the executor's curation:

- When a finding includes a code snippet with file:line reference, use that
  snippet as your primary evidence. Tool use is for reaching nearby code
  the snippet did not include, not for verifying the snippet itself.
- When the executor provides CLAUDE.md excerpts or project guidelines,
  treat them as authoritative.
- When initial severity assignments are provided, confirm or revise them --
  do not re-derive them from scratch.

Your `effort: xhigh` budget permits verification tool use when it is
cross-cutting (does finding N manifest in adjacent code the executor did
not scan?). Batch such verifications in a single turn: issue multiple Read
or Glob calls in parallel, not one-per-turn. Your budget is 3 turns total.

One-shot: if the executor packages 4 findings and you want to check
whether a 5th related file exhibits the same bug, issue Glob + Read
in parallel in one turn, then synthesize in your final response.
```

**Why this differs from advisor version:** Legitimately permits more tool use (xhigh earns it), but insists on parallel batching. This matches the 05.2-FIELD-TEST.md insight that the ONLY successful reviewer call used 2 batched tools, and the ONLY failed reviewer calls used 5-8 sequential tools.

### Security-Reviewer Context Trust Contract (sketch, ~22 lines final)

Role-specific angle: efficient threat verification. Same as reviewer plus OWASP/threat-model-aware.

Draft text:

```markdown
## Context Trust Contract

The executor packages curated security findings with OWASP category tags,
code snippets, and a threat model context block. Your job is threat
analysis and attack-chain synthesis -- not rediscovery. Trust the
executor's curation:

- When a finding includes a CVE identifier, version number, or line
  reference, use Read or Glob only to confirm that specific fact, in a
  single targeted tool call. Do not re-scan the surrounding code unless
  the finding's validity depends on it.
- When the executor provides a threat model context block (assets, actors,
  blast radius), treat it as authoritative. Do not re-derive the threat
  model from the code.
- When an OWASP category is suggested, you may revise the mapping but not
  the finding itself.

Your `effort: xhigh` budget permits deeper verification, but batch it.
Issue multiple Read or Glob calls in a single turn when you need to
cross-reference (e.g., Read package.json + Read package-lock.json in one
turn to verify a dependency finding). Your budget is 3 turns total.

One-shot: if the executor packages a finding about @compodoc/compodoc
being a transitive rather than direct dependency, Read package.json AND
package-lock.json in a single parallel turn -- that's the winning
pattern for quick verification.
```

**Why the third one-shot is Compodoc-specific:** It matches the 05.2-FIELD-TEST.md security-review success case (2 tool calls: Read package-lock.json + Read project.json). Grounding the one-shot in the scenario the UAT will exercise means the advisor has a perfectly calibrated example to generalize from.

## UAT Automation Harness Design

Concrete design for D-14, D-16. Covers: subagent prompt template, log-parsing approach, grader rubric, orchestration shape.

### Orchestration shape

```
Phase 5.3 planner creates a top-level orchestration plan that:

1. Baseline pass (pre-fix):
   - git checkout a commit BEFORE the prompt-edit plans land (or a baseline branch)
   - Spawn 4 general-purpose subagents in parallel, one per skill
   - Each: Agent(isolation: "worktree") -> plugin repo worktree auto-created
   - Each runs the Compodoc scenario against its skill
   - Each returns captured stream-json + parsed metrics in final message

2. Post-fix pass:
   - git checkout a commit AFTER the prompt-edit plans land (or main)
   - Spawn 4 general-purpose subagents in parallel, same shape

3. Grader subagent:
   - Receives 8 transcripts + metrics
   - Writes 05.3-UAT-RESULTS.md with A/B comparison + gate verdict

Total subagents: 9 (4 baseline + 4 post-fix + 1 grader).
Total wall-clock time estimate: 10-15 min per pass (parallel) + 5 min grader.
```

### Log-parsing approach (per D-14)

The `claude -p --verbose --output-format stream-json` command emits NDJSON to stdout with one event per line. Each event is a JSON object with `type` field such as `message_start`, `content_block_start` (with sub-type `tool_use` / `text` / `tool_result`), and `message_stop`. The Agent-tool invocation of lz-advisor:advisor produces nested events tagged with a `parent_tool_use_id` that matches the outer Agent call.

**UAT 1 (tool-call count)** -- jq filter:

```
jq -s '
  [.[] | select(.type == "content_block_start"
                and .content_block.type == "tool_use"
                and .parent_tool_use_id != null
                and (.agent_name // "" | test("lz-advisor:"))) ]
  | length
' uat-trace.ndjson
```

**UAT 2/3 (word-count cap)** -- concat all text blocks from the advisor agent's final message, split on whitespace, count. Heuristic: if > 110 (advisor) or > 330 (reviewer/security-reviewer), flag as cap exceeded (10% tolerance).

**UAT 4 (tone drift)** -- see Grader rubric below.

**UAT 6 (maxTurns: 3 at xhigh)** -- count `message_start` events with `parent_tool_use_id` matching the agent spawn. If count == 3 AND final block is tool_use not text, mark as exhausted. Expected frequency should drop between baseline and post-fix passes if fixes work.

```
## UAT 4 tone-drift rubric

For each of the 4 reviewer/security-reviewer transcript pairs
(review baseline, review post-fix, security-review baseline, security-review
post-fix), score the advisor's final response on three 1-5 Likert dimensions:

- **Directness (1 = validation-forward / hedged / 5 = direct and
  committing):** Does the response open with substantive content or with
  phrases like "It's important to note that..." / "There are a few things
  to consider..." / "Overall, the code looks..."?
- **Severity-classification consistency (1 = inconsistent with
  executor's initial ratings / 5 = consistently confirms or revises
  with reasoning):** Does the reviewer explain when they revise severity?
  Is severity applied uniformly across similar findings?
- **Remediation specificity (1 = generic ("validate inputs") / 5 =
  actionable and concrete ("use parameterized queries on line 42")):**
  Does the reviewer point to specific lines / functions / fixes?

Compare baseline vs post-fix scores per dimension per skill. Flag:
- Any dimension dropping by > 1 point: regression.
- Any dimension increasing by > 1 point: improvement.
- Stable: no change.

Note: 4.7 release notes warn that "more direct tone" may blunt tact.
A directness increase is expected; a severity-consistency or remediation-
specificity DROP is a concern to raise explicitly.
```

**Why this grader is robust:** Pair-relative scoring (not absolute) sidesteps the "what does 3/5 really mean?" problem. The rubric calls out the known risk (4.7 directness over-applying) so the grader doesn't just celebrate every directness increase.

## Risks and Unknowns

Items the planner must mitigate or explicitly accept.

### Risk 1: `Agent(isolation: "worktree")` silent failure (HIGH)

Multiple open GitHub issues document cases where `isolation: "worktree"` silently places the subagent in the main repo instead of a worktree:

- [anthropics/claude-code#33045](https://github.com/anthropics/claude-code/issues/33045) -- Agent isolation fails silently for team agents
- [anthropics/claude-code#37549](https://github.com/anthropics/claude-code/issues/37549) -- Same issue combined with team_name parameter
- [anthropics/claude-code#39886](https://github.com/anthropics/claude-code/issues/39886) -- Generic silent failure

**Impact if triggered:** All 4 parallel subagents would operate in the same main plugin-repo directory, race on branch checkouts, and cross-contaminate. Baseline pass could accidentally run against the post-fix code if the checkout race happens.

**Mitigation (planner must implement):**
- **Explicit worktree check at subagent start.** Each UAT subagent must run `git rev-parse --git-dir` and `pwd` as its FIRST two commands, and assert the paths indicate a worktree (not the main plugin repo). If the check fails, the subagent should return a failure metric and NOT proceed.
- **Don't rely on team_name.** The failure in #37549 is specific to the `team_name` parameter. Our UAT uses plain `general-purpose` subagents without teams -- should not hit that specific bug, but the plain-Agent bug in #39886 is still open.
- **Claude Code version pin in the UAT plan.** Document which Claude Code version the UAT ran against. The fix for #39886 may land between baseline and post-fix runs, which would invalidate A/B comparability. Target: both passes on the same Claude Code version.
- **Fallback if isolation fails reliably:** Serial execution with explicit `git worktree add` per subagent, managed by the orchestrator (not the Agent tool). Slower but deterministic.

### Risk 2: Opus 4.7 availability on outer `claude -p` (LOW, but verify early)

The UAT subagents' OUTER invocation is `claude -p --model sonnet`. This is fine -- Sonnet is the executor. But the INNER advisor/reviewer/security-reviewer agents use `model: opus` which auto-routes to 4.7 on Team Plan per the 260417-lhe assessment. If the user runs the UAT on a non-Team Plan account, `model: opus` might resolve to 4.6, invalidating the "Opus 4.7 UAT follow-ups" premise of the phase.

**Mitigation:**
- **Version capture in every UAT metric.** Each subagent must capture the `model_id` field from the stream-json output for each advisor consultation. Grader asserts all 8 runs used `claude-opus-4-7[1m]` (or `claude-opus-4-7` -- the `[1m]` suffix indicates 1M context and may vary).
- **If mixed versions detected,** the grader flags the pass as invalid and the phase cannot close per D-04 gate.

### Risk 3: Worktree cleanup on Windows Dev Drive (ReFS) (MEDIUM)

The user is on Windows 11 arm64 with a Windows Dev Drive formatted as ReFS (per user global CLAUDE.md). Known quirks:
- ReFS supports git worktrees but filesystem case sensitivity differs from NTFS.
- `git worktree remove --force` on Windows sometimes fails if Claude Code still holds file handles.
- The `feedback_worktree_artifact_loss.md` memory entry documents that uncommitted SUMMARY/ASSESSMENT files were lost on worktree remove in a prior quick task.

**Mitigation:**
- **Do all captures inside the subagent's final message,** not as files left in the worktree. The subagent prompt template above enforces this ("return metrics JSON + summary text as your final message").
- **Verify worktree cleanup.** After each pass, orchestrator runs `git worktree list` and asserts no UAT worktrees remain. If they do, `git worktree prune` + manual `rmdir /s` (escalated).
- **Don't delete the plan-repo worktree while the grader is reading metrics.** Since metrics are returned in the subagent's final message (not left on disk), the grader doesn't depend on worktrees. Plugin worktree cleanup can happen any time after subagent returns.

### Risk 4: Subagent no-change behavior with `Agent(isolation: "worktree")` (LOW)

CONTEXT.md Reusable Assets section asserts "teardown is automatic if the subagent makes no changes." This is the documented behavior per the 2026-02 Claude Code release notes. But: what if the subagent invokes `claude -p` which in turn modifies files inside the worktree (e.g., plan files from `/lz-advisor.plan`)? That's a "change" from the inner claude-p's perspective but not from the outer Agent's perspective -- it's unclear which wins the cleanup rule.

**Mitigation:**
- **Explicit cleanup in subagent prompt.** The subagent prompt template above includes step 5: "Clean up your Compodoc worktree: git worktree remove --force ..." Covers the ngx-smart-components worktree which is under the subagent's explicit control.
- **Plugin-repo worktree:** Accept that it may or may not auto-clean; planner adds a `git worktree prune` step to the orchestrator's post-pass cleanup. Documentation artifacts (if any) are captured in the subagent's return message BEFORE cleanup.

### Risk 5: UAT prompt embedding size (MEDIUM)

The Nx guide is ~5.8 KB. Embedded verbatim in the UAT skill prompt, it pushes the prompt length up. Combined with the user's original task and the skill's own prompt scaffolding, the total may approach claude-p's command-line argument limit on Windows (~8 KB).

**Mitigation:**
- **If command-line length is a problem,** write the prompt body to a temp file and use `claude -p @<file>` or stdin-redirect. Windows cmd line limit is ~8 KB for a single argument, so inline may work but is risky.
- **Test during planning.** Planner should verify one bare `claude -p "<full prompt>"` invocation works before committing the UAT harness design.

### Risk 6: Stream-json tool-use attribution to correct agent (MEDIUM)

The parsing design assumes `content_block_start.type == "tool_use"` events can be attributed to a specific agent (advisor vs reviewer vs security-reviewer) via `parent_tool_use_id` or `agent_name` fields. This is my understanding from the WebSearch-retrieved stream-json examples, but those examples showed single-agent traces. Multi-agent traces (executor + agent) may require different parsing logic.

**Mitigation:**
- **Validate parsing on a single real run FIRST** before parallel-running all 8. The planner should sequence the UAT harness implementation: Plan 3a: build and validate the parser on one known-good run. Plan 3b: scale to 8 parallel runs.
- **Fallback:** If stream-json attribution is ambiguous, count the agent's inner tool calls by matching against agent spawn boundaries (Agent tool calls are distinct tool_use events; everything nested inside a specific spawn is the agent's).

### Unknown 1: Does Claude Code plugin-dir resolution follow worktree-path or always main-repo path?

If the subagent is in a plugin-repo worktree at commit X, and runs `claude -p --plugin-dir <worktree>/plugins/lz-advisor`, does Claude Code resolve skill/agent files from the worktree path or from the main repo's plugins directory? The research did not find a definitive answer.

**Action:** Planner's Plan 3a validation run should verify plugin-dir resolution by making an intentional change in a worktree's agent file and confirming `claude -p --plugin-dir <worktree>/...` picks up the change. If it resolves to main-repo path, the whole isolation premise collapses.

### Unknown 2: Is there a race between subagent worktree creation and plugin-dir loading?

`Agent(isolation: "worktree")` creates the worktree at spawn time. But the inner `claude -p --plugin-dir <worktree>/...` depends on the worktree existing BEFORE that process starts. If there's a race (e.g., worktree creation is async), the inner claude-p could fail with a missing-path error.

**Action:** Subagent prompt step 1 explicitly creates the Compodoc worktree with a synchronous `git worktree add`. Plugin-repo worktree race is implicit in `Agent(isolation: "worktree")`. Planner should note this in the UAT plan: if inner claude-p fails with "plugin-dir not found," the fallback is Agent()-without-isolation + manual worktree management.
