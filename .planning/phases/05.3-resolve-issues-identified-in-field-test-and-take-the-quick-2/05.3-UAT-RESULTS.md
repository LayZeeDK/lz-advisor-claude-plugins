## Note on ngx-smart-components State During UAT

All 8 UATs ran against ngx-smart-components HEAD = `8696de7` (at the time of worktree creation). That commit, together with `c3d9d0a`, carried a partial Compodoc+Storybook integration from an earlier pre-phase test pass. Both commits were removed from main AFTER the UATs completed (user reset main back to `019a26a`; reflog expired; `git gc --prune=now` purged the unreachable objects). Implications:

- **A/B comparison stands.** Baseline and post-fix passes both saw the identical ngx state (`8696de7`); the deltas between passes are attributable to the plugin change (Plans 01+02), not to ngx variation.
- **D-04 gate verdict stands.** D-04 measures advisor behavior (first-try success, tool-call efficiency, tone), not the correctness of specific findings against a specific ngx state.
- **Specific finding text references a now-dead state.** Reviewer/security-reviewer outputs cite artifacts from the removed commits (e.g., "self-referential `browserTarget`", ".gitignore line 54 documentation.json entry"). Those findings should not be transplanted into the current ngx main; they were correct analyses of the state the advisors were asked to analyze.
- **Scenario validity.** A half-completed Compodoc integration is actually a strong stress test for the advisor (triage + complete vs. plan-from-nothing), so the scenario content was arguably MORE informative than a greenfield scenario.

Re-running the same 8 UATs today would produce different *finding content* but likely very similar *advisor behavior* (word count, tool calls, tone, first-try success). If a future phase wants to revalidate, re-run against current ngx HEAD -- the harness and parser are unchanged.
