### Skill Naming Convention

- **D-01:** Skill `name` fields use `lz-advisor.<skill>` pattern. New names: `lz-advisor.plan`, `lz-advisor.execute`, `lz-advisor.review`, `lz-advisor.security-review`. Qualified names become `lz-advisor:lz-advisor.plan`, etc. User-facing shorthand `/lz-advisor.plan` is the priority; internal qualified-name redundancy is acceptable.
- **D-02:** Skill directories rename to match name fields: `skills/lz-advisor.plan/`, `skills/lz-advisor.execute/`, `skills/lz-advisor.review/`, `skills/lz-advisor.security-review/`.
- **D-03:** All cross-references updated end-to-end in this phase: skill descriptions, CLAUDE.md directory structure docs, eval query files, README. No deferred cleanup.
- **D-04:** Version bump to 0.3.0 for plugin manifest (`plugin.json`) and all four skill `version` fields. Skill rename is a breaking change warranting a minor bump.
- **D-05:** Eval directories rename to match: `evals/lz-advisor/lz-advisor.plan-workspace/`, etc.
- **D-06:** Agent names stay as-is (`advisor`, `reviewer`, `security-reviewer`). Agents are skill-orchestrated, not user-invoked. No clash risk.
- **D-07:** Accept dot-namespace separator risk. If Claude Code treats the dot as a plugin/skill separator, `/lz-advisor.plan` might not resolve correctly. The qualified form `lz-advisor:lz-advisor.plan` always works as fallback. Bet on name-field resolution.
- **D-08:** Trigger phrases in descriptions use dotted form only (`lz-advisor.plan`, not both `lz-advisor plan` and `lz-advisor.plan`).
- **D-09:** Use `git mv` for directory renames to preserve file history.
- **D-10:** Sibling skills referenced by shorthand in description cross-references: `lz-advisor.execute`, `lz-advisor.review`, etc. (not the verbose qualified form).

### Preamble Fix Strategy

- **D-11:** Primary fix: change advisor agent (`advisor.md`) effort from `high` to `medium`. Opus 4.6 docs explicitly state that lower effort levels cause the model to "proceed directly to action without preamble." This addresses the root cause that Phase 5.1's prompt-level Final Response Discipline could not resolve.
- **D-12:** Differentiated effort levels per agent role. Advisor agent: `effort: medium` (conciseness priority). Reviewer and security-reviewer agents: `effort: high` (analysis depth priority).
- **D-13:** Keep Final Response Discipline section on all three agent system prompts. Defense in depth: effort parameter addresses behavior, FRD addresses wording. Reviewers at `effort: high` especially benefit from prompt-level guidance.
- **D-14:** Keep `tools: ["Read", "Glob"]` on all three agents. effort: medium is the fix for preamble, not tool removal. Advisor retains verification capability.
- **D-15:** Success criterion (stricter than Phase 5.1): zero wasted tokens. First token of advisor output must be substantive content. Any narration preamble ("Let me verify...", "I'll check...") is a failure.
- **D-16:** Two-stage verification: (1) automated `claude -p` test per skill with `--plugin-dir plugins/lz-advisor` to check for zero preamble tokens, (2) manual re-test with the compodoc Angular/Storybook setup scenario across all 4 skills (plan, execute, review, security-review) -- the same scenario that originally exposed the preamble problem.
- **D-17:** Accept reviewer preamble at `effort: high`. Reviewers need depth; preamble is the price of thorough analysis. FRD mitigates it somewhat.
- **D-18:** Keep `effort` parameter only, no explicit thinking configuration. Claude Code's Agent tool manages adaptive thinking automatically. `effort: medium`/`high` is the control surface.

### Deferred Preamble Decisions

- **D-19:** Fallback strategy if effort: medium doesn't fully eliminate preamble -- deferred. Test first, then decide escalation path (effort: low, tool removal, or accept partial improvement).
- **D-20:** maxTurns revisit -- deferred. User is open to revisiting if justified, but effort: medium should be tested first. Security mitigations (T-01-03, T-04-03/04/08/09) remain intact for now.

<specifics>
## Specific Ideas

- The Opus 4.6 effort docs explicitly state: "Lower effort levels tend to: Combine multiple operations into fewer tool calls, Make fewer tool calls, Proceed directly to action without preamble, Use terse confirmation messages after completion." This is the smoking gun for why `effort: high` caused preamble despite the Phase 5.1 prompt fix.
- "Think", "Think hard", "Think harder", and "Ultrathink" are valid control keywords on Opus/Sonnet 4.6 that trigger progressively deeper extended thinking. These are NOT problematic on 4.6 (unlike 4.5 where "think" was sensitive when extended thinking was disabled). The research doc should reflect this correction.
- Opus 4.6 is naturally less verbose: "More direct and grounded" and "may skip verbal summaries for efficiency unless prompted otherwise." This means `effort: medium` combined with the model's natural terseness should strongly suppress preamble.
- The compodoc Angular/Storybook scenario is the canonical test case because it was used in Phase 5 testing and produced preamble in all 5 sub-agent sessions. Re-testing with the same scenario provides a controlled comparison.
- Opus 4.6 context window increased from 200K to 1M. This doesn't directly affect the plugin (advisor receives packaged context, not full transcripts) but is important for the research doc update.

</specifics>
