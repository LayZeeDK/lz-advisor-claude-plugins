## Summary

Reviewed the full lz-advisor plugin: 4 skill definitions, 3 agent definitions (advisor, reviewer, security-reviewer), 4 eval JSON files, plugin manifest, README, advisor-timing reference, CLAUDE.md project instructions, and the Opus model optimization research document.

The plugin is well-structured with consistent versioning (all 0.4.0), clear skill descriptions with proper trigger phrase coverage, and balanced eval sets (10 positive + 10 negative each). Skills correctly reference their respective agents via `allowed-tools`.

One critical issue: the reviewer and security-reviewer agents have `maxTurns: 1` while their system prompts instruct them to use Read/Glob tools for verification -- a combination that produces silent failure when the agent attempts tool use. Two warnings relate to documentation inconsistencies in CLAUDE.md. Two informational items cover minor documentation quality issues.

## Critical Issues

### CR-01: reviewer.md and security-reviewer.md Have maxTurns: 1 With Tool-Use Instructions

**File:** `plugins/lz-advisor/agents/reviewer.md:45` and `plugins/lz-advisor/agents/security-reviewer.md:45`
**Issue:** Both agents set `maxTurns: 1` in their frontmatter while also declaring `tools: ["Read", "Glob"]` and containing system prompt instructions to use those tools for verification. CLAUDE.md line 79 documents this exact failure mode: "maxTurns: 3 -- 2 tool-use turns + free synthesis turn; prevents preamble-as-response from maxTurns: 1." The advisor agent was fixed by bumping to `maxTurns: 3`, but reviewer and security-reviewer still have `maxTurns: 1`.

With `maxTurns: 1`, if the agent attempts a Read or Glob call to verify a finding, it consumes its only turn on the tool call and produces no text response to the executor. The executor receives either silence or a truncated response. This makes the verification paths described in the prompts (reviewer.md lines 87-89: "use Read or Glob to verify against actual project files"; security-reviewer.md lines 91-95: same) impossible to follow.

The failure is silent and intermittent -- it only manifests when the agent decides it needs to verify a finding, making it difficult to reproduce consistently.

**Fix:** Bump `maxTurns` to `3` for both agents, matching the advisor agent pattern:

```yaml
# In reviewer.md and security-reviewer.md frontmatter:
maxTurns: 3
```

This gives the agent 2 tool-use turns for verification plus 1 synthesis turn for the text response. Update the "Final Response Discipline" sections accordingly to reflect the available turns. Alternatively, if tool verification is not desired for these agents, remove `tools: ["Read", "Glob"]` from their frontmatter and remove all tool-use instructions from their system prompts -- but this reduces review quality.

## Warnings

### WR-01: CLAUDE.md Claims "Single Agent" But Codebase Has Three

**File:** `CLAUDE.md:66`
**Issue:** Line 66 states "Single agent, multiple skills: The advisor persona is constant; skills define different workflows that invoke it." The Alternatives Considered table (line 161) reinforces this: "Multiple agents | Single `lz-advisor` agent | Per-skill agents." However, the codebase contains three agents: `advisor.md`, `reviewer.md`, and `security-reviewer.md`. The directory tree immediately above (lines 47-49) correctly shows all three files, creating an internal contradiction within the same document section.

This is misleading for contributors and new sessions that read the prose conventions without cross-referencing the directory tree. Since CLAUDE.md is loaded as primary context, stale claims here actively influence how agents reason about the architecture.

**Fix:** Update line 66 to reflect the three-agent architecture:

```markdown
- **Three agents, four skills**: A general-purpose advisor for plan and execute workflows, plus specialized reviewer and security-reviewer agents for review and security-review skills.
```

Update the Alternatives Considered row (line 161) to reflect the actual design decision: role-specialized agents rather than a single shared agent.

### WR-02: CLAUDE.md Agent Format Table Documents Only advisor.md

**File:** `CLAUDE.md:70-79`
**Issue:** The "Agent File Format" section documents only the advisor agent's frontmatter fields in its table. The reviewer and security-reviewer agents have different `color` values (`cyan` and `yellow` vs `magenta`), different `maxTurns` values (currently `1` vs `3`), and different output word budgets (300 words vs 100 words). A contributor using CLAUDE.md as the authoritative agent spec would not know about these differences.

Line 78 adds "reviewer and security-reviewer also use `high`" for the `effort` field, and line 79 documents `maxTurns: 3` -- but these apply to the advisor, while reviewer and security-reviewer actually use `maxTurns: 1` (pending the fix in CR-01).

**Fix:** Expand the Agent File Format section to document all three agents:

```markdown
### Agent Configuration
| Field | advisor.md | reviewer.md | security-reviewer.md |
|-------|-----------|-------------|---------------------|
| `model` | `opus` | `opus` | `opus` |
| `color` | `magenta` | `cyan` | `yellow` |
| `effort` | `high` | `high` | `high` |
| `tools` | `["Read", "Glob"]` | `["Read", "Glob"]` | `["Read", "Glob"]` |
| `maxTurns` | `3` | `3` | `3` |
| Word budget | 100 words | 300 words | 300 words |
```

(Update `maxTurns` values after CR-01 is resolved.)
