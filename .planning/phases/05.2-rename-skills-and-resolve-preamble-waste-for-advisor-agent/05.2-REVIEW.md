## Warnings

### WR-01: CLAUDE.md Claims "Single Agent" But Codebase Has Three

**File:** `CLAUDE.md:66`
**Issue:** The directory structure comment on line 66 states "Single agent, multiple skills: The advisor persona is constant; skills define different workflows that invoke it." The Alternatives Considered table on line 159 reinforces this: "Multiple agents | Single `lz-advisor` agent | Per-skill agents (plan-advisor, review-advisor) | The advisor persona is constant; skills define different workflows. One agent, many skills." However, the codebase actually contains three agents: `advisor.md`, `reviewer.md`, and `security-reviewer.md`. The directory tree immediately above (lines 47-49) correctly shows all three files, creating an internal contradiction within the same document section. This is misleading for contributors who read the prose without cross-referencing the tree.
**Fix:** Update lines 66 and 159 to reflect the actual three-agent architecture. The current design uses a shared advisor agent for plan/execute and specialized reviewer agents for review/security-review. Example correction for line 66:

```markdown
- **Three agents, four skills**: A general-purpose advisor for plan and execute workflows, plus specialized reviewer and security-reviewer agents for code quality and security review skills respectively.
```

Update the Alternatives Considered row (line 159) similarly to reflect the chosen approach of role-specialized agents rather than a single agent.

### WR-02: CLAUDE.md Agent Format Table Only Documents `advisor.md`

**File:** `CLAUDE.md:70-77`
**Issue:** The "Agent File Format" section (line 69) subsection "advisor.md -- The Opus Advisor Agent" documents only the advisor agent's frontmatter fields. The reviewer and security-reviewer agents have different `effort` levels (`high` vs `medium`), different `color` values (`cyan`/`yellow` vs `magenta`), and different word budget constraints (300 words vs 100 words). A contributor referring to CLAUDE.md as the authoritative spec for agent format would incorrectly assume all agents use `effort: medium` and `color: magenta`. Since CLAUDE.md serves as the primary context for new sessions, this gap could cause regressions in future agent edits.
**Fix:** Either expand the Agent File Format section to document all three agents, or generalize the table to describe shared fields and note per-agent differences:

```markdown
### Agent File Format
| Field | advisor.md | reviewer.md | security-reviewer.md |
|-------|-----------|-------------|---------------------|
| `model` | `opus` | `opus` | `opus` |
| `color` | `magenta` | `cyan` | `yellow` |
| `effort` | `medium` | `high` | `high` |
| `tools` | `["Read", "Glob"]` | `["Read", "Glob"]` | `["Read", "Glob"]` |
| `maxTurns` | `1` | `1` | `1` |
```

### IN-02: MODEL-OPTIMIZATION-OPUS.md Has Nested Code Blocks

**File:** `research/prompt-engineering/MODEL-OPTIMIZATION-OPUS.md:103-129`
**Issue:** Multiple sections embed triple-backtick code blocks inside outer triple-backtick markdown blocks (e.g., lines 103-129 where a ` ```json ` block at line 119 is nested inside a ` ```markdown ` block opened at line 103). Standard markdown renderers treat the inner closing ` ``` ` (line 124) as closing the outer block, causing the remainder to render as plain text. This pattern repeats in sections at lines 183-204, 220-259, 286-317, and others.
**Fix:** Use four-backtick fences (` ```` `) for outer blocks, or replace inner code blocks with indented code (4-space indent), or restructure to avoid nesting.

### IN-03: Execute Eval Negative Case Tests "execute" Keyword Discrimination

**File:** `evals/lz-advisor/lz-advisor-execute-eval.json:59`
**Issue:** The negative test case "execute this SQL query: SELECT * FROM users WHERE active = true" is a well-designed discrimination test since "execute" appears in the skill's trigger phrases. However, it contains a literal SQL query with a table name (`users`) and a `WHERE` clause that, if accidentally passed to a database-connected tool in a testing environment, could return real data. This is purely a test prompt string (not executed), so the risk is informational only.
**Fix:** No code change needed. If desired for extra caution, the query could use a clearly fictional table: "execute this SQL query: SELECT * FROM example_widgets WHERE active = true".
