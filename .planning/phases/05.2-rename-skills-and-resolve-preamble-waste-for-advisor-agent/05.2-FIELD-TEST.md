# Phase 05.2 Field Test Report

Real-world exercise of all 4 lz-advisor skills against a non-trivial Angular tooling task. Complements the formal UAT (05.2-UAT.md) with observations about emergent behavior patterns that shaped the next-phase fix plan.

## Test Context

**Project:** ngx-smart-components -- Angular Nx monorepo, library package with Storybook 10.3.5 and @compodoc/compodoc 1.2.1 already installed as transitive deps.

**Task:** Enable Compodoc integration in Storybook to auto-generate component documentation. Required changes spanned project.json (Nx targets), .storybook/main.ts (Storybook config), stories.ts (per-component metadata), .gitignore (generated artifact), and package.json (explicit devDependency).

**Skills exercised in sequence:** plan -> execute -> review -> security-review.

## Per-Skill Observations

### 1. lz-advisor.plan

**Outcome:** Produced a correct 5-step plan. Advisor caught version-specific Storybook API considerations and provided concrete file-level guidance.

**Advisor consultation:** 1 invocation, succeeded first try. Used 3 Read calls (project.json, tsconfig.json, .storybook/main.ts) then synthesized findings into numbered steps.

**Issue observed:** Executor's Phase 1 (Orient) over-investigated -- made ~20 Bash calls diving into `node_modules/@storybook/angular` internals (chunk files, builder source) when it already had enough context. User interrupted with "Why are you inspecting every chunk of node_modules/@storybook/angular?" and the executor acknowledged it was over-investigating.

**Root cause:** Executor lacks an orientation stopping heuristic. No explicit guidance to avoid reading bundled/minified source or to stop when enough context exists to formulate the advisor question.

### 2. lz-advisor.execute

**Outcome:** End-to-end success. Correctly followed plan, caught a Storybook 8 -> 10 API mismatch mid-execution (`docs.autodocs` removed), documented the deviation, committed atomically, and acted on the final advisor's recommendations (explicit devDependency declaration + build verification).

**Advisor consultations:** 3 invocations, 1 succeeded first try.

| Consultation | Result | Tool calls | Pattern |
|---|---|---|---|
| Pre-execute | Failed (no recovery) | 4 sequential (Read, Glob, Read, Glob) | maxTurns exhausted |
| Mid-execute | Succeeded | 4 batched (Glob+Glob parallel, Read+Read parallel) | Fit in 2 tool turns + 1 synthesis |
| Final review | Failed, recovered | 6 sequential (Read x4, Glob, Read) | maxTurns exhausted; background resume + re-prompt took 5m53s |

**Key observation:** The only successful execute-skill consultation was the mid-execute one, which happened to batch calls in parallel. Both pre-execute and final-review consultations made sequential single-tool-per-turn calls and exhausted the budget.

**Recovery mechanism:** The executor's recovery path (detect preamble-only response -> resume agent in background -> re-prompt with follow-up question) works but adds ~6 minutes per incident. The recovered output quality is high -- the final review caught a transitive dependency risk (@compodoc/compodoc not explicit in package.json) and the autodocs-per-story API nuance.

### 3. lz-advisor.review

**Outcome:** Clean structured output -- 0 Critical, 1 Important, 2 Suggestions. Caught a pre-existing Nx boilerplate issue (self-referencing `browserTarget` in project.json) that neither the plan nor execute skill flagged. Connected findings to a shared root cause (Nx scaffold accepted without review).

**Reviewer consultations:** 2 invocations, 1 succeeded first try.

| Consultation | Result | Tool calls |
|---|---|---|
| First invocation | Failed (preamble only) | 8 tools used, including a wrong-path error |
| Re-invocation with inlined file contents | Succeeded | 5 tools used despite explicit "do not read files" instruction |

**Key observation:** The executor adapted after the first failure -- it re-invoked the reviewer with full file contents inlined and the instruction "I am including full file contents so you do not need to read files." The reviewer **ignored** this instruction and still made 5 file reads. This reveals that soft language ("you do not need to") is insufficient; stronger prompt constraints are needed.

### 4. lz-advisor.security-review

**Outcome:** Cleanest execution of all 4 skills. OWASP-classified findings, context-aware severity downgrades (HIGH CVE -> LOW in dev-only context), attack chain analysis, actionable remediation (enforce `npm ci`, not `npm install`).

**Reviewer consultation:** 1 invocation, succeeded first try. Used only 2 tool calls (Read package-lock.json lines 0-49, Read project.json for line numbers).

**Why it succeeded:** The executor's context packaging was the most thorough of all 4 skills:
- Explicit threat model (dev-only, no prod exposure, attacker profiles, protected assets)
- Pre-curated findings with CVE identifiers and OWASP categories
- File contents and line numbers inlined
- Clear request ("validate these 4 findings against OWASP Top 10")

The reviewer needed tool calls only to verify version numbers -- everything else was in the prompt.

## Cross-Cutting Findings

### F1: Advisor maxTurns exhaustion is consultation-type specific

| Consultation style | Success rate | Typical tool pattern |
|---|---|---|
| Focused question (plan, mid-execute) | High | 2-4 batched parallel calls |
| Broad verification (pre-execute, final-review, review) | Low | 4-8 sequential calls, one file per turn |

The pattern is clear: when the advisor is asked a targeted question ("what's the right approach given these facts?"), it batches calls efficiently. When asked to verify or review ("check all changes against the plan"), it reads each file one-per-turn and exhausts the budget.

### F2: Agents ignore soft instructions not to re-read files

Even with explicit "you do not need to read files" and full file contents in the prompt, both the advisor and reviewer agents re-read files. This means:
- Soft discouragement is insufficient
- Context packaging alone does not solve the problem
- Stronger prompt constraints are needed (e.g., turn budget awareness, explicit conditions under which Read/Glob may be used)

### F3: Executor context packaging quality predicts consultation success

The security-review executor produced the cleanest advisor consultation because it packaged:
- Structured threat model
- Pre-curated findings with context tags (CVE, OWASP)
- Inline file contents
- Targeted validation request

The pre-execute consultation in the execute skill included file contents but lacked the equivalent structure -- just "here are the facts, what do you think?" The advisor responded by independently re-verifying instead of answering.

### F4: Executor recovery mechanism works but at high cost

When the advisor returns a preamble-only response, the executor correctly detects the failure and either:
- Resumes the agent in the background with a follow-up message (~6 min delay)
- Re-invokes the agent with an adapted prompt (immediate retry)

The recovered output quality is consistently high. The cost is time (6+ minutes for background resume) and context budget (re-reading the whole conversation).

### F5: Executor orientation lacks stopping heuristic

The plan skill's Phase 1 over-investigated node_modules internals. The executor kept exploring until the user interrupted. This is a separate issue from the advisor maxTurns problem but manifests in the same skill.

## Recommendations for Next Phase

Based on field-test evidence, the next phase should:

### High priority (prompt-level fixes, no structural changes)

1. **Add batching instruction to advisor/reviewer/security-reviewer agents.** Explicit language: "You have 3 turns total. Make parallel tool calls in a single turn (e.g., multiple Read calls together), not one tool per turn."

2. **Add context-trust instruction.** Explicit language: "When file contents are included below, DO NOT use Read on those files. Only use Read/Glob for files not provided. This is a hard constraint, not a preference."

3. **Standardize executor context packaging.** Adopt the security-review executor's packaging as the template for all skills:
   - Explicit threat/scope statement
   - Pre-curated findings or facts with context tags
   - Inline file contents for any file the advisor might need
   - Targeted request, not "what do you think?"

### Medium priority (structural fallback if prompt fix insufficient)

4. **Consider maxTurns bump to 5 only if prompt fixes fail validation.** Current evidence suggests maxTurns: 3 is sufficient when the advisor batches. Increasing the budget without fixing batching masks the real problem and adds cost.

### Lower priority (separate concern)

5. **Add orientation stopping heuristic to plan/execute skills.** Explicit budget or signal for when to stop exploring and consult the advisor. Not related to the advisor maxTurns issue but observed in the same session.
