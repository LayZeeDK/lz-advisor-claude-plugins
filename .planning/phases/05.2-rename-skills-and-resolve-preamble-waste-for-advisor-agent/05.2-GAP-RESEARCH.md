**Researched:** 2026-04-14
**Domain:** Claude Code subagent prompt engineering, maxTurns/effort interaction, output format control

## Summary

This research investigates prompt engineering techniques for a Claude Code plugin advisor agent that uses Read/Glob tools with `maxTurns: 3` and `effort: high`. The UAT-RESEARCH.md already proved the structural fix: `maxTurns: 1` is the root cause of preamble waste, not effort level. This document addresses the follow-up question: given that maxTurns: 3 makes intermediate text invisible to the parent, what prompt patterns produce the best advisor output?

The key finding is that three mechanisms work together: (1) the structural guarantee that only the final text-only response reaches the parent (maxTurns: 3 provides enough turns for tool use + synthesis), (2) effort: high gives Opus 4.6 deep strategic reasoning appropriate for an advisor role, and (3) the system prompt enforces output format (enumerated steps, under 100 words) and explicitly describes the visibility model ("your tool calls and intermediate text are internal; only your final text-only response reaches the executor").

**Primary recommendation:** Use maxTurns: 3, effort: high, and restructure the advisor system prompt to describe the visibility model, use positive format instructions, and rely on structural guarantees rather than behavioral prohibitions.

## Research Question 1: Prompt Patterns for Tool-Using Advisors

### The Visibility Model Problem

When a Claude Code subagent has tools and `maxTurns > 1`, the parent receives **only the final text-only response**. All intermediate text (preamble before tool calls, tool call narration, mid-turn commentary) stays inside the subagent's context and never reaches the parent. [VERIFIED: code.claude.com/docs/en/sub-agents]

This means:
- With `maxTurns: 1`: Agent writes preamble + tool calls in one turn. Tools execute. maxTurns hit. Agent never gets a synthesis turn. Parent receives the pre-tool-call preamble as the "Response." [VERIFIED: code.claude.com/docs/en/agent-sdk/agent-loop]
- With `maxTurns: 3`: Agent writes preamble + tool calls (turn 1, invisible to parent). Tools execute. Agent optionally does more reads (turn 2, invisible). Agent writes text-only synthesis (final turn, returned to parent). [VERIFIED: same source]

### Key Insight: With maxTurns: 3, Preamble Is Harmless

The structural fix eliminates preamble from the parent's view entirely. The agent can write "Let me verify..." before its tool calls and it costs nothing in the parent's context -- that text stays inside the subagent. The system prompt should therefore focus on making the **final response** high-quality rather than trying to eliminate intermediate narration.

### Recommended Prompt Pattern: Describe the Visibility Model

Tell the agent explicitly what the parent sees. From the official Anthropic subagent docs: "Only the subagent's final text response comes back to your context, plus a small metadata trailer with token counts and duration." [VERIFIED: code.claude.com/docs/en/sub-agents]

**Pattern:**
```
Your tool calls and intermediate text are internal to your context.
Only your final text-only response reaches the executor. Structure that
response as enumerated actionable steps.
```

This is more effective than "don't narrate your tool calls" because:
1. It explains the mechanism (positive instruction) rather than prohibiting behavior (negative instruction)
2. Opus 4.6 is "more responsive to the system prompt" and responds better to clear, direct instructions [VERIFIED: platform.claude.com/docs/en/build-with-claude/prompt-engineering/claude-prompting-best-practices]
3. Anthropic's own best practices say "Tell Claude what to do instead of what not to do" [VERIFIED: same source]

### Comparison with Built-in Subagent Patterns

Claude Code's built-in Explore subagent uses: "Communicate your final report directly as a regular message -- do NOT attempt to create files. NOTE: You are meant to be a fast agent that returns output as quickly as possible." [CITED: github.com/Piebald-AI/claude-code-system-prompts]

Claude Code's Worker Fork subagent uses: "Be concise -- as short as the answer allows, no shorter. Plain text, no preamble, no meta-commentary." and "One shot: report once and stop. No follow-up questions, no proposed next steps, no waiting for the user." [CITED: same source]

Both use imperative, positive instructions about what to include, not what to avoid.

## Research Question 2: Output Format Enforcement

### Enumerated Steps Pattern

Anthropic's own advisor tool documentation includes this trimming instruction: "The advisor should respond in under 100 words and use enumerated steps, not explanations." This line cut advisor output tokens by 35-45% in internal testing. [VERIFIED: platform.claude.com/docs/en/agents-and-tools/tool-use/advisor-tool]

### Effective Format Control Techniques (Ranked)

1. **Explicit format instruction in system prompt** -- "Respond in under 100 words. Use enumerated steps, not explanations." This is the most reliable technique. [VERIFIED: Anthropic advisor tool docs]

2. **Structural constraint through role** -- "You are a strategic engineering advisor specializing in concise, actionable guidance." Setting a role focuses behavior. [VERIFIED: prompting best practices]

3. **Few-shot examples** -- Anthropic recommends 3-5 examples for best format control. For the advisor, one example of a well-formatted response in the system prompt would reinforce the pattern. [VERIFIED: prompting best practices]

4. **XML tags for format regions** -- Wrapping the desired output format in XML tags helps Claude parse intent. [VERIFIED: prompting best practices]

### What Does NOT Work Reliably

- **"Do not start with..."** -- Negative instructions can backfire through reverse-priming. Anthropic advises positive framing instead. [VERIFIED: prompting best practices, "Eliminating preambles" migration section]
- **Prefilled assistant responses** -- Deprecated on Claude 4.6 models. The migration path is: "Respond directly without preamble. Do not start with phrases like 'Here is...', 'Based on...'." [VERIFIED: prompting best practices, "Migrating away from prefilled responses"]
- **effort: medium/low to force conciseness** -- Effort controls reasoning depth, not output format. Lower effort reduces thinking tokens and tool calls, but does not reliably change the text structure of the final response. [VERIFIED: platform.claude.com/docs/en/build-with-claude/effort]

## Research Question 3: Internal vs External Response Awareness

### Should the System Prompt Describe the Visibility Model?

**Yes, explicitly.** This is the single most impactful prompt change for the advisor agent.

Current prompt says: "The executor sees only your final text response, not your tool call sequence."

This is partially correct but should be strengthened to:

```
Your tool calls and intermediate text are internal. Only your final
text-only response reaches the executor. The executor cannot see your
reasoning, your tool outputs, or any text you produce alongside tool
calls. Put all substantive guidance in your final response.
```

**Why this matters:** With maxTurns: 3, the agent gets multiple turns. If it "knows" the parent only sees the final text, it can freely verify claims via tools in earlier turns and then craft a polished final response. Without this knowledge, the agent may try to compress everything into a single message alongside tool calls, producing worse output.

### Evidence from Anthropic's Advisor Tool Design

The API advisor tool's design validates this pattern. The advisor tool docs state: "The advisor itself runs without tools and without context management. Its thinking blocks are dropped before the result returns; only the advice text reaches the executor." [VERIFIED: advisor tool docs]

In the Claude Code plugin context, the advisor DOES have tools, but the principle is the same: only the final text matters. Making this explicit in the system prompt aligns the agent's behavior with the structural reality.

## Research Question 4: Effort: high Interaction with Tool Use

### Official Documentation on Effort + Tool Use

From the effort parameter docs: [VERIFIED: platform.claude.com/docs/en/build-with-claude/effort]

**Lower effort levels tend to:**
- Combine multiple operations into fewer tool calls
- Make fewer tool calls
- **Proceed directly to action without preamble**
- Use terse confirmation messages after completion

**Higher effort levels may:**
- Make more tool calls
- **Explain the plan before taking action**
- Provide detailed summaries of changes
- Include more comprehensive code comments

### Analysis: Why effort: high is Correct for the Advisor

The "explain the plan before taking action" behavior at high effort is exactly what caused the preamble at maxTurns: 1 -- the agent narrated its intent before tool calls, and that narration became the only response.

With maxTurns: 3, this behavior is **harmless and actually beneficial**:
- The agent's planning text before tool calls stays internal (invisible to parent)
- Higher effort means deeper strategic reasoning in the final response
- The advisor's value proposition is "Opus-level strategic guidance" -- that requires high effort
- effort: medium would reduce the reasoning depth that makes the advisor worth consulting

### Effort Levels in Anthropic's Own Agent Design

The Agent SDK docs recommend effort: high for "refactors, debugging" and effort: low for "file lookups, listing directories." [VERIFIED: code.claude.com/docs/en/agent-sdk/agent-loop]

The advisor agent performs strategic reasoning, not file lookups. effort: high is the correct level.

Anthropic's suggested effort for research agents is `effort: high`. [CITED: blakecrosley.com/guides/claude-code]

### The Effort Claim from CONTEXT.md D-11 Was Wrong

CONTEXT.md D-11 cited: "Lower effort levels tend to proceed directly to action without preamble." This is true for the API, but **irrelevant** when the fix is structural (maxTurns: 3). The preamble happens regardless of effort level; maxTurns: 3 simply makes it invisible to the parent. Reducing effort to medium sacrifices reasoning depth to solve a problem that maxTurns: 3 already eliminates completely.

## Research Question 5: Context Packaging for Advisor Prompts

### Minimizing Tool Calls Through Better Context Packaging

The more context the skill packages into the advisor prompt, the fewer Read/Glob calls the advisor needs. With maxTurns: 3, the advisor has budget for 2 tool-use turns + 1 synthesis turn, but ideally it should use 0-1 tool turns and deliver guidance immediately.

### Anthropic's Advisor Tool Guidance on Context

From the API advisor tool docs: "The server constructs the advisor's view from the full transcript automatically." [VERIFIED: advisor tool docs]

In the Claude Code plugin pattern, this is different -- the advisor starts with a fresh context and receives only what the skill packages. The skill MUST include:

1. **The user's original task** (quoted)
2. **Key orientation findings** (summarized, not raw file dumps)
3. **User-pasted source material** (verbatim, not paraphrased -- per Phase 5.1 decision)
4. **A specific question** for the advisor to answer

### Current Skill Pattern (Already Well-Designed)

The plan skill's consult phase already follows this pattern well:
```
1. The user's original task (quote their request)
2. Key findings from orientation (files examined, patterns found,
   constraints discovered)
3. A specific question: "Given these findings, what approach and
   sequence of steps would you recommend?"
```
[VERIFIED: plugins/lz-advisor/skills/lz-advisor.plan/SKILL.md]

**Recommendation:** No changes needed to context packaging. The current pattern is aligned with Anthropic's guidance. If the advisor consistently makes 2 tool-use turns, consider whether the skills are under-packaging context.

## Common Pitfalls

### Pitfall 1: Confusing Effort with Output Format
**What goes wrong:** Using effort: low/medium to control output verbosity
**Why it happens:** Effort docs say "proceed directly without preamble" at lower levels
**How to avoid:** Effort controls reasoning depth and token spend. Output format is controlled by system prompt instructions. Use effort for reasoning quality, system prompt for format.
**Warning signs:** Advisor gives shallow guidance but in the "right" format

### Pitfall 2: Negative Instructions for Format Control
**What goes wrong:** "Do NOT start with 'Let me verify...'" causes reverse-priming
**Why it happens:** Mentioning the unwanted pattern activates it in the model's distribution
**How to avoid:** Use positive instructions: "Respond with enumerated steps from the first token"
**Warning signs:** Agent produces the exact phrasing you told it not to use

### Pitfall 3: maxTurns Too Low for Tool-Using Agents
**What goes wrong:** Agent writes preamble + tool calls, maxTurns hit, preamble becomes response
**Why it happens:** maxTurns counts tool-use turns only; the final text-only response is free
**How to avoid:** Set maxTurns to at least N+1 where N is the expected number of tool-use turns
**Warning signs:** Parent receives "Let me verify..." or "I'll check..." as the advisor response

### Pitfall 4: Over-Constraining Internal Turns
**What goes wrong:** Trying to make intermediate (invisible) text concise
**Why it happens:** Not understanding that only the final response reaches the parent
**How to avoid:** Describe the visibility model in the system prompt; focus constraints on the final response only
**Warning signs:** Spending prompt budget on instructions that affect invisible behavior

## Open Questions

1. **Does maxTurns: 3 provide enough buffer for Glob-then-Read patterns?**
   - What we know: Turn 1 = Glob to find files, Turn 2 = Read specific files, Turn 3 would be synthesis but is the 3rd tool-use turn if the agent issues tool calls
   - What's unclear: Whether the agent ever needs 3 full tool-use turns
   - Recommendation: maxTurns: 3 is correct because the final text-only response does NOT count as a turn. The agent gets 3 tool-use turns + 1 free synthesis turn. This is more than enough.

2. **Should the system prompt include a few-shot example of the desired output format?**
   - What we know: Anthropic recommends 3-5 examples for format control
   - What's unclear: Whether the word budget constraint ("under 100 words") leaves room for examples in the system prompt without bloat
   - Recommendation: Defer to execution. The current format instructions are clear enough; add an example only if empirical testing shows format inconsistency.
