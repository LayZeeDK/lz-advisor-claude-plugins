# Phase 5.2: Rename Skills and Resolve Preamble Waste for Advisor Agent - Research

**Researched:** 2026-04-14
**Domain:** Claude Code plugin skill renaming, Opus 4.6 effort parameter behavior, cross-reference maintenance
**Confidence:** HIGH

## Summary

This phase involves two orthogonal workstreams: (1) renaming all four skill directories and `name` fields from bare names (`plan`, `execute`, `review`, `security-review`) to dotted-prefix names (`lz-advisor.plan`, etc.) with end-to-end cross-reference updates, and (2) changing the advisor agent's effort parameter from `high` to `medium` to eliminate the preamble waste that Phase 5.1's prompt-level Final Response Discipline failed to resolve.

Both workstreams are well-defined mechanical operations with minimal ambiguity. The effort parameter change is strongly supported by official Anthropic documentation that explicitly states lower effort levels cause the model to "proceed directly to action without preamble." The skill renaming uses `git mv` for directory renames (preserving history) and manual find/replace for cross-references across 20+ files.

The primary risk is the dot character in skill names (D-07 in CONTEXT.md already accepts this risk). Claude Code's skill resolution system uses `plugin:skill` colon-separated qualified names, and all official examples use kebab-case without dots. The dotted names produce qualified forms like `lz-advisor:lz-advisor.plan` which are redundant but functional. The `/lz-advisor.plan` shorthand may or may not resolve correctly -- this is an accepted bet per D-07.

**Primary recommendation:** Execute directory renames first (atomic git operations), then bulk-edit frontmatter and cross-references, then effort parameter change on advisor.md, then MODEL-OPTIMIZATION-OPUS.md rewrite, with verification at each stage.

## Effort Parameter Behavior on Opus 4.6

### Official Documentation Findings [VERIFIED: platform.claude.com/docs/en/docs/build-with-claude/effort]

The effort parameter is **generally available** (GA) on Opus 4.6 -- no beta header required. This is a significant change from Opus 4.5 where it was in beta.

**Available levels:** `low`, `medium`, `high` (default), `max` (new on 4.6)

**Key behavioral difference at lower effort levels (direct quote from Anthropic docs):**

> "Lower effort levels tend to:
> - Combine multiple operations into fewer tool calls
> - Make fewer tool calls
> - **Proceed directly to action without preamble**
> - Use terse confirmation messages after completion"

> "Higher effort levels may:
> - Make more tool calls
> - **Explain the plan before taking action**
> - Provide detailed summaries of changes
> - Include more comprehensive code comments"

This is the smoking gun for why `effort: high` caused preamble despite Phase 5.1's prompt-level Final Response Discipline. At `effort: high`, the model is designed to "explain the plan before taking action" -- which is exactly the "Let me verify..." pattern observed in live testing. Changing to `effort: medium` activates the "proceed directly to action without preamble" behavior.

### Effort Level Recommendation

| Agent | Current | Recommended | Rationale |
|-------|---------|-------------|-----------|
| advisor.md | `effort: high` | `effort: medium` | Conciseness priority. "Proceed directly to action without preamble" matches the zero-preamble requirement (D-15). |
| reviewer.md | `effort: high` | `effort: high` (keep) | Analysis depth priority. Reviewers benefit from thorough reasoning. Preamble acceptable per D-17. |
| security-reviewer.md | `effort: high` | `effort: high` (keep) | Same as reviewer. Depth outweighs preamble cost for security analysis. |

### Adaptive Thinking Interaction [VERIFIED: platform.claude.com/docs/en/build-with-claude/adaptive-thinking]

On Opus 4.6, Claude Code's Agent tool manages adaptive thinking automatically. The `effort` parameter controls thinking depth:

- At `high` (default) and `max` effort, Claude **almost always thinks**
- At `medium`, Claude uses **moderate thinking** and may skip thinking for simple queries
- At `low`, Claude **minimizes thinking**

For the advisor agent at `effort: medium`, this means Opus may skip extended thinking for straightforward advisory queries and go directly to the response. This is fine -- the advisor's 100-word constraint already limits output, and the thinking overhead on short advisory responses is wasteful.

**No explicit `thinking` configuration needed** (D-18). The `effort` parameter alone is the control surface in Claude Code's Agent tool context.

## Git mv for Directory Renames

### Pattern for Skill Directories [ASSUMED]

`git mv` works recursively on directories. For renaming a skill directory:

```bash
git mv plugins/lz-advisor/skills/plan plugins/lz-advisor/skills/lz-advisor.plan
```

This atomically renames the directory and all contents (SKILL.md in this case), staging both the source deletion and destination creation. Git treats this as a rename, preserving file history.

### Pattern for Eval Workspace Directories

```bash
git mv evals/lz-advisor/lz-advisor-plan-workspace evals/lz-advisor/lz-advisor.plan-workspace
```

Same pattern -- directory rename is atomic with `git mv`.

### Gotcha: Edit After Rename [ASSUMED]

After `git mv`, the file is at the new path. All subsequent edits (frontmatter updates, cross-reference changes) use the new path. The rename and edit should ideally be in separate commits for clean history, but combining them in a single commit is also acceptable.

### Gotcha: Windows Path with Dots

Dots in directory names are valid on all platforms (Windows NTFS/ReFS, macOS, Linux). No filesystem issues expected. Git handles dots in paths without issue.

## Skill Naming in Claude Code

### How Skill Resolution Works [CITED: code.claude.com/docs/en/skills]

Claude Code discovers skills by scanning `skills/*/SKILL.md` directories. The `name` field in SKILL.md frontmatter becomes the skill identifier. For plugins, names are qualified as `plugin-name:skill-name`.

Current names: `lz-advisor:plan`, `lz-advisor:execute`, `lz-advisor:review`, `lz-advisor:security-review`

After rename: `lz-advisor:lz-advisor.plan`, `lz-advisor:lz-advisor.execute`, `lz-advisor:lz-advisor.review`, `lz-advisor:lz-advisor.security-review`

### Dot Character Risk Assessment

No official Claude Code documentation explicitly addresses dots in skill `name` fields. All official examples use kebab-case (hyphens only). The colon `:` is the established plugin/skill separator.

**Risk:** Claude Code could interpret the dot as a separator, causing resolution failures.
**Mitigation:** D-07 accepts this risk. The qualified form `lz-advisor:lz-advisor.plan` always works as a fallback. The `/lz-advisor.plan` shorthand is the bet.
**Testing:** Automated `claude -p` verification (D-16) will validate whether the shorthand resolves.

### Convention Deviation

The CLAUDE.md Naming Conventions section currently says: "Skill directories: kebab-case (`plan/`, `execute/`, `review/`, `security-review/` -- unprefixed; plugin namespace is implicit)". This section needs updating to reflect the new dotted names and the rationale for the deviation from pure kebab-case.

## Common Pitfalls

### Pitfall 1: Changing Agent References Instead of Skill References

**What goes wrong:** During cross-reference updates, accidentally changing `lz-advisor:advisor` (agent qualified name) to `lz-advisor.advisor` (which would be a skill name pattern).
**Why it happens:** Both skill and agent references use the `lz-advisor:` prefix. The pattern `lz-advisor:plan` is a skill reference; `lz-advisor:advisor` is an agent reference.
**How to avoid:** The rule is simple: agent names are UNCHANGED (D-06). Only references that match the four skill names (`plan`, `execute`, `review`, `security-review`) get the dotted treatment. Agent names (`advisor`, `reviewer`, `security-reviewer`) never appear with dots.
**Warning signs:** Any edit to `allowed-tools:` frontmatter, any change to `Agent(lz-advisor:...)` patterns, any edit to agent `.md` file names or `name:` fields.

### Pitfall 2: Inconsistent Cross-Reference Forms

**What goes wrong:** Some references use `lz-advisor.plan` (new shorthand) while others still use `lz-advisor:plan` (old qualified form) in the same document.
**Why it happens:** Multiple reference patterns exist in the codebase. Skill descriptions use the qualified form for sibling references; README uses dotted form; CLAUDE.md uses both.
**How to avoid:** D-10 specifies: sibling skills referenced by shorthand (`lz-advisor.execute`, not `lz-advisor:execute`). Apply this consistently.
**Warning signs:** Running `git grep "lz-advisor:plan\|lz-advisor:execute\|lz-advisor:review\|lz-advisor:security-review"` after edits -- any matches in non-planning files are stale references.

### Pitfall 3: Eval JSON Queries Not Updated

**What goes wrong:** Eval workspace directories get renamed but the trigger queries inside the eval JSON files still use the old `"lz-advisor plan"` (space-separated) pattern.
**Why it happens:** Eval JSONs contain natural-language queries, not code references. Easy to overlook during a mechanical rename.
**How to avoid:** Per D-22, update eval query files. Specifically, the queries that start with `"lz-advisor plan..."`, `"lz-advisor execute..."`, `"lz-advisor review..."`, and `"lz-advisor security-review..."` should become `"lz-advisor.plan..."` etc.
**Warning signs:** Exactly 4 query strings (one per eval file) that start with the old `lz-advisor <space> <skill>` pattern.

### Pitfall 4: Forgetting to Update CLAUDE.md Directory Diagram

**What goes wrong:** The directory structure ASCII art in CLAUDE.md still shows `plan/`, `execute/`, `review/`, `security-review/` after the rename.
**Why it happens:** CLAUDE.md is documentation, not code. Easy to forget in a code-focused phase.
**How to avoid:** D-03 explicitly requires all cross-references including CLAUDE.md. The directory diagram, naming conventions section, and qualified name comments all need updating.
**Warning signs:** The directory diagram is at approximately CLAUDE.md lines 43-62.

### Pitfall 5: MODEL-OPTIMIZATION-OPUS.md Stale Claims

**What goes wrong:** After updating the research doc, some claims still reference Opus 4.5 behavior (e.g., "200K context only", "effort parameter is BETA", "replace 'think' with alternatives").
**Why it happens:** The document is 885 lines with Opus 4.5 information deeply embedded in prose, code examples, and tables.
**How to avoid:** Systematic section-by-section review. Key items to find and update: "200K" context references, "BETA" effort references, "think" word replacement guidance, `budget_tokens` examples, prefilling examples, pricing (may have changed).
**Warning signs:** Searching for "4.5", "200K", "BETA", "budget_tokens" after editing.

## Open Questions

1. **Dot in skill name resolution**
   - What we know: Claude Code uses `plugin:skill` colon-separated qualified names. All official examples use kebab-case.
   - What's unclear: Whether `/lz-advisor.plan` resolves via the name field or gets confused by the dot.
   - Recommendation: Proceed per D-07 (accepted risk). Automated testing in D-16 will validate. Fallback is the qualified form `lz-advisor:lz-advisor.plan`.

2. **Effort: medium impact on advisory quality**
   - What we know: At medium effort, Claude uses moderate thinking and may skip it for simple queries. The advisor's 100-word constraint already limits output.
   - What's unclear: Whether medium effort reduces the QUALITY of strategic advice (not just preamble).
   - Recommendation: Proceed per D-11. If quality degrades, D-19 (deferred) provides escalation paths.
