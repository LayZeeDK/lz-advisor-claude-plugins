# Phase 5.1: Advisor Consultation Refinements - Context

**Gathered:** 2026-04-14
**Status:** Ready for planning
**Source:** Direct decisions from empirical testing discussion (2026-04-14 session)

<domain>
## Phase Boundary

Close three empirical gaps discovered during live marketplace testing of the v1.0 plugin. All three relate to advisor consultation quality and fidelity. None are blocking for v1 marketplace release; all reduce plugin polish and are worth fixing before wider distribution.

Scope: surgical edits to three agent system prompts (advisor.md, reviewer.md, security-reviewer.md) and two skill files (plan/SKILL.md, execute/SKILL.md), plus one reference file (advisor-timing.md). No structural changes, no threat-model revisit, no new requirements.

Version bump: 0.1.0 → 0.2.0 on completion (polish fixes warrant minor bump, not patch, per semver — they change observable behavior).

</domain>

<decisions>
## Implementation Decisions

### Fix A: Final Response Discipline

- **D-01:** Remove the "ask the executor to clarify" Edge Case clause from `agents/advisor.md` lines 101-103. It directly contradicts D-03 from Phase 1 ("Advisor always commits to a recommendation even with incomplete context. Never responds with 'I need more context' -- that wastes the single turn"). This was a regression introduced during Wave 2's agent expansion.

- **D-02:** Add a "Final Response Discipline" section to all three agent system prompts (`advisor.md`, `reviewer.md`, `security-reviewer.md`) that explicitly forbids:
  - Preamble phrasing ("Let me verify...", "I'll check...", "First I'll...")
  - Announcing intent without executing it in the same turn
  - Requesting clarification (no more turns available under `maxTurns: 1`)
  - Narrating Read/Glob tool use
  
  And explicitly requires:
  - Committing to guidance based on available context
  - Stating assumptions explicitly when context is incomplete ("Assuming X, do Y; if X is wrong, do Z")
  - Using Read/Glob silently for verification; folding results into the final answer

- **D-03:** Keep `maxTurns: 1` unchanged on all three agents. This is a deliberate Phase 1 decision (D-02) with security mitigations (T-01-03, T-04-03/04/08/09) and cost guarantees depending on it. Raising maxTurns would invalidate 5 threat mitigations and violate the documented "single inference pass per consultation" cost ceiling. The preamble problem is not caused by `maxTurns: 1`; it is caused by system-prompt permissiveness that `maxTurns: 1` cannot rescue.

- **D-04:** Keep `tools: ["Read", "Glob"]` unchanged on all three agents. Adding `WebFetch` or `WebSearch` would break the API advisor pattern parity — the real Anthropic API advisor has NO tools and sees the full transcript server-side. Our plugin's Read/Glob compensates for missing transcript access, not for lack of exploration capability. Exploration belongs to the executor, not the advisor.

### Fix B: Verbatim-vs-Summary for Source Material

- **D-05:** Add a "What to include in the advisor prompt verbatim" section to `skills/plan/SKILL.md` and `skills/execute/SKILL.md` consult-phase instructions. The section MUST specify:
  - Preserve verbatim (do not paraphrase): user-pasted documentation excerpts, RFC/ADR text, API specs, error messages, configuration blobs, code snippets from third-party sources, tool output the user surfaced as evidence
  - Summarize only the executor's own orientation findings (files read, patterns observed, architectural relationships)
  - If user-pasted material exceeds ~2k words, quote the sections relevant to the decision verbatim rather than paraphrasing the whole document

- **D-06:** Update `references/advisor-timing.md` to add a clarifying sentence: web content the executor pre-fetches during orient must be included in the advisor's prompt verbatim. The advisor has no network access; treat it as a judge of evidence, not a fetcher. This aligns with the API advisor documentation's "Combining with other tools" section which shows `web_search` as an executor tool, not an advisor tool.

- **D-07:** Do NOT add a URL-fetching step to the orient phase. The existing user experience (user pastes relevant content into the prompt or references specific codebase files) is sufficient when paired with Fix B. Adding orient-phase URL fetching would introduce scope creep, tool dependencies (WebFetch success depends on the fallback chain in CLAUDE.md), and latency. If users reference URLs without pasting content, the existing failure mode (advisor works from whatever summary the executor produces) is acceptable given Fix A and Fix B reduce its severity.

### Fix E: Re-Consultation Triggers During Execute

- **D-08:** Add a "Re-consultation triggers during execute phase" section to `skills/execute/SKILL.md`. The section MUST enumerate these triggers:
  - Empirical evidence contradicts the plan or pre-execute guidance (type error, failing test, tool output invalidating an assumption)
  - Considering a non-trivial change of approach (planned step doesn't work; substitute requires strategic judgment not just surface fix)
  - Silent failure discovered (tool exit code zero but expected artifact missing or wrong)
  - Deviation from the plan changes its strategic shape (moving config between files, swapping integration patterns, adding a workaround with ongoing maintenance cost)
  
  And MUST explicitly deprioritize:
  - Trivial fixes (typos, import paths, obvious refactors) — proceed solo
  - Format/style corrections — proceed solo

- **D-09:** Re-consultation prompts must be brief (2-3 sentences of context): what changed, what options are being considered, what decision is needed. Not a full orient repackage. Rationale: mid-execution consultations should be narrow and targeted; they reuse the pre-execute consultation context implicitly.

### Cross-cutting decisions

- **D-10:** No new REQUIREMENTS.md entries. Phase 5.1 refines existing requirements (ADVR-02, IMPL-02, IMPL-03), not adds new ones. Traceability through existing IDs.

- **D-11:** No UAT file for Phase 5.1. Validation is empirical: re-run the same compodoc prompt used in Phase 5 testing after fixes land, verify via `.output` files that agent Msg 1 now contains substantive content (not preamble). Success criterion: Msg 1 output_tokens > 50 and no preamble phrasing.

- **D-12:** Sonnet 4.6 medium remains the target test configuration. Fixes must produce better behavior specifically at `--effort medium` since that's the marketplace default.

- **D-13:** Single wave. All three fixes are independent in their file surface (different files for each fix); they could run in parallel. But scope is small enough to warrant a single-plan implementation — no need for parallel wave execution machinery.

### Claude's Discretion

- Exact wording of the "Final Response Discipline" section in each agent file — must achieve D-02's goals but phrasing is flexible
- Exact wording of the "verbatim-vs-summary" rule in each skill file — must achieve D-05's goals but structure is flexible
- Exact trigger enumeration in the re-consultation section — must cover D-08's four triggers but examples can be tuned
- Whether to bundle all three fixes into one plan file or split (planner's judgment; single plan is likely correct given the small scope)

</decisions>

<specifics>
## Specific Examples and Evidence

### Preamble failure evidence (confirmed via .output file inspection)

All five sub-agent sessions from Phase 5 testing exhibited the same two-message pattern. Msg 1 contained only a preamble (2-4 output tokens) plus tool_use calls. Msg 2, produced only after the skill executor sent a continuation user turn, contained the substantive response (237-957 output tokens).

Example preambles from the `.output` files:
- `advisor.md` plan skill: "Let me verify the executor options and Storybook 10 config format quickly."
- `advisor.md` execute pre-consult: "Let me verify the exact executor schema and main.ts structure before advising."
- `advisor.md` execute final review: "Let me verify a few things quickly."
- `reviewer.md` review skill: "Let me verify the actual file contents to confirm the findings."
- `security-reviewer.md` security-review skill: "I'll verify the findings against the actual project files before responding."

The preamble pattern is stable across all three agents and both consult-timing phases (pre-execute and final-review). This is strong evidence that the fix must touch all three agent system prompts consistently.

### Example executor compression (Fix B motivation)

In the Phase 5 plan-skill test, the user pasted the full Nx docs markdown. The executor sent the advisor only this 4-bullet summary:
```
The Nx guide (user provided) recommends:
1. Install @compodoc/compodoc
2. Configure compodoc in main.ts
3. Add docs:json script to package.json
4. Enable docs.autodocs: true in main.ts
```
The advisor's recommendation used the old Storybook API (`docs: { autodocs: true }`) which had been removed in Storybook 8. The executor hit a TypeScript error during execute phase and self-corrected to `tags: ['autodocs']` per-story. A verbatim pass-through might have let the advisor catch the Storybook 8 migration during planning.

### Missed re-consultation (Fix E motivation)

During the Phase 5 execute-skill test, the executor encountered three decision points mid-execution:
1. TypeScript error: `autodocs` not in `DocsOptions` (required approach change from main.ts global to per-story tags)
2. Silent failure: build succeeded but `documentation.json` didn't generate (required debugging)
3. Discovery: `hasTsConfigArg` only checks `-p`, forcing flag substitution (`--tsconfig` → `-p`)

None triggered a mid-execution advisor call. Sonnet 4.6 medium made correct technical judgments on all three solo, but decision-artifact quality suffered.
