# Phase 5.1: Advisor Consultation Refinements - Research

**Researched:** 2026-04-14
**Domain:** Claude Code agent system prompt engineering for preamble elimination, context fidelity, and mid-execution consultation triggers
**Confidence:** HIGH

## Summary

Phase 5.1 delivers three surgical fixes to existing agent and skill files. Research confirms all three fixes are technically sound and mutually non-conflicting. The primary risk area is Fix A (preamble elimination), where the interaction between `maxTurns: 1`, `tools: ["Read", "Glob"]`, and `effort: high` creates a specific behavioral pattern: the model emits a narration preamble + tool_use calls as Msg 1 (stop_reason: tool_use), then produces substantive content as Msg 2 after a continuation prompt. The fix must shift substantive content into Msg 1 by forbidding narration that announces tool use. Fix B (verbatim-vs-summary) and Fix E (re-consultation triggers) are additive instruction changes with no interaction risks.

No official Claude Code plugins use `maxTurns: 1` -- the lz-advisor project is the only observed case. No official plugins include preamble-prevention instructions. The fix must be crafted from first principles based on the empirical evidence from Phase 5 testing.

**Primary recommendation:** Add a "Final Response Discipline" section to all three agent system prompts that forbids narration of intent and requires silent tool use. Remove the contradictory "ask to clarify" clause from advisor.md Edge Cases. Add verbatim-vs-summary rules to plan and execute skill consult phases. Add re-consultation trigger enumeration to execute skill.

## Architecture Patterns

### Pattern 1: The Preamble Problem (maxTurns: 1 + tools)

**What:** When an agent has `maxTurns: 1` AND `tools: ["Read", "Glob"]`, the model's natural behavior is to narrate intent before issuing tool calls. This creates a two-message pattern: Msg 1 = preamble + tool_use (stop_reason: tool_use), Msg 2 = substantive response after the Claude Code framework sends tool results and a continuation prompt.

**Root cause analysis:** The model treats Read/Glob calls like any tool -- it narrates what it is going to do, then calls the tool. With unlimited turns, this is invisible (narration is internal to the agent's loop). With `maxTurns: 1`, the narration becomes the visible "preamble" because the agent's final response to the skill executor is Msg 2, not Msg 1. The preamble is NOT caused by `maxTurns: 1` alone -- it is caused by the model's narration habit combined with `maxTurns: 1`'s visibility constraint. [VERIFIED: Phase 5 .output file inspection]

**Key insight:** The Claude Code Agent tool returns the agent's **last** message. With `maxTurns: 1`, if the agent emits text + tool_use in Msg 1, then receives tool results and produces Msg 2, the executor gets Msg 2. But Msg 1's preamble text was wasted output tokens, and the user sees it in the terminal as a misleading first message. [VERIFIED: Phase 5 empirical evidence, 5/5 sub-agent sessions]

**Fix mechanism:** Forbid the narration habit in the system prompt. Tell the model to use Read/Glob silently -- fold tool results into the substantive response rather than announcing tool use beforehand. This way, if the agent calls tools, Msg 1 is ONLY tool_use (no text preamble), and Msg 2 is the substantive response. Or better: the agent may decide to respond substantively without calling tools at all (Msg 1 = substantive text, stop_reason: end_turn). [ASSUMED -- empirical validation required in Phase 5.1 testing]

### Pattern 2: Silent Tool Use Instruction

**What:** Instruct the agent to use Read/Glob for verification silently -- never narrate intent before tool calls, never describe what was read after.

**Why it works:** The model's default behavior is to explain its actions ("Let me check the file..."). By explicitly forbidding this narration, the model shifts to issuing tool calls directly (no text block before tool_use) or folding results into the final answer without announcing the verification step. [ASSUMED -- based on general prompt engineering principles for Claude models]

**Example wording (for planner to adapt):**

```
When using Read or Glob to verify claims, do so silently. Issue tool calls
without announcing them -- no "Let me check..." or "I'll verify..." phrasing.
Fold what you learn from tools directly into your response. The executor sees
only your final text; tool call mechanics are invisible to it.
```

### Pattern 3: Verbatim Context Preservation

**What:** Distinguish between executor-generated summaries (safe to compress) and user-provided source material (preserve verbatim). [CITED: Phase 5.1 CONTEXT.md D-05]

**Why it matters:** The Phase 5 empirical test showed the executor compressing Nx/Storybook documentation into a 4-bullet summary, causing the advisor to recommend a deprecated API. The API advisor tool avoids this because the advisor sees the full transcript server-side. Our plugin must compensate by instructing the executor to preserve user-pasted material verbatim. [VERIFIED: Phase 5 compodoc test evidence in 05.1-CONTEXT.md]

**Implementation:** Add a rule to the consult-phase instructions in both plan and execute skills. The rule distinguishes:
- **Preserve verbatim:** User-pasted documentation, API specs, error messages, config blobs, code snippets from third-party sources, tool output the user surfaced as evidence
- **Summarize:** The executor's own orientation findings (files read, patterns observed, architectural relationships)
- **Truncation guidance:** If user-pasted material exceeds ~2k words, quote the sections relevant to the decision verbatim rather than paraphrasing the whole document

### Pattern 4: Re-Consultation Trigger Enumeration

**What:** Explicit list of conditions under which the executor should re-consult the advisor during execution, beyond the existing "stuck" and "changing approach" categories. [CITED: Phase 5.1 CONTEXT.md D-08]

**Existing coverage:** The execute skill already has a "When to Consult Again" section (lines 83-96) with "Stuck" and "Changing approach" bullets. Fix E expands this with more specific triggers based on empirical evidence from Phase 5 testing. [VERIFIED: execute SKILL.md current content]

**New triggers from D-08:**
1. Empirical evidence contradicts the plan or pre-execute guidance (type error, failing test, tool output invalidating an assumption)
2. Considering a non-trivial change of approach (planned step doesn't work; substitute requires strategic judgment)
3. Silent failure discovered (tool exit code zero but expected artifact missing or wrong)
4. Deviation from plan changes its strategic shape (moving config between files, swapping integration patterns, adding a workaround with ongoing maintenance cost)

**Explicitly deprioritized:**
- Trivial fixes (typos, import paths, obvious refactors) -- proceed solo
- Format/style corrections -- proceed solo

## Common Pitfalls

### Pitfall 1: Aggressive Language in Agent Prompts

**What goes wrong:** Using "NEVER", "MUST", "CRITICAL", "ALWAYS" in agent system prompts violates ADVR-06 (Phase 1 requirement: calm natural language) and the plugin-dev design pattern (positive-only instructions observed across all 19 official agents). [VERIFIED: REQUIREMENTS.md ADVR-06 + Phase 1 D-09]
**Why it happens:** Preamble elimination feels urgent, so the natural impulse is to write "NEVER announce tool use" or "ALWAYS respond directly."
**How to avoid:** Use calm, direct phrasing: "Use Read or Glob silently" rather than "NEVER narrate Read or Glob calls." State what to do, not what not to do.
**Warning signs:** Any use of ALL CAPS in agent system prompt body.

### Pitfall 2: Contradicting the Existing "Edge Cases" Section

**What goes wrong:** Adding a "Final Response Discipline" section that says "never request clarification" while the Edge Cases section still says "ask the executor to clarify" (advisor.md lines 100-103). [VERIFIED: advisor.md current content]
**Why it happens:** Fix A addresses D-02 (add discipline section) but might miss D-01 (remove contradiction).
**How to avoid:** D-01 and D-02 must be applied together. Remove the contradictory clause first, then add the discipline section. The first paragraph of advisor.md's Edge Cases section (lines 100-103) must be replaced, not just augmented.
**Warning signs:** Two sections in the same agent file giving opposite instructions.

### Pitfall 3: Inconsistency Across Three Agent Files

**What goes wrong:** The "Final Response Discipline" section is added to advisor.md but with different wording or omissions in reviewer.md and security-reviewer.md. The preamble pattern was observed across all three agents identically. [VERIFIED: Phase 5 .output file evidence]
**Why it happens:** Each agent has a different system prompt structure (advisor has Output Constraint + Process + Awareness + Response Structure + Edge Cases + Boundaries; reviewer has different sections; security-reviewer has yet different sections). The discipline section must fit into each without conflicting.
**How to avoid:** Write the discipline section once, then adapt it for each agent's context. The core prohibitions and requirements are identical across all three; only the framing sentence differs.

### Pitfall 4: Verbatim Rule Contradicting "Keep the prompt summarized"

**What goes wrong:** The existing skill instructions say "Keep the prompt summarized -- do not paste entire files" (plan SKILL.md line 58, execute SKILL.md line 69). The new verbatim rule says "preserve user-pasted documentation verbatim." These appear contradictory. [VERIFIED: current plan and execute SKILL.md content]
**Why it happens:** The original instruction was correct for executor-generated content but too broad -- it inadvertently covers user-pasted external content too.
**How to avoid:** The verbatim-vs-summary rule must explicitly differentiate: "Summarize your own findings; preserve user-provided source material verbatim." The existing "do not paste entire files" should remain as guidance for executor-generated file reads, not for user-pasted content. The new section should be placed AFTER the existing "Keep the prompt" instruction, refining rather than replacing it.

### Pitfall 5: Re-Consultation Section Duplicating Existing Content

**What goes wrong:** The execute skill already has "When to Consult Again" (lines 83-96) and "Reconciliation" (lines 99-104) sections. Fix E adds a new "Re-consultation triggers" section that duplicates or contradicts these.
**Why it happens:** D-08 introduces four specific triggers that overlap with existing "Stuck" and "Changing approach" bullets.
**How to avoid:** Expand the existing "When to Consult Again" section rather than adding a parallel section. The four D-08 triggers are more specific instances of the existing two categories plus two new categories. Restructure the section to contain all triggers.

### Suggested "Final Response Discipline" section for advisor.md

```markdown
## Final Response Discipline

Respond with substantive guidance from the first message. Do not open with
phrasing that announces intent without delivering on it in the same breath --
phrases like "Let me verify...", "I'll check...", or "First I'll..." are not
followed by another opportunity to respond.

When using Read or Glob to verify claims, issue tool calls without narration.
Fold what you learn into the final answer. The executor sees only your final

Commit to guidance based on available context. If context is incomplete, state
assumptions and provide conditional recommendations rather than requesting
clarification.
```

### Suggested verbatim-vs-summary addition for plan SKILL.md consult phase

```markdown
### What to Include Verbatim

When packaging the advisor prompt, distinguish between source material and
your own findings:

- Preserve verbatim: user-pasted documentation excerpts, API specs, error
  messages, configuration blobs, code snippets from third-party sources,
  and tool output the user surfaced as evidence
- Summarize: your own orientation findings (files read, patterns observed,
```

### Suggested re-consultation triggers expansion for execute SKILL.md

```markdown
### When to Consult Again

During execution, consult the advisor again when:

- **Empirical contradiction**: Evidence contradicts the plan or pre-execute
  guidance -- a type error, failing test, or tool output that invalidates
  an assumption the approach depends on
- **Non-trivial approach change**: The planned step does not work and the
  substitute requires strategic judgment, not just a surface fix
- **Silent failure**: A tool exits successfully but the expected artifact is
  missing, empty, or wrong -- the cause is unclear
- **Strategic shape change**: A deviation from the plan changes its overall
  structure -- moving configuration between files, swapping integration
  patterns, or adding a workaround with ongoing maintenance cost

Do not re-consult for:
- Trivial fixes: typos, import paths, obvious refactors -- proceed solo
- Format or style corrections -- proceed solo

Keep re-consultation prompts brief (2-3 sentences): what changed, what
options are being considered, what decision is needed. These calls reuse
the pre-execute consultation context implicitly; a full orient repackage
is unnecessary.
```

## Interaction Analysis: Fix A x Fix B x Fix E

### Fix A x Fix B (no conflict)

Fix A modifies agent system prompts. Fix B modifies skill instructions (SKILL.md files) and one reference file. They operate on different files and different layers (agent behavior vs executor behavior). No interaction. [VERIFIED: different file targets in CONTEXT.md]

### Fix A x Fix E (no conflict)

Fix A makes agents respond without preamble. Fix E tells the executor WHEN to invoke the agent again. The agent's response quality (no preamble) is independent of when it is invoked. In fact, Fix A improves Fix E's value: if mid-execution re-consultations no longer waste a turn on preamble, the brief 2-3 sentence re-consultation prompts get substantive guidance faster. [ASSUMED -- positive interaction, not negative]

### Fix B x Fix E (no conflict)

Fix B adds verbatim preservation rules to the consult phase (Phase 2 in both skills). Fix E adds re-consultation triggers to the execute phase (Phase 3 in execute skill). They affect different skill phases. The only interaction: Fix E's re-consultation prompts should follow Fix B's verbatim-vs-summary guidance. Since D-09 says re-consultation prompts are "brief (2-3 sentences)" and are about what changed (not about user-pasted source material), the verbatim rule is irrelevant to re-consultations. [VERIFIED: D-09 specifies brief context, not full orient repackage]

### effort: high interaction with Fix A

The `effort: high` setting on all three agents causes Opus to engage adaptive thinking. High-effort thinking increases the model's tendency to "plan before acting" -- which manifests as the preamble narration. Fix A's discipline instruction must be strong enough to override this tendency. The existing Output Constraint section already says "Respond in under 100 words. Use enumerated steps, not explanations." -- this constrains output length but not output timing. The new Final Response Discipline section constrains timing explicitly. [ASSUMED -- effort: high may make preamble more persistent; empirical test will confirm]

### maxTurns: 1 interaction with Fix A

With `maxTurns: 1`, the agent gets one agentic "turn" (a turn = one model inference that may include tool calls). If the model issues tool calls in its inference, Claude Code executes the tools and sends results back for a second inference. The second inference's output is what the executor receives. Fix A aims to make the first inference either (a) produce substantive text without tool calls, or (b) produce tool calls without narration text, so that the second inference (if any) is purely substantive. This is the desired behavior and does not conflict with maxTurns: 1. [VERIFIED: Claude Code agent architecture from ARCHITECTURE.md]
